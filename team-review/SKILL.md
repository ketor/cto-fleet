**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--focus=area1,area2`：聚焦审查领域（可选，默认全面审查）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 每轮审查报告确认 + 修复方案确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅首轮审查报告确认 + 收尾汇总 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **审查发现分级不确定** → 对应 reviewer 自行判断，首轮报告中说明
- **修复方案有争议** → 采纳对应维度 reviewer 的建议
- **迭代超 3 轮仍未达标** → **不可跳过，必须暂停问用户**（熔断机制）
- **修复引入新问题** → fixer 自行回退并重新修复
- **双 reviewer 评分尺度差异过大（同维度校准差 > 2 分）** → **不可跳过，必须暂停问用户**（熔断机制）

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（迭代超 3 轮、评分尺度差异过大时仍必须暂停问用户）。

使用 TeamCreate 创建 team（名称格式 `team-review-{YYYYMMDD-HHmmss}`，如 `team-review-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

**文件命名规范**：

| 角色输出 | 文件名 |
|---------|--------|
| Scanner 报告 | `scanner-report.md` |
| Reviewer-N 第R轮 | `reviewer-{N}-round-{R}.md` |
| 合并报告第R轮 | `merged-report-round-{R}.md` |
| 根因分组 | `root-cause-groups-round-{R}.md` |
| Fixer 第R轮 | `fixer-round-{R}.md` |
| Tester 第R轮 | `tester-round-{R}.md` |
| Architect-N 方案 | `architect-{N}-design.md` |
| 任务拆解 | `task-breakdown.md` |

| 审查任务T | `review-task-{T}.md` |
| 集成测试第R轮 | `integration-test-round-{R}.md` |
| 最终报告 | `final-report.md` |

> 仅当角色存在于当前 skill 时使用对应命名。未列出的角色用 `{role}-{context}.md` 格式。

## 文件交接规范

本 skill 使用**文件交接模式**管理 agent 间通信，防止上下文溢出。

**团队工作目录**：Team lead 在 TeamCreate 后立即执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`，将路径传入每个 agent 的 prompt。

**所有 agent 必须遵守**：
1. **写入文件**：将完整报告写入 `/tmp/{team-name}/{role}-{context}.md`（≤2000 行，超大则拆分为 summary + details 文件）
2. **发送引用**：SendMessage 仅发送文件路径 + ≤500 字符摘要（格式见下方模板）
3. **按需读取**：接收方用 Read 工具读取文件，发送方不内联完整内容
4. **路径转发**：Team lead 转发报告时只发路径+摘要，不 Read 后再 SendMessage

**遵从性校验**：Team lead 收到 agent 消息时，如果消息超过 1000 字符且不包含 `/tmp/team-` 前缀的文件路径，要求该 agent 将内容写入文件后重新发送路径+摘要。

**⚡必须 Read 的节点**：在以下决策节点，Team lead 必须 Read 文件而非仅依赖摘要：
- 合并双路分析报告时
- 判断是否达标/继续迭代时
- 向用户展示关键报告时
- 生成最终报告时

**目录清理**：收尾阶段 TeamDelete 前执行 `rm -rf /tmp/{team-name}`。

## 流程概览

```
阶段零  项目分析 + 自动扫描 → scanner 运行工具收集客观指标 + 双 reviewer 阅读代码
         ↓
阶段一  双维度审查 → reviewer-1 结构审查 + reviewer-2 运行时审查（独立并行）
         → 合并报告 + 交叉校准评分 → 根因归类 → 用户确认
         ↓
阶段二  迭代修复循环（最多 5 轮）：
         ├─ 修复：fixer 按根因分组修复（含自检）
         ├─ 测试：tester 验证 + 覆盖率变化分析
         ├─ 复审：相关维度 reviewer 复审 + 交叉校准
         └─ 判断：达标？改善？停滞？
         ↓
阶段三  收尾 → 最终报告 + 评分历程 + 覆盖率变化 + 清理
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 识别项目技术栈和可用工具，运行自动化分析（lint/类型检查/安全扫描/覆盖率/依赖审计），输出客观指标报告。**仅在扫描阶段工作，完成后关闭。** |
| reviewer-1 | **结构维度审查**：代码质量、架构设计、文档与可维护性。输出结构化审查报告和评分。**不编写或修改代码。独立审查，不与 reviewer-2 交流。** |
| reviewer-2 | **运行时维度审查**：安全性、错误处理、性能、测试覆盖。输出结构化审查报告和评分。**不编写或修改代码。独立审查，不与 reviewer-1 交流。** |
| fixer | 根据审查报告按根因分组修复问题、重构代码、补充测试。修复前简述方案，修复后执行自检。**只做 reviewer 指出的问题，不做额外重构。** |
| tester | 运行测试、验证修复效果、检查回归、分析覆盖率变化、识别测试盲区。**输出测试报告。** |

## 评分体系

双 reviewer 按各自负责的维度独立评分，team lead 按权重合并为总分。满分 10 分。

### 审查维度分配

| 维度 | 权重 | 负责人 | 审查内容 |
|------|------|--------|---------|
| 代码质量 | 20% | reviewer-1 | 命名规范、代码风格一致性、可读性、重复代码、复杂度 |
| 架构设计 | 20% | reviewer-1 | 模块划分、职责清晰度、依赖关系、扩展性 |
| 文档与可维护性 | 5% | reviewer-1 | 关键逻辑注释、API 文档、README 准确性 |
| 安全性 | 15% | reviewer-2 | OWASP Top 10、输入验证、敏感数据处理、依赖安全 |
| 错误处理 | 15% | reviewer-2 | 异常处理完整性、错误传播、边界条件、容错能力 |
| 性能 | 10% | reviewer-2 | 明显的性能问题、资源泄漏、不必要的开销 |
| 测试覆盖 | 15% | reviewer-2 | 单测覆盖率、测试质量、边界用例、测试可维护性 |

**reviewer-1 总权重：45%**（代码质量 20% + 架构设计 20% + 文档 5%）
**reviewer-2 总权重：55%**（安全 15% + 错误处理 15% + 性能 10% + 测试 15%）

**达标线：总分 ≥ 8.0 分 且无 Critical/Major 遗留**，双 reviewer 可宣布满意并提前结束迭代。

**`--focus` 参数处理**：如果用户指定了 `--focus`，对应维度的 reviewer 加大关注力度，但仍输出所有维度评分。聚焦领域的问题优先修复。

### 交叉校准机制

独立评分后，team lead 将合并的完整报告（含双方评分和理由）发给两位 reviewer：
- 各自可调整**自己负责维度**的分数（看到对方的评估后校准尺度）
- 不得修改对方维度的分数
- 如果某维度校准前后差值 > 2 分，team lead 要求该 reviewer 说明理由

## 审查发现分级

每个发现按严重程度分级：

| 级别 | 含义 | 是否必须修复 |
|------|------|-------------|
| **Critical** | 安全漏洞、数据丢失风险、生产事故隐患 | 必须修复 |
| **Major** | 明显 bug、架构缺陷、严重性能问题 | 必须修复 |
| **Minor** | 代码异味、轻微不一致、可改进项 | 建议修复 |
| **Info** | 最佳实践建议、风格偏好 | 可选 |

## 阶段零：项目分析 + 自动扫描

### 步骤 1：启动 scanner 和双 reviewer

三者并行启动。

**Scanner 自动扫描**：
1. 阅读项目结构，识别技术栈和包管理器
2. 识别项目中已配置的工具（lint 配置、类型检查配置、安全扫描配置、测试配置）
3. 运行所有可用工具，收集客观指标：

| 指标类别 | 示例工具 | 输出 |
|---------|---------|------|
| Lint 违规 | ESLint/Pylint/Clippy/golangci-lint | 违规数量 + 按规则分类 |
| 类型错误 | tsc/mypy/pyright | 错误数量 + 位置 |
| 复杂度分析 | radon/lizard/gocyclo | 函数/方法复杂度分布、高复杂度函数列表 |
| 安全漏洞 | npm audit/pip-audit/cargo audit | CVE 列表 + 严重级别 |
| 测试覆盖率 | jest --coverage/pytest-cov/go test -cover | 总覆盖率% + 各模块覆盖率 |
| 过时依赖 | npm outdated/pip list --outdated | 过时包数量 + 版本差距 |
| 代码标记 | grep TODO/FIXME/HACK/XXX | 数量 + 分布位置 |

4. 将完整客观指标报告写入 `/tmp/{team-name}/scanner-metrics.md`，通过 SendMessage 向 team lead 发送文件路径和≤500 字符摘要
5. Scanner 完成后关闭（不参与后续阶段）

**如果某工具未安装或无法运行**：scanner 标注"该指标不可用"，不阻塞流程。

**双 reviewer 同时阅读项目**：
- 阅读项目结构（目录布局、入口文件、配置文件）
- 理解技术栈和框架
- 了解现有测试（框架、覆盖范围、运行方式）
- 各自将项目概况写入 `/tmp/{team-name}/{role}-project-overview.md`，通过 SendMessage 向 team lead 发送文件路径和≤500 字符摘要

### 步骤 2：分发 scanner 报告

Team lead 收到 scanner 报告文件路径后，将该路径转发给两位 reviewer（不 Read 文件内容），作为后续审查的基准数据。

Reviewer 在评分时必须参考客观指标——例如 scanner 报告覆盖率仅 30%，reviewer-2 给测试覆盖的评分应当体现这一事实。

## 阶段一：双维度审查

### 步骤 3：独立并行审查

两位 reviewer 各自审查负责的维度，**互不交流**。

**Reviewer-1（结构维度）**：
- 逐模块/逐文件阅读核心代码
- 评估代码质量（命名、风格、重复、复杂度），参考 scanner 的 lint 数据和复杂度数据
- 评估架构设计（模块划分、依赖关系、抽象层次）
- 评估文档与可维护性
- 记录每个发现：文件路径 + 行号 + 严重程度 + 问题描述 + 修复建议

**Reviewer-2（运行时维度）**：
- 逐模块/逐文件阅读核心代码，重点关注数据流和控制流
- 评估安全性（OWASP Top 10、输入验证、敏感数据），参考 scanner 的安全扫描数据
- 评估错误处理（异常路径、边界条件）
- 评估性能（明显瓶颈、资源泄漏）
- 评估测试覆盖（质量、边界用例），参考 scanner 的覆盖率数据
- 记录每个发现：文件路径 + 行号 + 严重程度 + 问题描述 + 修复建议

### 步骤 4：合并报告 + 交叉校准

两份审查报告完成后：

1. **Team lead 合并报告**：Read 两份审查报告文件，合并后 Write 到 `/tmp/{team-name}/merged-report-round-0.md`（7 个维度全覆盖）
2. **交叉校准**：将合并报告文件路径转发给两位 reviewer（不内联内容），各自 Read 后查看对方维度的评分和理由，可调整自己维度的分数
3. **Team lead 计算总分**：按权重加权

审查报告格式：

```
## 审查报告 - 第 0 轮（初始审查）

### 总分：X.X / 10.0

### 客观指标（来自 scanner）
- Lint 违规：XX 个 | 类型错误：XX 个 | 高复杂度函数：XX 个
- 安全漏洞：XX 个 | 测试覆盖率：XX% | 过时依赖：XX 个 | TODO/FIXME：XX 个

### 各维度评分
**结构维度（reviewer-1）**：
- 代码质量：X.X/10 - [理由]
- 架构设计：X.X/10 - [理由]
- 文档与可维护性：X.X/10 - [理由]

**运行时维度（reviewer-2）**：
- 安全性：X.X/10 - [理由]
- 错误处理：X.X/10 - [理由]
- 性能：X.X/10 - [理由]
- 测试覆盖：X.X/10 - [理由]

### 发现列表（按严重程度排序）
1. [Critical] 文件:行号 - 问题描述 - 修复建议 - 来源：reviewer-1/reviewer-2
2. [Major] 文件:行号 - 问题描述 - 修复建议 - 来源：reviewer-1/reviewer-2
...
```

### 步骤 5：根因归类

Team lead 对合并后的发现列表做根因分析，将相关发现分组：

```
根因 1：[根因描述] — 优先级：[根因优先级]
  ├─ [Critical] 文件:行号 - 问题描述
  ├─ [Major] 文件:行号 - 问题描述
  └─ [Minor] 文件:行号 - 问题描述
  修复方向：[统一修复策略]

根因 2：[根因描述] — 优先级：[根因优先级]
  └─ ...
```

**根因优先级** = 组内最高严重级别 × 涉及发现数量。含 Critical 的根因排最前。

无法归类的独立发现单独列为"散项"。

### 步骤 6：用户确认

Team lead 向用户展示：
- 审查报告摘要（总分 + 客观指标 + 各维度分数）
- 根因分组摘要（每组的根因描述 + 发现数 + 优先级）
- 建议的修复顺序

AskUserQuestion 确认：
- 接受报告，开始修复
- 调整修复优先级或范围
- 补充审查意见

**单轮确认模式**：首轮报告必须经用户确认。
**完全自主模式**：自动决策，不询问用户。

**如果初始评分已 ≥ 8.0 且无 Critical/Major**：
- 向用户展示报告，询问是否仍需修复（可能有 Minor/Info 级别的改进空间）
- 用户选择结束则直接跳到阶段三收尾

## 阶段二：迭代修复循环

**最多 5 轮。** 每轮包含：修复 → 测试 → 复审 → 判断。

### 步骤 7：启动 fixer 和 tester

首次进入阶段二时启动 fixer 和 tester，两者全程保持存活直到收尾。

### 步骤 8：创建修复任务

Team lead 按根因分组创建修复任务（TaskCreate）：

**分批策略（按根因优先级）**：
- 第 1 轮：含 Critical 的根因组 + 优先级最高的根因组
- 第 2 轮：剩余含 Major 的根因组
- 第 3 轮：重要的 Minor 根因组
- 第 4-5 轮：剩余散项（如果仍未达标）

每个修复任务包含：
- 根因描述和修复方向
- 涉及的所有发现（文件/行号/问题）
- 验收标准（修复后应满足的条件）

### 步骤 9：Fixer 修复

Fixer 按根因分组修复：

1. **修复前方案简述**：对每个根因组，fixer 先用 1-2 句描述修复方案发送给 team lead
   - Team lead 将方案转发给对应维度的 reviewer 快速确认方向（不需要详细审查，只判断方向是否合理）
   - Reviewer 确认后 fixer 开始编码
   - **单轮确认模式/完全自主模式**：跳过方案确认，fixer 直接修复
2. **编码修复**：
   - 严格按审查发现修复，不做额外改动
   - 如果修复涉及测试：先修改测试使其与新行为一致，再改实现
   - 如果需要补充测试：编写针对修复点的单元测试
   - 无法修复的问题标注原因，通知 team lead
3. **自检**（提交前必须完成）：
   - 运行 lint/format（如项目有配置）
   - 运行全量测试套件确保无回归
   - 自检通过后才通知 tester

### 步骤 10：Tester 验证

Fixer 自检通过后，tester 执行验证：

**基础验证**：
- 运行全量测试套件
- 检查修复是否生效（针对每个修复项）
- 检查是否有回归（新增失败的测试）
- 如果项目有 lint/类型检查，也一并运行

**覆盖率变化分析**：
- 运行覆盖率工具，对比修复前后的覆盖率变化
- 识别新增/修改代码中未覆盖的行
- 报告测试盲区（改了代码但没有对应测试的区域）

Tester 将完整测试报告写入 `/tmp/{team-name}/tester-report-round-{R}.md`（包含：测试通过/失败数、覆盖率变化、测试盲区列表），通过 SendMessage 向 team lead 发送文件路径和≤500 字符摘要。

**如果测试失败**：
- Tester 报告失败详情
- Fixer 修复失败的测试（不计入轮次，属于当轮内修复）
- 修复后 fixer 重新自检，tester 重新验证
- 当轮内修复最多重试 2 次，仍然失败则升级 team lead

### 步骤 11：Reviewer 复审评分

测试通过后，**按需复审**（效率优化）：

- 本轮修复涉及哪个 reviewer 的维度，只由该 reviewer 复审
- 两者都涉及时双方复审
- 未涉及的维度沿用上轮分数

复审内容：
- 检查修复是否正确解决根因（不只是修补症状）
- 检查是否引入新问题
- 重新评估涉及的维度，输出新一轮评分

**评分回归分析**：当某维度分数下降时，reviewer 必须标注原因：
- **修复引入**：fixer 的修复导致新问题（需要 fixer 处理）
- **新发现**：reviewer 本轮发现了之前遗漏的问题（更新发现列表，不算 fixer 的责任）

**交叉校准**：如果本轮双方都复审，执行交叉校准（同步骤 4）。

将审查报告写入 `/tmp/{team-name}/{role}-review-round-{R}.md`（格式同步骤 4，轮次递增），通过 SendMessage 向 team lead 发送文件路径和≤500 字符摘要，**与上一轮对比**：标注每个维度的分数变化（↑/↓/→）。

### 步骤 12：判断是否继续

Team lead 根据复审结果决定：

**达标（总分 ≥ 8.0 且无 Critical/Major 遗留）**：
- 双 reviewer 宣布满意
- 跳到阶段三收尾

**未达标但有改善**：
- **标准模式**：向用户展示本轮报告和分数变化，AskUserQuestion 确认是否继续下一轮
- **单轮确认模式**：自动继续下一轮（第 3 轮后触发熔断，必须问用户）
   - **完全自主模式**：自动继续下一轮（第 3 轮后触发熔断，必须问用户）

**未达标且无明显改善（总分变化 < 0.3）**：
- 暂停，向用户报告情况，建议调整策略或终止

**达到第 5 轮仍未达标**：
- 强制停止迭代
- 向用户输出完整的迭代历程和当前状态
- 列出剩余未解决的问题

## 核心原则

- **工具先行**：先用自动化工具收集客观指标，再人工审查高价值问题
- **双维度互补**：结构审查 + 运行时审查独立并行，交叉校准确保评分一致
- **根因驱动**：按根因分组修复，一次解决一类问题，不逐条修补症状
- **审查驱动**：所有修改必须基于审查发现，不做无关改动
- **量化跟踪**：每轮评分 + 客观指标，用数据说明改进效果
- **有限迭代**：最多 5 轮，第 3 轮后单轮确认模式/完全自主模式触发熔断
- **不引入回归**：fixer 自检 + tester 全量验证，双重保障

### 共识度计算

team lead 按五维度评估双路分析的共识度：

| 维度 | 权重 |
|------|------|
| 发现一致性（相同问题/结论） | 20% |
| 互补性（独有但不矛盾的发现） | 20% |
| 分歧程度（直接矛盾的结论） | 20% |
| 严重度一致性（同一问题的严重等级差异） | 20% |
| 覆盖完整性（两路合并后的覆盖面） | 20% |

共识度 = 各维度加权得分之和

- **≥ 60%**：自动合并，分歧项由 team lead 裁决
- **50-59%**：合并但标注分歧，收尾时汇总争议点
- **< 50%**：触发熔断，暂停并向用户确认方向

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 项目无法构建/运行 | Fixer 优先修复构建问题，计为 Critical 根因 |
| 没有测试框架 | Scanner 在报告中标注，reviewer 纳入评分，fixer 第一轮搭建基础测试环境 |
| 自动化工具未安装/无法运行 | Scanner 标注"该指标不可用"，reviewer 基于人工审查评分 |
| 修复导致回归 | Fixer 回退修复，重新设计方案后再改 |
| Reviewer-1 与 reviewer-2 评分尺度差异过大 | Team lead 要求双方说明理由；差值 > 2 分触发熔断问用户 |
| Reviewer 与 fixer 方案分歧 | 标准模式：升级用户决策；单轮确认模式/完全自主模式：采纳对应维度 reviewer 建议 |
| 测试环境问题 | Tester 自行排查；无法解决则升级 team lead |
| 评分回归（某维度分数下降） | Reviewer 标注原因（修复引入 vs 新发现），对应处理 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前轮次恢复。如果是 fixer 崩溃，检查已有代码变更决定是否保留。 |
