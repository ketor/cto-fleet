**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--period=1m|3m|6m|1y`：分析周期（默认 `3m`，即三个月）
- `--module=路径`：聚焦分析特定模块（可选，默认全量分析）

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 项目概况确认 + 健康指标确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **分析周期不确定** → team lead 根据项目历史长度自行判断，在最终报告中说明
- **指标异常值过多** → reporter 标注异常，team lead 综合判断后裁决，收尾时汇总
- **异常指标超过 50%** → **不可跳过，必须暂停问用户**（熔断机制）
- **投资建议语言审查未通过项超过 3 项** → **不可跳过，必须暂停问用户**（熔断机制）
- **项目过大无法完整分析** → analyst 识别核心模块，聚焦核心模块分析
- **`--module` 参数处理**：如果用户指定了 `--module`，analyst 优先分析指定模块，其余模块仅概览级别

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（异常指标超过 50%、语言审查未通过项超过 3 项时仍必须暂停问用户）。

分析周期说明：

| 周期 | 适用场景 |
|------|---------|
| `1m` | 近期冲刺回顾、短期负载分析 |
| `3m` | 季度健康检查（默认，最常用） |
| `6m` | 半年度趋势分析、中期知识分布变化 |
| `1y` | 年度回顾、长期团队结构演变 |

使用 TeamCreate 创建 team（名称格式 `team-capacity-{YYYYMMDD-HHmmss}`，如 `team-capacity-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  Git数据提取 → analyst 采集贡献者统计、文件归属、评审模式 → 输出原始数据集
         ↓
阶段一  多维分析 → analyst 计算巴士因子 + 知识分布 + 负载均衡 + 评审瓶颈 + 上下文切换
         ↓
阶段二  健康评分与报告 → reporter 生成健康仪表盘 + 投资建议 → 检查熔断
         ↓
阶段三  框架审查 → reviewer 验证所有输出使用投资语言 → team lead 仲裁
         ↓
阶段四  收尾 → 保存报告 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| analyst | 从 git 历史中提取数据（每位作者的提交分布、文件归属模式、评审参与度、模块知识覆盖度）。运行 git 命令采集原始数据，计算多维健康指标。**只做数据采集和指标计算，不做报告撰写。** |
| reporter | 基于 analyst 的指标数据，生成健康仪表盘、知识分布热力图、投资建议。**只做报告生成，不直接运行 git 命令。** |
| reviewer | 审查 reporter 的所有输出，确保使用"投资建议"而非"绩效评估/问责"框架。标注所有违规措辞并提供替代表述。**只做语言框架审查，不修改数据结论。** |

## 阶段零：Git 数据提取

### 步骤 1：启动 analyst

Team lead 启动 analyst，指示其采集以下 git 数据（限定在 `--period` 指定的时间范围内）：

- **提交统计**：每位作者的提交数量、新增行数、删除行数
- **文件归属模式**：每个文件/目录的主要贡献者及其提交占比
- **评审参与**：PR 评审者分布、评审频次（如果使用 GitHub/GitLab）
- **活跃时间线**：每位贡献者的首次和最近提交时间
- **模块级统计**：按顶层目录分组的贡献者分布

### 步骤 2：数据清洗

Analyst 对原始数据进行清洗：
- 过滤机器人提交（dependabot、renovate、CI bot 等，通过作者名/邮箱识别）
- 过滤自动生成文件（lock 文件、编译产物、vendor 目录等）
- 合并同一人的多个 git 身份（如果 `.mailmap` 存在则使用）

### 步骤 3：评估分析范围

Team lead 根据 analyst 的初步数据评估：
- 如果用户指定了 `--module`，聚焦该模块
- 如果未指定：
  - 小型项目（<10 位贡献者）→ 全量分析
  - 中型项目（10-50 位贡献者）→ 全量分析，高亮核心贡献者
  - 大型项目（>50 位贡献者）→ 按模块分组分析，识别核心团队

**标准模式**：向用户展示项目概况 + 贡献者总数 + 分析范围建议，AskUserQuestion 确认
**单轮确认模式**：team lead 自行决定，收尾汇总时说明
**完全自主模式**：自动决策，不询问用户

## 阶段一：多维分析

### 步骤 4：巴士因子计算

Analyst 对每个模块（顶层目录）计算巴士因子：
- 统计周期内每位贡献者对该模块的提交占比
- 巴士因子 = 提交占比 >10% 的贡献者数量
- 评分标准：
  - 1 人 = **高风险**（单点故障）
  - 2 人 = **中风险**（需要关注）
  - 3+ 人 = **健康**

### 步骤 5：知识孤岛识别

Analyst 识别知识孤岛：
- 扫描所有模块，找出单人提交占比 >80% 的模块
- 按风险程度排序（占比越高、模块越核心，风险越大）
- 记录孤岛持有者和模块重要性（基于代码量和被依赖程度）

### 步骤 6：负载均衡分析

Analyst 计算工作负载基尼系数：
- 统计每位贡献者在周期内的总提交数
- 计算基尼系数（0 = 完全均衡，1 = 一人承担所有工作）
- 基准参考：
  - 0.0 - 0.3 = **均衡**
  - 0.3 - 0.5 = **轻度不均**
  - 0.5 - 0.7 = **中度不均**
  - 0.7 - 1.0 = **严重不均**

### 步骤 7：评审瓶颈分析

Analyst 分析评审模式（如有 PR 数据）：
- 每位评审者的平均评审响应时间
- 评审任务分布是否均衡
- 识别评审瓶颈（单一评审者负担过重）

**如果无法获取 PR/评审数据**：analyst 标注"该指标不可用（无 PR 数据）"，不阻塞流程。

### 步骤 8：上下文切换分析

Analyst 计算每位开发者的聚焦度：
- 按周统计每位开发者涉及的不同模块数量
- 计算平均值（焦点分数）
- 评分标准：
  - 1-2 模块/周 = **高度聚焦**
  - 3-4 模块/周 = **适度聚焦**
  - 5+ 模块/周 = **碎片化**（上下文切换过多）

### 步骤 9：收集指标

Analyst 完成所有指标计算后，向 team lead 发送完整指标报告。Team lead 确认收到后，进入阶段二。

### 步骤 12：检查熔断条件

如果异常指标超过 50%（大多数指标处于"高风险"或"严重不均"状态）：
- **必须暂停**，team lead 向用户报告情况
- 可能原因：项目处于特殊阶段（重构期、冲刺期）、分析周期不合适
- 建议：调整分析周期或聚焦特定模块

异常指标 < 50%：继续下一阶段。

## 阶段三：框架审查

### 步骤 13：启动 reviewer

Team lead 启动 reviewer，将 reporter 的完整健康报告传递给 reviewer。

### 步骤 14：Reviewer 审查语言框架

Reviewer 逐项审查报告中的所有文字，确保符合"投资建议"框架：

| 违规类型 | 违规示例 | 正确表述 |
|---------|---------|---------|
| 问责性措辞 | "开发者X是瓶颈" | "模块 auth/ 将受益于交叉培训——目前有1位拥有深度知识的贡献者" |
| 绩效评判 | "团队成员Y提交量低" | "考虑在 payment/ 模块上进行结对编程以传播知识" |
| 负面标签 | "知识被Z垄断" | "模块 api/ 有集中投资的机会——增加第二位熟悉该模块的开发者将显著降低风险" |
| 比较性排名 | "A的贡献远超B" | "团队在 core/ 模块有深厚的专业知识积累，可以考虑将这些知识扩展到更多成员" |

Reviewer 输出：
1. **通过项清单**：符合投资语言框架的条目
2. **违规项清单**：违规措辞及建议替代表述
3. **审查通过率**：通过项 / 总项 × 100%

### 步骤 15：检查审查熔断

如果违规项 ≥ 3 项：
- **必须暂停**，team lead 要求 reporter 根据 reviewer 的建议修正报告
- Reporter 修正后重新提交 reviewer 审查
- 直到违规项 < 3 项

违规项 < 3 项：
- **标准模式**：team lead 向用户展示报告摘要，AskUserQuestion 确认
- **单轮确认模式**：必须经用户确认
- **完全自主模式**：自动决策，不询问用户

## 核心原则

- **投资视角**：所有输出必须以"投资建议"框架呈现，绝不使用"绩效评估"或"问责"语言
- **数据驱动**：所有结论基于 git 历史的客观数据，不做主观判断
- **职责分离**：analyst 只做数据采集和计算，reporter 只做报告生成，reviewer 只做语言审查
- **隐私尊重**：不对个人进行排名或比较，聚焦于模块和团队层面的投资机会
- **局限透明**：在报告中明确说明 git 数据作为知识代理的已知局限性

---

## 数据质量说明

Git 作者数据是知识分布的代理指标，存在已知局限：
- Squash 合并将所有工作归因于合并者
- 结对编程只显示一位作者
- 机器人提交（dependabot、CI）应被过滤
V1 版本按原始 git 作者处理。未来版本可能支持作者映射配置。

## 健康指标定义

| 指标 | 定义 | 评分标准 |
|------|------|---------|
| **巴士因子** | 每个模块中提交占比 >10% 的贡献者数量 | 1 = 单一贡献者（高风险），3+ = 健康 |
| **知识孤岛** | 单人提交占比 >80% 的模块 | 存在即需关注 |
| **负载基尼系数** | 工作量分布均匀度（0=完全均衡，1=一人承担所有） | 0-0.3 均衡，0.3-0.5 轻度不均，0.5-0.7 中度不均，0.7+ 严重不均 |
| **评审响应时间** | 每位评审者的平均 PR 评审响应时间 | 视团队规范而定 |
| **焦点分数** | 每位开发者每周涉及的不同模块平均数 | >5 = 碎片化 |

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| Git 历史为空或周期内无提交 | 提示用户调整 `--period` 参数，扩大分析范围 |
| 无法获取 PR/评审数据 | 标注"评审指标不可用"，跳过评审相关分析，不阻塞流程 |
| 贡献者数量过少（<2 人） | 跳过负载均衡和巴士因子分析，报告说明原因 |
| `.mailmap` 不存在导致作者身份重复 | 在报告中提示"建议配置 .mailmap 合并重复身份" |
| 异常指标超过 50% | 触发熔断，暂停问用户确认分析周期是否合适 |
| 语言审查违规 ≥ 3 项 | 触发熔断，reporter 修正后重新审查 |
| 项目过大（>100 位贡献者） | 按模块分组分析，聚焦核心团队 |
| 指定的 `--module` 路径不存在 | 提示用户检查路径，列出可用的顶层目录 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前阶段恢复 |
