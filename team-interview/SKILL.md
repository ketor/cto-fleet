**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--role=backend|frontend|fullstack|devops|sre`：目标岗位（默认 `fullstack`）
- `--level=junior|mid|senior|staff`：候选人级别（默认 `mid`）
- `--count=N`：题目数量（默认 `5`）

解析后将标志从项目路径或描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 代码库分析确认 + 题目内容确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终题目集确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **代码库分析有疑问** → team lead 根据项目结构自行判断，在最终输出中说明
- **两位 designer 出题方向重叠** → team lead 去重合并后裁决，收尾时汇总
- **重叠超过 50%** → **不可跳过，必须暂停问用户**（熔断机制）
- **reviewer 判定题目有重大问题（泄露敏感信息/无法在面试时间内完成/评分标准模糊）** → **不可跳过，必须暂停问用户**（熔断机制，单轮确认模式和完全自主模式均适用）

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效——触发熔断条件时是唯一会暂停询问用户的情况。
- **项目过大无法完整分析** → analyst 识别核心模块，聚焦高价值面试场景

岗位与级别说明：

| 岗位 | 侧重领域 |
|------|---------|
| `backend` | 后端架构、API 设计、数据存储、并发处理、性能优化 |
| `frontend` | 前端架构、组件设计、状态管理、性能优化、用户体验 |
| `fullstack` | 全栈覆盖，前后端均衡 |
| `devops` | CI/CD、基础设施、容器化、监控告警、自动化运维 |
| `sre` | 可靠性工程、故障排查、性能调优、容量规划、事故响应 |

| 级别 | 题目难度与考察深度 |
|------|-----------------|
| `junior` | 基础实现能力、代码风格、简单模块开发、常见 bug 定位 |
| `mid` | 模块设计能力、跨模块数据流理解、性能基础优化、Code Review 能力 |
| `senior` | 系统设计能力、架构 trade-off 分析、复杂问题拆解、技术决策论证 |
| `staff` | 跨系统架构、技术战略、大规模重构方案、团队技术规范制定 |

使用 TeamCreate 创建 team（名称格式 `team-interview-{YYYYMMDD-HHmmss}`，如 `team-interview-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  代码库分析 → analyst 全面分析项目模式/架构/复杂度热点 → 输出面试场景素材
         ↓
阶段一  双路出题 → designer-1 (编码题+调试题) + designer-2 (系统设计题+架构讨论题) 独立设计
         ↓
阶段二  审查与评分标准 → reviewer 验证题目公平性/可行性/安全性 → 生成评分细则
         ↓
阶段三  输出面试题包 → 整合题目集 + 评分标准 + 面试官指南 + 候选人评估表
```

## 角色定义

| 角色 | 职责 |
|------|------|
| analyst | 深入分析代码库，提取架构模式、数据流、常用抽象、复杂度热点，识别适合面试的典型场景。输出结构化的面试场景素材报告。**只做模式提取和场景识别，不出题。** **阶段零完成后关闭。** |
| designer-1 | 基于 analyst 的素材，设计编码挑战题和调试场景题。编码题从实际代码模式中简化/脱敏而来，调试题基于真实 bug 模式（脱敏）。**独立出题阶段不与 designer-2 交流。** **阶段二审查完成后关闭（步骤 9 确认后）。** |
| designer-2 | 基于 analyst 的素材，设计系统设计题和架构讨论题。系统设计题基于实际架构的扩展/类比场景，架构讨论题基于真实技术债务（脱敏）。**独立出题阶段不与 designer-1 交流。** **阶段二审查完成后关闭（步骤 9 确认后）。** |
| reviewer | 审查所有题目——验证公平性、面试时间可行性、评分标准清晰度、信息安全（无敏感信息泄露）。为每道题创建评分细则和参考答案。**只做审查和评分标准制定，不直接阅读代码，不自行出题。** 发现重大问题必须标注为"待修正"升级 team lead。**阶段二审查完成后关闭（步骤 9 确认后）。** |

## 阶段零：代码库分析

### 步骤 1：启动 analyst

Team lead 启动 analyst，指示其全面分析代码库以提取面试场景素材。**具体操作方法**：

- **目录结构**（顶层目录布局、最大深度 3 层）：`Bash("ls -la")` 顶层浏览 + `Bash("find . -maxdepth 3 -type d | head -100")` 获取目录树
- **入口文件**：`Glob("**/main.*", "**/index.*", "**/app.*", "**/server.*")` + `Grep("func main|if __name__|module\.exports|export default", head_limit=20)`
- **核心业务模块**：根据目录结构识别核心模块，`Read` 关键文件了解模块职责和接口
- **架构模式**：`Grep("interface|abstract class|trait|protocol|@Component|@Service|@Controller", head_limit=50)` 识别核心抽象
- **数据流路径**：`Grep("import|require|from.*import|use ", head_limit=50)` 追踪模块间依赖关系
- **复杂度热点**：`Bash("find . -name '*.go' -o -name '*.py' -o -name '*.ts' -o -name '*.java' | xargs wc -l | sort -rn | head -20")` 识别大文件
- **错误处理模式**：`Grep("catch|except|recover|Error|panic|throw", head_limit=30)` 分析错误处理策略
- **测试模式**：`Glob("**/test/**", "**/tests/**", "**/*_test.*", "**/*.test.*", "**/*.spec.*")` + 阅读典型测试用例了解测试风格
- **配置和构建**：`Glob("**/Makefile", "**/docker-compose*.yml", "**/package.json", "**/go.mod", "**/Cargo.toml", "**/pyproject.toml")`
- **已知技术债务**：`Grep("TODO|FIXME|HACK|XXX|DEPRECATED", head_limit=30)` 识别改进空间

### 步骤 2：Analyst 输出面试场景素材报告

Analyst 输出**面试场景素材报告**，包含：
- 项目技术栈概述（语言、框架、核心库）
- 架构模式总结（分层/微服务/事件驱动等）及判断依据
- 核心模块列表及职责描述
- 关键数据流路径（3-5 条典型路径）
- 复杂度热点（代码量大/逻辑复杂的模块）
- 典型设计模式和抽象（适合出编码题的素材）
- 已识别的技术债务和架构改进空间（适合出架构讨论题的素材）
- 错误处理和边界情况模式（适合出调试题的素材）
- 系统扩展性分析（适合出系统设计题的素材）
- **场景推荐列表**：按 `--role` 和 `--level` 筛选后，推荐 N×2 个候选面试场景（N 为 `--count`，多推荐以备筛选）

**如果某类信息无法识别**：analyst 标注"未检测到"，不阻塞流程。

### 步骤 3：确认分析结果

**标准模式**：向用户展示面试场景素材概要，AskUserQuestion 确认分析结果是否准确、是否有特别想考察的方向
**单轮确认模式**：team lead 自行确认，收尾汇总时说明
**完全自主模式**：自动决策，不询问用户

确认后关闭 analyst。

## 阶段一：双路出题

### 步骤 4：启动 designer-1 和 designer-2

两者并行启动。Team lead 将 analyst 的面试场景素材报告分发给两位 designer，同时告知 `--role`、`--level`、`--count` 参数。

**题目数量分配**：
- designer-1 负责编码挑战题 + 调试场景题，共出 `ceil(count / 2)` 道
- designer-2 负责系统设计题 + 架构讨论题，共出 `floor(count / 2)` 道
- 如 `--count=5`，则 designer-1 出 3 道、designer-2 出 2 道

**designer-1（编码挑战 + 调试场景）独立设计**，每道题输出：

1. **题目类型**：编码挑战 / 调试场景
2. **题目名称**：简短描述性名称
3. **题目描述**：完整的题面（已脱敏，不含真实类名/服务名/业务逻辑）
4. **来源场景**：基于代码库中哪个模式/问题简化而来（仅供内部参考，不对候选人展示）
5. **考察要点**：本题主要考察的技术能力（2-4 个要点）
6. **预期解题思路**：优秀候选人的解题路径
7. **时间估计**：建议作答时间
8. **难度适配**：当前 `--level` 的预期表现 + 针对更高/更低级别的追问方向
9. **代码模板**（如适用）：提供给候选人的起始代码框架
10. **测试用例**（如适用）：验证答案正确性的测试样例

**designer-2（系统设计 + 架构讨论）独立设计**，每道题输出：

1. **题目类型**：系统设计 / 架构讨论 / Code Review
2. **题目名称**：简短描述性名称
3. **题目描述**：完整的题面（已脱敏，基于实际架构的扩展/类比场景）
4. **来源场景**：基于代码库中哪个架构/技术债务演变而来（仅供内部参考）
5. **考察要点**：本题主要考察的技术能力（2-4 个要点）
6. **预期讨论方向**：优秀候选人会涉及的讨论维度
7. **时间估计**：建议讨论时间
8. **难度适配**：当前 `--level` 的预期表现 + 针对更高/更低级别的追问方向
9. **参考架构图**（如适用）：辅助题面理解的简化架构图
10. **引导问题清单**：面试官逐步引导候选人深入的问题序列

**Team lead 必须确保 designer-1 和 designer-2 不互相看到对方的题目。**

### 步骤 5：收集题目

两者完成后各自向 team lead 发送题目。Team lead 确认收到全部题目后，进入阶段二。

## 阶段二：审查与评分标准

### 步骤 6：Team lead 整合题目

Team lead 将两位 designer 的题目合并为完整题目集：
- 检查题目间是否有重叠（考察同一个知识点/使用同一个场景）
- 检查题型分布是否合理（编码/设计/调试/架构比例）
- 检查整体难度梯度是否匹配 `--level`
- 如有重叠，标注为"待去重"

合并时计算**覆盖度**：
- 考察维度覆盖 → "已覆盖"
- 考察维度缺失 → "未覆盖"
- 题目场景重叠 → "重叠"

**熔断检查**：如果重叠率 > 50%（超过半数题目考察同一维度或基于同一场景）：
- **必须暂停**，team lead 向用户报告情况
- 可能原因：代码库模式过于单一、`--count` 过大超出代码库可支撑的题目多样性
- 建议：减少题目数量或扩大考察范围

### 步骤 7：启动 reviewer 审查

Team lead 启动 reviewer，将以下内容传递：
- 完整题目集
- Analyst 的面试场景素材报告
- `--role`、`--level`、`--count` 参数
- 重叠/去重标注（如有）

Reviewer 审查维度：

| 审查维度 | 检查内容 |
|---------|---------|
| **信息安全** | 题目是否包含真实类名、服务名、业务逻辑、专有算法等敏感信息 |
| **公平性** | 题目是否需要项目内部知识才能作答，外部候选人是否能公平作答 |
| **时间可行性** | 题目是否能在建议时间内完成，难度是否匹配 `--level` |
| **评分清晰度** | 考察要点是否明确、是否有客观可衡量的评分维度 |
| **题型平衡** | 编码/设计/调试/架构比例是否合理 |
| **追问深度** | 追问问题是否能有效区分不同级别候选人 |

Reviewer 输出：
1. **审查通过题**：确认无问题的题目
2. **建议修改题**：小问题，附修改建议（如措辞不够清晰、时间估计偏差）
3. **重大问题题**：严重问题（泄露敏感信息/根本无法在面试时间完成/评分无法客观量化），标注为"待修正"

Reviewer 同时为每道审查通过/建议修改的题目创建**评分细则**：

| 评分维度 | 1 分（不达标） | 2 分（部分达标） | 3 分（达标） | 4 分（优秀） | 5 分（卓越） |
|---------|--------------|----------------|------------|------------|------------|
| [维度1] | [描述] | [描述] | [描述] | [描述] | [描述] |
| [维度2] | [描述] | [描述] | [描述] | [描述] | [描述] |

**熔断检查**：如果 reviewer 标注重大问题 >= 3 题：
- **必须暂停**，team lead 向用户报告，题目集可能存在系统性问题
- 建议：重新分析代码库或调整出题方向

### 步骤 8：处理审查反馈

Team lead 处理 reviewer 的反馈：
- **建议修改题**：team lead 直接修正题目措辞/时间估计等
- **重大问题题**（未触发熔断）：将问题发回对应 designer 要求修正，designer 修正后 team lead 更新题目集
- **重叠题**：
  - **标准模式**：向用户展示重叠情况，AskUserQuestion 让用户选择保留哪道或指定替换方向
  - **单轮确认模式**：team lead 综合判断后裁决
  - **完全自主模式**：自动决策，不询问用户

### 步骤 9：用户确认题目集

**标准模式**：向用户展示题目集摘要：
- 题目列表（名称 + 类型 + 时间估计 + 考察要点）
- Reviewer 审查结果概要（通过率、修改项数、重大问题数）
- 题型分布和考察维度覆盖情况
- AskUserQuestion 确认是否可以生成最终面试题包

**单轮确认模式**：team lead 自行确认，进入下一阶段
**完全自主模式**：自动决策，不询问用户

确认后关闭 designer-1、designer-2、reviewer。

## 题型详细说明

### 1. 编码挑战题

从代码库中实际的代码模式简化、脱敏而来的编程题。

**设计原则**：
- 基于代码库中真正出现过的问题，而非纯算法题
- 简化到可在 20-40 分钟内完成
- 提供代码模板和测试用例
- 考察实际工程能力而非刷题技巧

**脱敏要求**：
- 替换所有真实类名、函数名、变量名
- 替换业务领域术语为通用领域
- 保留核心模式和复杂度，改变具体上下文

### 2. 系统设计题

基于实际架构的扩展或类比场景——"设计一个类似 X 但用于 Y 的系统"。

**设计原则**：
- 基于代码库的真实架构，但改变业务场景
- 考察候选人的系统思维和 trade-off 分析能力
- 提供足够的约束条件（QPS、数据量、延迟要求等）
- 有多个合理的设计方案，无唯一正确答案

**脱敏要求**：
- 将真实业务场景替换为类似但不同的领域
- 保留架构复杂度和约束条件
- 不暴露真实的系统规模和性能指标

### 3. Code Review 题

脱敏后的真实 PR/代码片段，让候选人进行代码审查。

**设计原则**：
- 选取包含典型问题的代码片段（性能、安全、可维护性等）
- 包含好的实践和需要改进的部分
- 考察候选人的代码审查能力和沟通能力

**脱敏要求**：
- 替换所有业务相关的命名和注释
- 保留代码结构和问题模式
- 移除所有 git 历史信息和作者信息

### 4. 调试场景题

基于真实 bug 模式（脱敏）的故障排查题。

**设计原则**：
- 基于代码库中实际出现过的 bug 类型
- 提供症状描述和相关代码/日志
- 考察系统性排查能力，而非碰运气

**脱敏要求**：
- 替换所有服务名、日志格式中的真实信息
- 保留错误模式和排查路径
- 构造合理的虚拟环境背景

### 5. 架构讨论题

基于真实技术债务——"如何改进 X？"的开放讨论题。

**设计原则**：
- 基于代码库中真实存在的技术改进空间
- 考察候选人的架构判断力和技术视野
- 没有标准答案，关注思考过程和论证能力

**脱敏要求**：
- 描述架构现状时使用通用术语
- 不暴露具体的技术债务细节
- 保留核心矛盾和 trade-off

## 安全准则

所有面试题**必须**遵守以下安全规则：

| 规则 | 说明 |
|------|------|
| **类名脱敏** | 不使用代码库中的真实类名、接口名、函数名 |
| **服务名脱敏** | 不使用真实的微服务名称、数据库名、队列名 |
| **业务逻辑脱敏** | 不暴露真实的业务规则、定价逻辑、核心算法 |
| **数据脱敏** | 不使用真实的数据样本、表结构中的业务字段名 |
| **架构模式保留** | 保留设计模式和架构复杂度，仅替换具体细节 |
| **规模信息脱敏** | 不暴露真实的 QPS、数据量、用户规模等精确数字 |
| **内部工具脱敏** | 不提及内部开发的工具名称和专有系统 |

## 核心原则

- **源于实际**：所有题目基于真实代码库模式，而非凭空编造，确保考察内容与实际工作高度相关
- **双路独立**：designer-1 和 designer-2 从不同题型维度独立出题，确保题型多样性和视角差异
- **职责分离**：analyst 只做模式提取，designer 只做出题，reviewer 只做审查和评分标准，不交叉职责
- **安全第一**：所有题目必须严格脱敏，不泄露任何敏感信息
- **公平可评**：题目不依赖内部知识，评分标准客观可量化，不同面试官使用同一题目能得出一致评分
- **级别适配**：题目难度、追问深度、评分标准均根据 `--level` 调整
- **并行高效**：两位 designer 并行出题，最大化效率
- **熔断保护**：重叠率 > 50% 或重大问题 >= 3 题必须暂停问用户

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 项目代码量过小，无法支撑 N 道题 | Analyst 向 team lead 报告，建议减少 `--count` 或补充通用题型 |
| 代码库模式过于单一 | Designer 扩展出题范围，允许基于架构的合理推演出题 |
| 两位 designer 题目重叠超过 50% | 触发熔断，暂停问用户确认 |
| Reviewer 标注重大问题 >= 3 题 | 触发熔断，暂停问用户确认是否存在系统性问题 |
| 题目无法在建议时间内完成 | Reviewer 标注，designer 简化题目或拆分为子题 |
| 脱敏不充分 | Reviewer 标注为重大问题，必须修正后才能通过 |
| 项目使用非常规技术栈 | Analyst 基于代码内容推断模式，designer 出题时提供足够技术背景 |
| 候选人级别与代码库复杂度不匹配 | Designer 调整题目难度——junior 可简化场景，staff 可扩展场景 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前阶段恢复 |
