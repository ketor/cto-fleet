**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从 bug 描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | Bug 分析确认、根因确认、修复方案确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅修复方案确认（因为修复有风险） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **Bug 分析不确定** → team lead 自行判断，在修复方案确认时告知用户判定结果
- **analyzer-1/analyzer-2 根因分歧** → team lead 综合两方分析裁决，在根因确认时附带分歧摘要和裁决理由
- **修复后验证失败** → fixer 自行调整修复方案重试
- **分歧度 > 50%** → **不可跳过，必须暂停问用户**（熔断机制）
- **修复后验证失败超 3 轮** → **不可跳过，必须暂停问用户**（熔断机制）

完全自主模式下：所有节点（包括 Bug 分析确认、根因确认、修复方案确认等）均自动决策，不询问用户。收尾时汇总所有自动决策。唯一例外：验证失败超 3 轮仍自动停止（但不询问用户，而是 team lead 自行裁决是否继续）。

使用 TeamCreate 创建 team（名称格式 `team-debug-{YYYYMMDD-HHmmss}`，如 `team-debug-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  Bug 分析 → team lead 解析 bug 描述 → 提取症状/触发条件/影响范围 → 定位相关代码区域
         ↓
阶段一  复现确认 → reproducer 构建复现步骤/脚本 → 确认 bug 可复现
         ↓
阶段二  双路根因分析 → analyzer-1（代码路径）+ analyzer-2（数据流）独立分析
         → team lead merge 两份分析
         ↓
阶段三  根因确认 → 共识/分歧/仲裁 → 输出确认的根因和证据链
         ↓
阶段四  修复实施 → fixer TDD 修复（先写回归测试确认失败 → 修复代码 → 确认通过）+ 自检
         ↓
阶段五  验证收尾 → verifier 全量测试 + 复现脚本验证 + 边界条件检查
         → team lead 总结 + 清理
```

## 角色定义

| 角色 | 职责 |
|------|------|
| reproducer | 阅读 bug 描述，分析相关代码，构建复现步骤/脚本，确认 bug 可复现。**不修复代码，只做复现。** |
| analyzer-1 | 独立阅读相关代码和日志，**从代码路径角度**分析根因：调用链追踪、条件分支覆盖、异常处理路径。输出根因假设和证据链。**独立工作阶段不与 analyzer-2 交流。** |
| analyzer-2 | 独立分析，**从数据流/状态变化角度**分析根因：输入数据转换、中间状态变化、并发/时序问题、资源生命周期。输出根因假设和证据链。**独立工作阶段不与 analyzer-1 交流。** |
| fixer | 根据确认的根因编写修复代码和回归测试（TDD），执行自检。**只修复确认的根因，不做额外重构。** |
| verifier | 运行全量测试 + 复现脚本确认修复，检查是否引入回归，验证边界条件和相关场景。**不修改代码，只做验证。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| reproducer | 阶段一（步骤 3） | 阶段一完成后（步骤 4 复现成功） | 复现脚本交付后即释放 |
| analyzer-1 | 阶段二（步骤 5） | 阶段三完成后（步骤 8 根因确认） | 根因确认后释放 |
| analyzer-2 | 阶段二（步骤 5） | 阶段三完成后（步骤 8 根因确认） | 根因确认后释放 |
| fixer | 阶段四（步骤 10） | 阶段五验证通过后（步骤 13） | 验证失败时需保留用于迭代修复 |
| verifier | 阶段五（步骤 12） | 阶段五验证通过后（步骤 13） | 验证失败时需保留用于迭代验证 |

## 阶段零：Bug 分析

### 步骤 1：解析 Bug 描述

Team lead 解析用户提供的 bug 描述或 issue 链接，提取以下关键信息：

1. **症状描述**：具体的错误表现（错误消息、异常行为、崩溃现象）
2. **触发条件**：什么操作/输入/场景下触发（如有）
3. **影响范围**：受影响的功能模块、用户群体、数据范围
4. **环境信息**：运行环境、版本、配置（如有）
5. **已知线索**：用户提供的日志片段、错误堆栈、相关 commit（如有）

### 步骤 2：定位相关代码区域

Team lead 根据 bug 描述初步定位可能相关的代码文件和模块：
- 根据错误消息/堆栈反查源码位置
- 根据功能描述定位对应的模块和入口
- 识别项目技术栈、测试框架、构建工具
- 检查相关模块的最近变更（git log）作为参考线索

输出结构化的 Bug 分析报告，包含：
- 症状-触发条件-影响范围三元组
- 初步定位的代码区域（文件路径列表）
- 可能相关的技术栈和依赖
- 已知线索汇总

确认方式：
- **标准模式**：向用户展示 Bug 分析报告，AskUserQuestion 确认分析方向是否正确
- **单轮确认模式**：team lead 自行确认，在修复方案确认时一并展示
- **完全自主模式**：自动决策，不询问用户

## 阶段一：复现确认

### 步骤 3：启动 reproducer

启动 reproducer，传入 Bug 分析报告和相关代码区域信息。

Reproducer 执行以下工作：

1. **阅读相关代码**，理解功能的正常行为和预期流程

2. **构建复现脚本**：按以下分类方法编写具体的复现脚本：

   **单元级 bug（函数/方法级别）**：
   - 识别项目测试框架（Go: `go test`，Python: `pytest`，JS: `jest`/`vitest`，Rust: `cargo test`）
   - 在对应模块的测试目录下创建复现测试文件，命名为 `test_repro_<bug简述>.{ext}`
   - 编写一个精确触发 bug 的测试用例：构造触发输入 → 调用目标函数 → 断言预期行为（当前会失败）
   - 示例结构：`func TestRepro_NilPointerWhenInputEmpty(t *testing.T) { result := targetFunc(nil); assert(result != nil) }`

   **集成级 bug（跨模块/服务级别）**：
   - 创建独立复现脚本 `repro_<bug简述>.sh`（或 `.py`），放在项目根目录
   - 脚本结构：① 设置前置条件（启动服务/准备数据） → ② 执行触发操作（curl/CLI 命令/API 调用） → ③ 检查结果并输出 PASS/FAIL
   - 脚本必须自包含，运行后输出明确的 `[REPRO SUCCESS] Bug confirmed` 或 `[REPRO FAILED] Bug not triggered`

   **配置/环境级 bug**：
   - 创建 `repro_env_<bug简述>.md` 文档，记录：精确的环境变量设置 → 配置文件内容 → 操作步骤序列 → 预期 vs 实际行为
   - 如可能，同时创建脚本自动检测环境配置是否命中 bug 条件

3. **执行复现**：运行复现脚本，确认 bug 存在。记录完整的命令输出。

4. **记录复现结果**：输出结构化复现报告，包含：
   - 复现脚本路径和运行命令
   - 实际输出（完整错误信息/日志）
   - 预期输出
   - 触发条件摘要（输入数据、环境配置、操作序列）
   - 复现稳定性（每次必现 / 偶发 + 触发概率）

### 步骤 4：复现结果处理

| 复现结果 | 处理方式 |
|---------|---------|
| **成功复现** | 输出复现脚本和结果，进入阶段二 |
| **偶发复现**（多次尝试偶尔触发） | 记录触发概率和条件差异，标记为"偶发"后继续分析 |
| **无法复现** | Reproducer 尝试调整触发条件（输入数据、并发度、环境配置）最多 3 轮。仍无法复现则升级 team lead，由 team lead 用 AskUserQuestion 向用户获取更多信息 |

复现脚本将在后续阶段作为验证基准使用。

## 阶段二：双路根因分析

### 步骤 5：启动 analyzer-1 和 analyzer-2

同时启动两位 analyzer，分别传入 Bug 分析报告、复现脚本和复现结果。**两位 analyzer 独立工作，team lead 不让两位 analyzer 看到对方的分析，确保独立性。**

**analyzer-1（代码路径分析）**：
- 从触发点入手，追踪完整的代码调用链（函数调用栈、跨模块调用）
- 标记每个分支节点的条件和执行路径
- 检查异常处理路径是否有遗漏或错误（catch/finally、defer/cleanup）
- 对比正常路径和异常路径的差异，找出偏离点
- 检查最近的代码变更（git blame/log）是否引入了问题
- 输出：根因假设 + 代码级证据链（具体文件、行号、逻辑缺陷描述）

**analyzer-2（数据流/状态分析）**：
- 追踪从输入到输出的完整数据转换链路
- 检查每个环节的数据类型、格式、范围是否符合预期
- 检查中间状态的变化是否符合预期（状态机转换、缓存一致性）
- 分析并发/时序相关的状态竞争问题（锁粒度、原子性、happen-before 关系）
- 检查资源（内存、连接、文件句柄）的生命周期管理
- 输出：根因假设 + 数据流级证据链（状态变化序列、数据变形点、竞态窗口描述）

### 步骤 6：分析质量要求

每位 analyzer 的输出必须包含：
1. **根因假设**：用一句话描述根本原因
2. **证据链**：从症状到根因的完整推理链条，每一步有代码/数据佐证
3. **置信度**：高/中/低，附带理由
4. **影响评估**：该根因可能导致的其他问题（如有）
5. **排除项**：明确排除了哪些可能的原因及排除理由

## 阶段三：根因确认

### 步骤 7：Team lead 合并两份分析

两份分析完成后，team lead 执行 merge：

1. **对比两份根因假设**，计算共识度和分歧度：

**共识度计算公式**：
```
对比维度（共 5 项，每项 20%）：
  ① 根因位置（指向相同文件/函数）
  ② 根因类型（同类缺陷，如都认为是空指针/竞态/逻辑错误）
  ③ 触发条件（相同的触发路径）
  ④ 影响范围（相同的影响评估）
  ⑤ 修复方向（相似的修复建议）

共识度 = 一致维度数 × 20%
分歧度 = 100% - 共识度
```

| 一致性结果 | 分歧度 | 处理方式 |
|-----------|--------|---------|
| **完全一致** | 0% | 直接采纳为确认根因 |
| **互补型** | < 30%（≤1 项分歧） | 两者发现的是同一问题的不同侧面，合并为完整根因描述 |
| **部分分歧** | 30%-50%（2-3 项分歧） | Team lead 综合证据链判断，选择证据更充分的假设，或融合两者 |
| **严重分歧** | > 50%（≥3 项分歧） | 触发熔断：必须暂停，AskUserQuestion 向用户展示两份分析结果，请用户提供额外信息或判断方向 |

**分歧处理策略**：
- **共识部分**：直接纳入根因报告，标记为"双方共识"
- **互补部分**：一方发现另一方未覆盖的证据，合并纳入，标记为"互补发现"
- **冲突部分**：两方指向不同根因，team lead 比较证据链强度（证据数量 × 证据直接性），采纳更强的一方，标记为"仲裁结论"并记录被否决方的观点
- **盲区检查**：team lead 检查是否存在两方都未覆盖的分析角度，如有则在根因报告中标注为"未覆盖风险"

2. **整合证据链**：从两份分析中提取最有力的证据，形成统一的证据链
3. **输出确认的根因报告**：根因描述 + 统一证据链 + 修复方向建议

### 步骤 8：根因确认

确认方式：
- **标准模式**：向用户展示根因报告（含两份独立分析摘要、merge 结论、分歧处理说明），AskUserQuestion 确认根因正确
- **单轮确认模式**：team lead 自行确认（分歧度 ≤ 50% 时）；分歧度 > 50% 时必须问用户
- **完全自主模式**：自动决策，不询问用户

## 阶段四：修复实施

### 步骤 9：制定修复方案

Team lead 根据确认的根因，制定修复方案：
1. **修复策略**：最小化修改原则——只修复根因，不做额外重构
2. **修改范围**：列出需要修改的具体文件和模块
3. **修复方法**：描述具体的代码修改逻辑（如：添加空值检查、修复条件判断、调整调用顺序）
4. **回归风险评估**：修改可能影响的其他功能点
5. **回归测试范围**：需要编写的测试用例（覆盖根因场景 + 边界条件）
6. **替代方案**（如有）：其他可行的修复方式及其 trade-off

确认方式：
- **标准模式和单轮确认模式均需确认**：向用户展示修复方案（含根因摘要、修改范围、回归风险），AskUserQuestion 确认后再执行修复。这是唯一在单轮确认模式下仍需用户确认的节点（因为代码修改有风险，需要用户知情同意）
- **完全自主模式**：自动决策，不询问用户

### 步骤 10：启动 fixer，TDD 修复

启动 fixer，传入确认的根因报告、修复方案和复现脚本。

**TDD 修复流程**（严格按顺序执行）：

1. **编写回归测试（Red）**：根据根因编写精确的回归测试用例
   - 测试用例必须精确覆盖根因场景（触发条件 → 预期行为）
   - 包含边界条件测试（空值、极端值、并发场景等）
   - 如果 reproducer 已有复现测试用例，在此基础上扩展而非替换
   - 测试命名清晰描述被测场景（如 `test_should_not_panic_when_input_is_nil`）

2. **确认测试失败（Red 验证）**：运行回归测试，确认在未修复状态下测试失败
   - 如果测试直接通过，说明测试未精确覆盖根因，fixer 必须调整测试直到能复现失败
   - 记录失败的错误消息作为基准

3. **编写修复代码（Green）**：实现最小化修复，使回归测试通过
   - 严格按修复方案的修改范围执行
   - 不做超出修复范围的重构或优化

4. **确认测试通过（Green 验证）**：运行回归测试，确认修复有效
   - 如果仍有失败，继续调整修复代码直到全部通过

### 步骤 11：Fixer 自检

修复完成后，fixer 执行自检清单：

1. **回归测试通过**：新增的回归测试全部通过
2. **全量单测通过**：运行项目全量单元测试，无新增失败
3. **Lint/Format 通过**：运行 lint/format 检查（如项目有配置）
4. **修改范围检查**：自查代码修改是否在修复方案的范围内（无超出范围的改动）
5. **无编译警告**：编译无新增警告

自检全部通过后，通知 team lead 和 verifier。自检失败则自行修复，无法修复则升级 team lead。

## 阶段五：验证收尾

### 步骤 12：启动 verifier，全量验证

启动 verifier，传入修复后的代码位置、复现脚本、回归测试和修复方案。

Verifier 执行全量验证：

1. **复现脚本验证**：运行 reproducer 编写的复现脚本，确认 bug 已修复（原来的异常行为不再出现）
2. **回归测试验证**：运行 fixer 编写的回归测试，确认全部通过
3. **全量测试验证**：运行项目全量测试套件，确认无回归
4. **边界条件验证**：针对修复点测试边界情况
   - 空值/零值输入
   - 极端值/溢出场景
   - 并发/时序场景（如与根因相关）
5. **相关功能验证**：根据修复的影响范围，手动验证相关功能是否正常

### 步骤 13：验证结果处理

| 验证结果 | 处理方式 |
|---------|---------|
| **全部通过** | 进入步骤 14 收尾 |
| **复现脚本仍失败** | 修复未生效，打回 fixer 重新分析修复方案 |
| **回归测试失败** | 修复不完整，打回 fixer 补充修复 |
| **全量测试有新增失败** | 修复引入了回归，打回 fixer 修复回归问题 |
| **边界条件发现新问题** | 评估是否属于同一根因，是则打回 fixer 扩展修复；否则创建新 issue 记录 |

**验证失败迭代**：fixer 修复后 verifier 重新验证，循环直到通过。**最大迭代轮次：3 轮。** 超出后触发熔断，team lead 向用户汇报当前状态（已通过/未通过的验证项、失败原因分析），由用户决定继续、调整方向或终止。

## 核心原则

- **复现先行**：不可复现的 bug 不盲目修复，先投入精力复现
- **独立分析**：两位 analyzer 独立工作，避免锚定偏差，确保分析多样性
- **证据驱动**：根因假设必须有代码/数据级证据支撑，不凭直觉
- **最小修复**：只修复根因，不做超出范围的改动，降低引入新问题的风险
- **TDD 修复**：先写回归测试确认失败，再修复确认通过，确保修复有效且可验证
- **全量验证**：修复后不仅验证 bug 本身，还验证全量测试和边界条件
- **分歧即信号**：两路分析分歧度高意味着问题复杂度高，必须谨慎处理
- **有限迭代**：验证失败最多重试 3 轮，超出则暂停问用户
- **不确定就问**：标准模式下关键节点必须用户确认；单轮确认模式下修复方案仍需确认；完全自主模式下全部自动决策

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| Bug 描述信息不足 | Team lead 用 AskUserQuestion 向用户获取更多信息（错误日志、触发步骤、环境信息） |
| 无法复现 bug | Reproducer 调整触发条件重试最多 3 轮；仍无法复现则升级 team lead 向用户获取更多复现线索 |
| 偶发性 bug（概率性触发） | 标记为偶发，analyzer 重点关注并发/时序/资源竞争方向，增加复现尝试次数 |
| 两位 analyzer 根因严重分歧（>50%） | 触发熔断，暂停问用户。用户可提供额外信息或指定分析方向 |
| 修复后回归测试仍不通过 | Fixer 检查测试用例是否正确，重新分析修复方案；3 轮未解决则暂停问用户 |
| 修复引入新的回归失败 | Fixer 回退修改，重新设计修复方案，确保不影响已有功能 |
| 根因涉及第三方依赖/外部系统 | Team lead 通知用户，建议协调外部修复或设计 workaround |
| Fixer TDD 回归测试直接通过（未修复就通过） | 说明测试未精确覆盖根因，fixer 必须调整测试用例直到测试能准确反映 bug |
| 全量测试基础设施不存在 | Verifier 降级为手动验证复现脚本 + 回归测试，跳过全量测试，在总结中标注风险 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| Issue 链接无法访问 | Team lead 用 AskUserQuestion 请用户直接粘贴 bug 描述内容 |
| 项目过大无法完整分析 | Team lead 根据 bug 描述缩小范围，聚焦相关模块；analyzer 仅分析 bug 相关的代码路径和数据流 |
| Git 仓库状态异常（脏工作区/未提交变更） | Team lead 在阶段零检测工作区状态，提示用户先 stash 或提交变更，避免修复代码与未提交变更冲突 |
