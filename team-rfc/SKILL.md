**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--type=feature|refactor|migration|integration`：RFC 类型（默认根据需求自动推断）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 需求分析确认 + 方案评审确认 + RFC 文档确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅 RFC 文档确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **需求边界不明确** → team lead 自行界定范围，在最终文档附录中说明
- **两位 architect 方案分歧** → reviewer 评审后 team lead 综合裁决，收尾时汇总
- **architect 分歧 > 50%** → **不可跳过，必须暂停问用户**（熔断机制）
- **reviewer 否决全部方案** → **不可跳过，必须暂停问用户**（熔断机制）
- **researcher 报告"关键技术信息不足"** → **不可跳过，必须暂停问用户调整需求/补充上下文**（熔断机制）

完全自主模式下：所有节点均自动决策，不询问用户。仅熔断机制（architect 分歧 > 50%、reviewer 否决全部方案、researcher 报告关键技术信息不足）仍然生效，触发时必须暂停。

RFC 类型说明：

| 类型 | 适用场景 | 侧重点 |
|------|---------|--------|
| `feature` | 新功能/新系统设计 | 架构设计、API 设计、数据模型 |
| `refactor` | 已有系统重构 | 现状分析、重构策略、兼容性、灰度方案 |
| `migration` | 技术栈/平台迁移 | 迁移路径、数据迁移、回滚方案、双写策略 |
| `integration` | 系统集成/对接 | 接口协议、认证鉴权、数据映射、错误处理 |

解析后将标志从技术需求描述中移除。

使用 TeamCreate 创建 team（名称格式 `team-rfc-{YYYYMMDD-HHmmss}`，如 `team-rfc-20260311-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

**文件命名规范**：

| 角色输出 | 文件名 |
|---------|--------|
| Scanner 报告 | `scanner-report.md` |
| Reviewer-N 第R轮 | `reviewer-{N}-round-{R}.md` |
| 合并报告第R轮 | `merged-report-round-{R}.md` |
| 根因分组 | `root-cause-groups-round-{R}.md` |
| Fixer 第R轮 | `fixer-round-{R}.md` |
| Tester 第R轮 | `tester-round-{R}.md` |
| Architect-N 方案 | `architect-{N}-design.md` |

| 审查任务T | `review-task-{T}.md` |
| 集成测试第R轮 | `integration-test-round-{R}.md` |
| 最终报告 | `final-report.md` |

> 仅当角色存在于当前 skill 时使用对应命名。未列出的角色用 `{role}-{context}.md` 格式。

## 流程概览

```
阶段零  需求分析 → team lead 解析需求 → 确定 RFC 类型 → 识别约束条件
         ↓
阶段一  技术调研 → researcher 调研技术背景、现有方案、行业最佳实践
         ↓
阶段二  双路方案设计 → architect-1（方案 A）+ architect-2（方案 B）独立并行设计
         ↓
阶段三  方案评审 → 三人组会（architect-1 + architect-2 + reviewer）讨论收敛 → 用户确认
         ↓
阶段四  RFC 文档生成 → writer 基于评审结论生成标准化 RFC 文档
         ↓
阶段五  收尾 → 保存文档 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| researcher | 调研技术背景、现有方案、行业最佳实践、相关技术文档。使用 WebSearch/WebFetch 搜索网络 + Read/Glob/Grep 分析本地代码。输出调研报告（含技术现状、可选技术栈、已有系统分析、竞品方案）。每条信息标注来源。**调研完成后关闭。** |
| architect-1 | 基于调研报告和代码阅读，独立设计完整技术方案（方案 A）。包含：总体架构、技术选型、详细设计、数据模型、API 设计、实施计划。输出完整方案文档 + Mermaid 架构图。**独立工作，不与 architect-2 交流。** |
| architect-2 | 同 architect-1 的职责，独立设计完整技术方案（方案 B）。鼓励从不同角度切入（不同技术选型、不同架构风格、不同权衡取舍）。输出完整方案文档 + Mermaid 架构图。**独立工作，不与 architect-1 交流。** |
| reviewer | 参与三人组会评审，从可行性、风险、成本、时间线、运维复杂度角度挑战两个方案。提出质疑和改进建议。**只做评审和挑战，不修改方案，不做设计决策。** |
| writer | 基于评审后的最终方案和调研报告，生成标准化 RFC 文档。严格按照 RFC 模板格式输出。**不做设计判断，不自行搜索。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| researcher | 阶段一（步骤 4） | 阶段一完成后（步骤 6） | 调研报告交付后即释放 |
| architect-1 | 阶段二（步骤 7） | 阶段三评审确认后（步骤 13） | 需参与三轮评审讨论，评审确认后释放 |
| architect-2 | 阶段二（步骤 7） | 阶段三评审确认后（步骤 13） | 需参与三轮评审讨论，评审确认后释放 |
| reviewer | 阶段三（步骤 10） | 阶段三评审确认后（步骤 13） | 三轮评审完成后释放 |
| writer | 阶段四（步骤 14） | 阶段五收尾前（步骤 19） | RFC 文档确认后释放 |

## 阶段零：需求分析

### 步骤 1：解析技术需求

Team lead 解析 `$ARGUMENTS` 中的技术需求：
- 提取核心技术需求和设计目标
- 检查本地工作目录是否有相关代码和文档（扫描当前目录和 `~/Documents`）
- 确定 RFC 类型（`--type` 参数或根据需求自动推断）
- 确定文档语言（`--lang` 参数或默认中文）

### 步骤 2：识别约束条件

分析需求中的硬约束和软约束：
- **硬约束**：必须满足的技术限制（现有技术栈、兼容性要求、合规要求、性能指标）
- **软约束**：期望满足但可协商的条件（时间线、团队技能、成本预算）
- **非功能需求**：可用性、可观测性、安全性、可扩展性要求

### 步骤 3：确认需求范围

- 如果需求明确 → 直接进入阶段一
- 如果需求模糊或边界不清：
  - **标准模式**：向用户展示需求解读、RFC 类型、约束条件清单，AskUserQuestion 确认
  - **单轮确认模式**：team lead 自行界定范围，收尾时说明
  - **完全自主模式**：自动决策，不询问用户

## 阶段一：技术调研

### 步骤 4：启动 researcher

Team lead 启动 researcher，将以下内容传递：
- 技术需求描述
- RFC 类型
- 约束条件清单
- 本地代码仓库路径（如有）

### 步骤 5：Researcher 执行调研

Researcher 按以下维度进行调研，使用三级搜索策略：

**搜索策略（按优先级执行）**：

| 优先级 | 搜索方式 | 工具 | 适用场景 | 输出要求 |
|--------|---------|------|---------|---------|
| **1. 本地代码分析** | 阅读项目代码和文档 | `Glob`（文件发现）+ `Read`（读取内容）+ `Grep`（搜索模式） | 现有系统分析、技术栈识别、已有架构理解 | 标注文件路径，如 `src/api/router.go:42` |
| **2. 网络搜索** | 搜索技术文档和最佳实践 | `WebSearch`（关键词搜索） | 技术背景、行业最佳实践、技术选型对比、相关案例 | 标注搜索关键词和结果 URL |
| **3. 深度阅读** | 读取特定技术文档/博客 | `WebFetch`（URL 内容获取） | WebSearch 返回的高价值页面、官方文档、技术博客、GitHub README | 标注来源 URL 和提取的关键信息 |

**每个调研维度的具体搜索步骤**：

1. **技术背景**：
   - WebSearch: `"<技术领域> architecture best practices 2025 2026"` → 提取趋势和现状
   - WebFetch: 对 Top 3 搜索结果做深度阅读
   - 输出：技术领域现状概述 + 发展趋势 + 关键技术指标

2. **现有系统分析**（如本地有代码）：
   - Glob: `**/*.{go,py,ts,java,rs}` 识别主要语言 → `**/README*`/`**/docs/**` 查找文档
   - Grep: 搜索入口文件（`func main`/`if __name__`/`createApp`）、路由定义、数据模型
   - Read: 阅读核心模块的代码，理解现有架构和数据流
   - 输出：现有架构图（文字描述） + 技术栈清单 + 数据模型 + API 列表 + 技术债务

3. **行业最佳实践**：
   - WebSearch: `"<技术方向> design pattern production"` + `"<类似系统> architecture"`
   - WebFetch: 阅读官方文档、知名技术博客（如 engineering blog）
   - 输出：最佳实践清单 + 推荐架构风格 + 设计模式

4. **可选技术栈**：
   - WebSearch: `"<技术A> vs <技术B> comparison 2025"` 做多组对比
   - WebFetch: 阅读官方文档获取特性列表和限制
   - 输出：候选技术对比表（功能、性能、生态、学习曲线、社区活跃度）

5. **相关案例**：
   - WebSearch: `"<类似系统> case study"` + `"<技术方向> lessons learned"`
   - WebFetch: 阅读案例详情
   - 输出：案例摘要 + 关键经验教训 + 避坑指南

**搜索终止条件**：每个维度最多搜索 3 轮（每轮 = 1 次 WebSearch + 最多 3 次 WebFetch）。3 轮后仍无有效结果则标注"该方向信息不足，建议用户补充"并继续其他维度。**如果 ≥2 个核心维度标注"信息不足"，触发熔断，暂停问用户补充上下文。**

Researcher 输出调研报告，每条信息必须标注来源（本地路径 / URL / 搜索关键词）。

### 步骤 6：关闭 researcher，分发调研报告

收到调研报告后，关闭 researcher（shutdown_request），准备进入阶段二。

## 阶段二：双路方案设计

### 步骤 7：启动 architect-1 和 architect-2

两者并行启动。Team lead 将以下内容分别发给两位 architect：
- 技术需求描述
- RFC 类型和约束条件
- Researcher 的调研报告
- 本地代码仓库路径（如有，供 architect 自行阅读代码）

**重要**：不暗示设计方向，让两位 architect 独立思考。

### 步骤 8：Architect 独立设计

每位 architect 输出完整方案文档，必须包含：

1. **方案概述**：一段话说清核心思路
2. **总体架构**：系统组件划分、职责边界、交互关系
3. **技术选型**：选用的技术栈及理由
4. **详细设计**：
   - 核心模块设计
   - 数据模型变更（如适用）
   - API 变更（如适用）
   - 关键流程时序
5. **Mermaid 架构图**：至少 1 张架构图 + 1 张关键流程时序图
6. **实施计划**：分阶段里程碑、依赖项、预估工时
7. **风险评估**：已识别风险及缓解措施
8. **回滚方案**：出问题时的回退策略

两位 architect 完成后各自向 team lead 发送方案。

### 步骤 9：方案对比预分析

Team lead 收到两份方案后，进行初步对比：

**方案共识度计算公式**：
```
对比维度（共 6 项，每项约 16.7%）：
  ① 架构风格（单体/微服务/事件驱动/分层等）
  ② 核心技术选型（语言、框架、数据库、中间件）
  ③ 数据模型设计（表结构、数据流、存储方案）
  ④ API 设计风格（REST/gRPC/GraphQL、接口粒度）
  ⑤ 实施路径（分阶段策略、灰度方案、迁移策略）
  ⑥ 风险评估方向（识别的主要风险类型是否一致）

共识度 = 一致维度数 / 6 × 100%
分歧度 = 100% - 共识度
```

- 识别核心分歧点（技术选型差异、架构风格差异、权衡取舍差异）
- 如果分歧 > 50%（≥4 个维度不同）：
  - **必须暂停**，team lead 向用户报告两个方案概要和主要分歧，AskUserQuestion 确认是否继续评审
- 否则：继续进入阶段三

**分歧处理策略**：
- **共识部分**：两方案一致的维度，直接纳入评审基准，标记为"双方共识"
- **互补部分**：一方覆盖另一方未考虑的方面，在评审中要求双方补充
- **冲突部分**：明确标注分歧点，交由 reviewer 在评审中重点质疑
- **盲区检查**：team lead 检查是否存在两方都未考虑的关键维度（如安全性、可观测性、回滚方案），如有则在评审中要求补充

## 阶段三：方案评审

### 步骤 10：启动 reviewer

Team lead 启动 reviewer，将以下内容传递：
- 技术需求描述和约束条件
- Researcher 的调研报告
- Architect-1 的方案 A（标记为"方案 A"）
- Architect-2 的方案 B（标记为"方案 B"）

**重要**：传递时不透露 architect 编号，仅用"方案 A"和"方案 B"标记，避免暗示优先级。

### 步骤 11：三人组会评审

Team lead 组织三轮评审讨论：

**第一轮：Reviewer 提出质疑**

Reviewer 从以下角度逐一评审两个方案：

| 评审维度 | 关注点 |
|---------|--------|
| **可行性** | 技术可行性、团队能力匹配度、依赖项成熟度 |
| **风险** | 技术风险、安全风险、数据一致性风险、第三方依赖风险 |
| **成本** | 开发成本、运维成本、基础设施成本、迁移成本 |
| **时间线** | 实施周期合理性、里程碑可达性、并行度 |
| **运维复杂度** | 部署复杂度、监控难度、故障排查便利性 |
| **可扩展性** | 未来需求适应性、技术栈演进空间 |

Reviewer 输出：每个方案的优势清单、劣势清单、风险清单、改进建议。

**第二轮：Architect 回应质疑**

Team lead 将 reviewer 的质疑分别发给 architect-1 和 architect-2：
- 各自回应针对自己方案的质疑
- 解释设计取舍的理由
- 接受合理建议并说明如何调整
- 可选：吸收对方方案的优点进行改进

**第三轮：收敛讨论**

Team lead 将双方回应发给 reviewer 进行最终评价：
- Reviewer 输出最终评审意见：
  - 推荐方案（A / B / 混合方案）及理由
  - 必须解决的问题清单
  - 可选改进建议

### 步骤 12：检查熔断条件

如果 reviewer 否决全部方案（认为两个方案都不可行）：
- **必须暂停**，team lead 向用户报告评审结论和否决理由
- 建议：调整需求约束或补充调研

否则：继续。

### 步骤 13：方案确认

Team lead 综合评审结果，确定最终方案：
- 如果推荐方案 A 或 B → 采纳推荐方案，融入评审改进建议
- 如果推荐混合方案 → 融合两者优点，明确每部分来源

方案确认：
- **标准模式**：向用户展示推荐方案摘要、评审结论、与备选方案的核心差异，AskUserQuestion 确认
- **单轮确认模式**：team lead 采纳 reviewer 推荐，收尾时汇总决策过程
- **完全自主模式**：自动决策，不询问用户

确认后，关闭 architect-1、architect-2、reviewer。

## 核心原则

- **调研先行**：先充分调研技术背景和现有方案，再进行设计，避免闭门造车
- **双路独立设计**：两位 architect 必须完全独立工作，不互相看到对方方案，确保方案多样性
- **三人评审制**：architect-1 + architect-2 + reviewer 组会评审，确保方案经过充分质疑和讨论
- **职责分离**：researcher 只调研不设计，architect 只设计不评审，reviewer 只评审不修改，writer 只写作不判断
- **标准化输出**：RFC 文档严格遵循模板格式，确保可读性和可追溯性
- **回滚优先**：所有方案必须包含回滚方案，这是生产系统的底线要求
- **收敛而非发散**：评审阶段目的是收敛到一个可执行方案，不是无限讨论

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 需求过于模糊 | Team lead 提取关键信息，标准模式请用户确认需求边界 |
| 网络搜索无有效结果 | Researcher 调整关键词重试，最多 3 轮；仍无结果标注"该方向信息不足" |
| 本地无相关代码 | Researcher 标注"无本地代码"，仅基于网络信息和需求描述 |
| 两位 architect 方案分歧 > 50% | 触发熔断，暂停问用户确认方向偏好 |
| Reviewer 否决全部方案 | 触发熔断，暂停问用户确认是否调整需求约束 |
| Researcher 报告关键技术信息不足 | 触发熔断，暂停问用户补充上下文或调整技术方向 |
| WebFetch 目标页面无法访问 | Researcher 跳过该来源，标注为"无法访问" |
| Architect 方案缺少必要章节 | Team lead 要求 architect 补充完整 |
| 评审讨论无法收敛 | Team lead 列出分歧点和各方理由，升级给用户裁决 |
| RFC 编号冲突 | 检查 `docs/rfc/` 目录已有文件，自动递增编号 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前阶段恢复 |

### 共识度计算

team lead 按五维度评估双路分析的共识度：

| 维度 | 权重 |
|------|------|
| 发现一致性（相同问题/结论） | 20% |
| 互补性（独有但不矛盾的发现） | 20% |
| 分歧程度（直接矛盾的结论） | 20% |
| 严重度一致性（同一问题的严重等级差异） | 20% |
| 覆盖完整性（两路合并后的覆盖面） | 20% |

共识度 = 各维度加权得分之和

- **≥ 60%**：自动合并，分歧项由 team lead 裁决
- **50-59%**：合并但标注分歧，收尾时汇总争议点
- **< 50%**：触发熔断，暂停并向用户确认方向
