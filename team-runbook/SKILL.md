---
name: team-runbook
description: 启动一个运维手册生成团队（scanner/writer×2/reviewer），通过服务架构+告警清单扫描+双写手独立撰写+交叉验证，输出可操作的on-call运维手册并跟踪手册时效性。使用方式：/team-runbook [--auto (全自动，不询问)] [--once (仅确认一次后自动执行)] [--service=服务名] [--update] [--lang=zh|en] 描述或服务范围
argument-hint: [--auto (全自动，不询问)] [--once (仅确认一次后自动执行)] [--service=服务名 (聚焦特定服务)] [--update (更新已有手册而非生成新手册)] [--lang=zh|en] 描述或服务范围
---

**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--service=服务名`：聚焦特定服务（仅扫描和生成该服务的运维手册，跳过全量扫描）
- `--update`：更新模式（对已有手册进行增量更新，而非从头生成）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 固定节点 + 条件节点，全部 AskUserQuestion | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅固定节点（手册内容确认、合并方案确认） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **扫描范围不确定** → scanner 自行判断，在撰写前告知用户扫描结果
- **writer-1/writer-2 内容分歧** → reviewer 综合双路撰写结果裁决，在合并确认时附带分歧摘要和裁决理由
- **扫描期间需要扩大服务范围** → scanner 自行扩展
- **手册内容最终确认** → **不可跳过，必须暂停问用户**（运维手册直接影响 on-call 操作安全）
- **时效性检测发现大量过期** → **不可跳过，必须暂停问用户**（需要用户确认更新优先级）
- **更新模式下的变更确认** → reviewer 自行决策，收尾时汇总

完全自主模式下：所有节点均自动决策，不询问用户。仅熔断机制（手册内容最终确认、时效性检测发现大量过期）仍然生效，触发时必须暂停。

使用 TeamCreate 创建 team（名称格式 `team-runbook-{YYYYMMDD-HHmmss}`，如 `team-runbook-20260311-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 手册严重程度定义（用于排序生成优先级）

| 优先级 | 描述 | 生成策略 |
|--------|------|---------|
| **P0 服务** | 核心服务 / 面向用户的关键路径 / 数据存储层 | 必须生成完整手册，所有告警全覆盖 |
| **P1 服务** | 重要内部服务 / API 网关 / 消息队列 | 生成完整手册，重点告警覆盖 |
| **P2 服务** | 辅助服务 / 监控系统 / 内部工具 | 生成精简手册，仅覆盖关键告警 |
| **P3 服务** | 低优先级服务 / 实验性功能 / 非关键批处理 | 可选生成，列出告警清单即可 |

## 流程概览

```
阶段零  服务与告警盘点 → scanner 扫描服务架构、告警规则、现有手册
         ↓
阶段一  双路撰写 → writer-1 + writer-2 独立撰写运维手册
         ↓
阶段二  交叉验证 → reviewer 对比两份手册，合并最佳实践
         ↓
阶段三  时效性检测 → 对比现有手册与代码变更，标记过期内容
         ↓
阶段四  输出 + 可选提交
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 盘点服务清单、依赖关系、告警规则（从 observability.json 或代码中提取）、现有手册状态。输出服务与告警的结构化清单。**仅在盘点阶段工作，盘点完成后释放。** |
| writer-1 | 独立撰写运维手册——从**架构与依赖角度**（服务概览、依赖拓扑、SLO 定义、降级方案、回滚策略）。**不与 writer-2 交流，独立输出手册草稿。** |
| writer-2 | 独立撰写运维手册——从**告警与运维角度**（告警含义、诊断步骤、修复命令、升级路径、历史故障经验）。**不与 writer-1 交流，独立输出手册草稿。** |
| reviewer | 交叉验证两份手册草稿，合并最佳内容，确保步骤可操作、命令可执行、覆盖完整。输出最终合并版手册。 |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段零（步骤 1） | 阶段零完成后（步骤 2 盘点确认） | 盘点报告交付后即释放 |
| writer-1 | 阶段一（步骤 3） | 阶段二完成后（步骤 6 合并确认） | 如需补充内容则保留到合并结束 |
| writer-2 | 阶段一（步骤 3） | 阶段二完成后（步骤 6 合并确认） | 如需补充内容则保留到合并结束 |
| reviewer | 阶段二（步骤 5） | 阶段四完成后（步骤 10） | 全程保持存活（验证 + 时效性检测） |

## 阶段零：服务与告警盘点

### 步骤 1：启动 scanner 进行服务盘点

如果用户通过 `--service` 指定了服务名，scanner 仅扫描该服务及其直接依赖，跳过全量盘点。

如果指定了 `--update`，scanner 优先扫描 `docs/runbooks/` 下已有手册，对比代码变更。

否则，启动 scanner，执行全量盘点：

1. **服务架构扫描**：
   - 扫描项目目录结构，识别服务模块（微服务目录、Dockerfile、Kubernetes YAML、docker-compose 等）
   - 读取 `~/.gstack/data/{slug}/observability.json`（如存在），提取告警定义
   - 扫描代码中的告警配置（Prometheus rules、Alertmanager config、Grafana alerts 等）
   - 识别服务间依赖关系（API 调用、消息队列、数据库连接等）

2. **现有手册扫描**：
   - 检查 `docs/runbooks/` 目录下已有手册
   - 检查 `~/.gstack/data/{slug}/runbooks/` 目录下的 JSON 元数据
   - 记录每份手册的最后更新时间

3. **告警规则提取**：
   - 从 observability.json 提取告警名称、阈值、严重程度
   - 从 Prometheus/Alertmanager 配置提取告警规则
   - 从代码中提取自定义健康检查和告警逻辑
   - 按服务分组整理告警清单

4. **服务优先级判定**：按上述优先级定义表（P0-P3）对每个服务进行分级

### 步骤 2：盘点确认

Scanner 输出结构化盘点报告：
- 服务清单（名称、优先级、依赖关系）
- 告警清单（按服务分组，含告警名、阈值、严重程度）
- 现有手册状态（已有/缺失/过期）
- 建议的生成范围和优先级排序

确认方式：
- **标准模式**：向用户展示盘点报告，AskUserQuestion 确认生成范围
- **单轮确认模式**：scanner 自行判定，team lead 在后续阶段告知用户盘点结果
- **完全自主模式**：自动决策，不询问用户

盘点确认后释放 scanner（关闭 teammate）。

## 阶段一：双路撰写

### 步骤 3：启动双路独立撰写

同时启动 writer-1 和 writer-2，各自独立撰写，team lead 不让两位 writer 看到对方的撰写过程和内容。

按服务优先级从高到低依次撰写。每个服务的手册需遵循标准格式（见下方"手册输出格式"）。

**Writer-1（架构与依赖角度）**：
- 梳理服务的核心功能和业务场景
- 绘制上下游依赖关系（调用哪些服务、被谁调用）
- 定义关键 SLO（可用性、延迟、错误率）
- 编写服务降级方案（哪些功能可降级、降级步骤）
- 编写回滚策略（版本回滚、配置回滚的具体命令）
- 编写服务概览和架构说明
- 输出：以架构为核心的手册草稿

**Writer-2（告警与运维角度）**：
- 逐条编写告警的含义说明（用通俗语言解释告警触发原因）
- 编写诊断步骤（精确到可直接复制执行的命令）
- 编写修复步骤（精确到可直接复制执行的命令）
- 定义升级路径（不同时间节点该通知谁）
- 参考历史故障经验（如有 postmortem 记录）
- 输出：以告警运维为核心的手册草稿

### 步骤 4：撰写期间的决策升级

- Writer 需要扩大调查范围（如查阅其他服务的配置了解依赖关系）：
  - **标准模式**：通知 team lead，team lead 决策
  - **单轮确认模式/完全自主模式**：writer 自行扩展
- Writer 发现告警定义缺失或不合理：
  - 记录到"告警改进建议"中，不阻塞手册撰写
- Writer 发现服务缺少必要的监控覆盖：
  - 记录到"监控覆盖空白"中，继续撰写现有告警的手册

## 阶段二：交叉验证

### 步骤 5：合并对比分析

两份手册草稿完成后，启动 reviewer 进行交叉验证。

**对比维度（共 5 项，每项 20%）**：
```
① 服务概览完整性（架构描述是否清晰、依赖关系是否完整）
② 告警覆盖率（是否覆盖所有已知告警规则）
③ 诊断步骤可操作性（命令是否可直接执行、步骤是否有遗漏）
④ 修复步骤准确性（修复命令是否正确、回滚方案是否可行）
⑤ 升级路径完整性（升级条件是否明确、联系方式是否完整）

覆盖度 = 达标维度数 × 20%

任一草稿覆盖度 < 60%（≥3 项不达标）→ 需要补充（退回 writer 补充后重新审查）
```

**情况一：两份草稿互补性强**（常见情况）
- Writer-1 的架构视角 + Writer-2 的运维视角合并为完整手册

**情况二：两份草稿在同一告警上有不同诊断/修复方案**
- Reviewer 评估两个方案的可操作性和准确性，选择更优方案
- 如两方案各有优势，合并为"首选方案"和"备选方案"

**情况三：两份草稿均有重大遗漏**
- Reviewer 将遗漏项列表退回给对应 writer 补充
- 如仍无法补充：
  - **标准模式**：AskUserQuestion 向用户请求补充信息
  - **单轮确认模式**：reviewer 标注为"待补充"项，收尾时汇总
  - **完全自主模式**：自动标注，不询问用户

### 步骤 6：合并确认

Reviewer 输出合并后的最终手册：
- 每个服务一份完整手册（遵循标准格式）
- 合并说明（采纳了哪位 writer 的哪些内容）
- 待补充项列表（如有）

确认方式：
- **标准模式**：AskUserQuestion 向用户确认手册内容
- **单轮确认模式**：reviewer 自行确认，进入时效性检测
- **完全自主模式**：自动决策，不询问用户

**手册内容最终确认**无论标准模式还是单轮确认模式/完全自主模式，如果是 P0 服务的手册，**必须暂停问用户**（P0 服务手册直接影响 on-call 操作安全，不可自动决策）。

## 阶段三：时效性检测

### 步骤 7：对比代码变更与手册内容

Reviewer 执行时效性检测：

1. **代码变更扫描**：
   - 对比手册中涉及的文件/目录与最近的 git 提交历史
   - 检查告警规则是否有新增/修改/删除
   - 检查服务依赖关系是否有变更（新增依赖、移除依赖）
   - 检查 Kubernetes 配置变更（资源限制、副本数、环境变量）

2. **命令有效性检查**：
   - 验证手册中的诊断命令是否仍然有效（工具版本、路径变更等）
   - 检查引用的配置文件路径是否仍然存在
   - 检查引用的服务名/Pod 名/命名空间是否仍然正确

3. **时效性评分**：

| 状态 | 条件 | 处理 |
|------|------|------|
| **最新** | 手册更新日期晚于所有相关代码变更 | 无需操作 |
| **可能过期** | 相关代码有变更但不影响核心流程 | 标记提醒 |
| **需要更新** | 告警规则已变更 / 依赖关系已变更 / 关键命令已失效 | 高优先级更新 |
| **严重过期** | 服务已重构 / 手册引用的组件已不存在 | 必须立即重写 |

### 步骤 8：时效性报告

Reviewer 输出时效性报告：
- 每个手册的时效性状态
- 需要更新的具体内容（精确到章节）
- 建议的更新优先级

确认方式：
- **标准模式**：AskUserQuestion 向用户展示时效性报告
- **单轮确认模式/完全自主模式**：如发现 ≥3 份手册"严重过期"，**必须暂停问用户**（熔断机制）

## 阶段四：输出与可选提交

### 步骤 9：生成输出文件

按 `--lang` 指定的语言生成以下文件：

1. **手册文件**：`docs/runbooks/{service}.md`（每个服务一份）
2. **JSON 元数据**：`~/.gstack/data/{slug}/runbooks/{service}.json`（含最后更新时间、覆盖的告警列表、时效性状态）
3. **时效性报告**：`docs/runbooks/STALENESS_REPORT.md`（所有手册的时效性汇总）

如果是 `--update` 模式：
- 仅更新有变更的章节，保留用户手动修改的自定义内容
- 在手册末尾追加更新日志

### 步骤 10：收尾总结

Team lead 按 `--lang` 指定的语言向用户输出总结报告：

```
## 运维手册生成总结报告

### 元信息
- 生成时间：YYYY-MM-DD HH:mm:ss
- 团队名称：team-runbook-{YYYYMMDD-HHmmss}
- 执行模式：标准模式 / 单轮确认模式 / 完全自主模式
- 输出语言：zh / en
- 服务范围：全量 / 指定服务（{service}）
- 运行模式：新建 / 更新
```

**一、生成概要**
- 扫描到的服务数量和告警数量
- 生成/更新的手册数量
- 手册覆盖率（已有手册的服务数 / 总服务数）

**二、服务与手册清单**

| 服务名 | 优先级 | 告警数 | 手册状态 | 时效性 |
|--------|--------|--------|---------|--------|
| {service} | P0/P1/P2/P3 | {count} | 新建/更新/未变更 | 最新/可能过期/需要更新/严重过期 |

**三、时效性报告摘要**
- 最新手册数量
- 可能过期手册数量
- 需要更新手册数量
- 严重过期手册数量（含具体列表和过期原因）

**四、告警覆盖分析**
- 已覆盖告警数量和列表
- 未覆盖告警数量和列表（含建议的补充方向）
- 告警覆盖率

**五、质量检查结果**
- 所有手册中命令的有效性检查结果
- 缺失信息汇总（标注为"待补充"的项目）
- 改进建议

**六、后续建议**
- 需要优先更新的手册列表
- 建议新增的告警规则
- 建议补充的监控覆盖
- 建议的手册维护周期

**（单轮确认模式/完全自主模式）自动决策汇总**：

| 决策节点 | 决策内容 | 理由 |
|---------|---------|------|
| [阶段/步骤] | [决策描述] | [理由] |

**附录：撰写对比说明**
- writer-1（架构与依赖角度）草稿特点：[摘要]
- writer-2（告警与运维角度）草稿特点：[摘要]
- 覆盖度计算：[X/5 维度达标 = XX%]
  - ① 服务概览完整性：达标/不达标 — [说明]
  - ② 告警覆盖率：达标/不达标 — [说明]
  - ③ 诊断步骤可操作性：达标/不达标 — [说明]
  - ④ 修复步骤准确性：达标/不达标 — [说明]
  - ⑤ 升级路径完整性：达标/不达标 — [说明]
- 分歧处理记录：[如有合并取舍，记录 reviewer 裁决理由]

### 步骤 11：跨技能衔接建议（可选）

Team lead 根据生成情况向用户建议后续动作：
- **发现告警规则缺失**：建议运行 `/team-incident` 模拟故障演练验证手册有效性
- **发现架构文档缺失**：建议运行 `/team-arch` 补充架构文档
- **最近有故障发生**：建议运行 `/team-postmortem` 做事后复盘，复盘结果回补手册
- **手册大面积过期**：建议设置定期更新提醒（每月/每季度运行 `/team-runbook --update`）
- 用户可选择执行或跳过，不强制。

### 步骤 12：清理

关闭所有 teammate，用 TeamDelete 清理 team。

## 手册输出格式

每个服务的手册必须遵循以下标准格式：

```markdown
# 运维手册: {服务名称}
最后更新: YYYY-MM-DD | 负责人: {团队/个人}

## 服务概览
- 功能描述: {服务做什么}
- 上下游依赖: {上游服务/下游服务}
- 关键 SLO: {可用性/延迟/错误率等}

## 告警: {告警名称}
### 含义说明
{用通俗语言解释告警触发原因}

### 严重程度
{P1/P2/P3/P4}

### 诊断步骤
1. {包含精确命令的步骤}
2. {步骤}

### 修复方案
1. {包含精确命令的修复步骤}
```

## 核心原则

- **可操作优先**：手册中的每个步骤都必须包含可直接执行的命令，不接受模糊描述
- **双路撰写**：架构视角和运维视角独立撰写，确保手册既有全局观又有可执行性
- **覆盖完整**：每条告警都必须有对应的诊断和修复步骤，不留空白
- **时效性管理**：手册必须跟随代码变更保持更新，过期手册比没有手册更危险
- **升级明确**：每条告警都必须有明确的升级路径和时间节点
- **回滚兜底**：每个修复方案都必须附带回滚方案，确保操作可逆
- **格式统一**：所有手册遵循同一标准格式，降低 on-call 人员学习成本
- **增量更新**：`--update` 模式下保留用户自定义内容，仅更新有变更的部分
- **证据驱动**：手册内容基于代码和配置分析，不接受未验证的假设

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 项目中无法识别服务结构 | Scanner 列出已扫描的目录结构，team lead 用 AskUserQuestion 向用户确认服务划分 |
| observability.json 不存在 | Scanner 从代码中搜索告警配置（Prometheus rules、Alertmanager config 等），如仍无法找到则向用户请求告警信息 |
| 现有手册格式不统一 | Reviewer 在合并时统一格式，保留原有内容语义 |
| 告警规则无法关联到具体服务 | Scanner 按告警标签（namespace、service、job 等）推断所属服务，无法推断的标记为"待归类" |
| Writer 撰写的命令无法验证 | Reviewer 标注为"待验证"，收尾报告中高亮提示用户 |
| `--service` 指定的服务不存在 | Scanner 列出所有已识别服务，team lead 用 AskUserQuestion 向用户确认 |
| `--update` 模式但目标手册不存在 | 自动切换为新建模式，通知用户 |
| 服务依赖链过长导致扫描范围膨胀 | Scanner 仅扫描直接依赖（一跳），间接依赖记录但不展开 |
| Git 历史过长导致时效性扫描缓慢 | 限制时效性检测的 git log 范围为最近 90 天 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| 合并时两份草稿冲突严重 | Reviewer 以覆盖度更高的草稿为基础，从另一份中补充缺失项 |
