**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--from=<源技术>`：源技术栈（必需，如 `--from=webpack`、`--from=REST`、`--from=MySQL`）
- `--to=<目标技术>`：目标技术栈（必需，如 `--to=vite`、`--to=gRPC`、`--to=PostgreSQL`）
- `--dry-run`：仅生成迁移计划（MIGRATION.md），不执行任何代码变更
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从迁移目标描述中移除。如缺少 `--from` 或 `--to`，立即用 AskUserQuestion 要求用户补充。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 固定节点 + 条件节点，全部 AskUserQuestion | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅固定节点（影响确认、方案确认） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **planner-1/planner-2 方案分歧** → team lead 综合两方观点裁决，在方案确认时附带争议摘要和裁决理由
- **回滚验证失败** → validator 尝试调整回滚策略（最多 2 次），仍失败则标记为高风险，team lead 裁决
- **迭代超 3 轮** → **不可跳过，必须暂停问用户**（熔断机制）

完全自主模式下：所有节点（包括影响确认、方案确认等固定节点）均自动决策，不询问用户。收尾时汇总所有自动决策。唯一例外：迭代超 3 轮仍自动停止（但不询问用户，而是 team lead 自行裁决是否继续）。

使用 TeamCreate 创建 team（名称格式 `team-migration-{YYYYMMDD-HHmmss}`，如 `team-migration-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 迁移风险分类

| 类型 | 描述 | 风险等级 |
|------|------|---------|
| **config** | 配置/构建工具迁移（如 webpack→vite） | Low |
| **library** | 依赖库替换（如 moment→dayjs） | Medium |
| **framework** | 框架迁移（如 Express→Fastify） | Medium |
| **language** | 语言/运行时迁移（如 JS→TS、Python2→3） | High |
| **protocol** | 通信协议迁移（如 REST→gRPC） | High |
| **database** | 数据库/存储迁移（如 MySQL→PostgreSQL） | Critical |
| **architecture** | 架构迁移（如 单体→微服务、同步→异步） | Critical |

## 流程概览

```
阶段零  影响面扫描 → scanner 扫描全部源技术引用 → 绘制影响面地图 → 确定依赖排序
         ↓
阶段一  双路迁移规划 → planner-1 + planner-2 独立设计分阶段迁移方案 → team lead 合并评审 → 用户确认
         ↓
阶段二  方案合并与回滚验证 → validator 校验方案可行性 → 验证回滚流程 → 标注风险点
         ↓
阶段三  生成 MIGRATION.md → 输出分阶段迁移计划（含 checkbox 跟踪）
         ↓
阶段四  执行跟踪（可选） → 按阶段执行迁移 → 更新 MIGRATION.md 进度
         （--dry-run 模式在阶段三结束后停止）
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 扫描代码库中源技术的所有引用——文件、函数、测试、配置、文档。绘制影响面地图，确定依赖排序。输出影响面报告后关闭。**不参与方案设计。** |
| planner-1 | 独立制定分阶段迁移方案——阶段划分、执行顺序、回滚点设计、风险标注。**不与 planner-2 交流，不编写实现代码。** |
| planner-2 | 同 planner-1 职责，独立制定迁移方案。**不与 planner-1 交流，不编写实现代码。** |
| validator | 校验迁移方案可行性——检查回滚流程是否可执行、依赖排序是否合理、风险标注是否完整。对比两份方案的共识和分歧。**不编写或修改代码，只做验证。** |

## 阶段零：影响面扫描

### 步骤 1：解析迁移目标

Team lead 将用户的迁移描述转化为结构化目标：

1. **源技术**：`--from` 指定的当前技术栈
2. **目标技术**：`--to` 指定的目标技术栈
3. **迁移类型**：从上表中识别（config / library / framework / language / protocol / database / architecture）
4. **迁移目标描述**：具体的迁移目标（"将所有 REST API 迁移为 gRPC 服务"而非"换协议"）
5. **安全约束**：不能中断的服务、必须保持的兼容性、数据不可丢失等

### 步骤 2：风险预评估

Team lead 根据迁移类型做初步风险评估：

| 组合 | 风险等级 | 处理方式 |
|------|---------|---------|
| config | 低 | 正常流程 |
| library | 中 | 正常流程，关注 API 差异 |
| framework | 中高 | 正常流程，加强回滚验证 |
| language / protocol | 高 | 每阶段必须验证回滚 |
| database / architecture | 严重 | 每阶段用户确认，不可自动决策 |

确认方式：
- **标准模式**：向用户展示结构化目标和风险评估，AskUserQuestion 确认
- **单轮确认模式**：team lead 自行制定，在方案确认时一并展示（database/architecture 级除外，必须确认）
- **完全自主模式**：自动决策，不询问用户

### 步骤 3：启动 scanner

启动 scanner，传入源技术和目标技术信息。Scanner 执行全面影响面扫描：

1. **源技术引用扫描**——具体操作方法：
   - 代码引用：`Grep("<源技术关键字>|<源技术包名>|<源技术 API>")` 全局搜索所有代码中的源技术引用
   - Import/依赖引用：`Grep("import.*<源技术>|require.*<源技术>|from.*<源技术>|use.*<源技术>")` 提取所有导入关系
   - 包管理器依赖：分析 `package.json`/`requirements.txt`/`go.mod`/`Cargo.toml`/`pom.xml` 等中的源技术依赖
   - 配置文件引用：`Grep("<源技术>", glob="*.{yaml,yml,json,toml,ini,conf,env}")` 搜索配置中的引用
   - 文档引用：`Grep("<源技术>", glob="*.md")` 搜索文档中的引用
   - CI/CD 引用：`Grep("<源技术>", path="**/.github/**")` 或其他 CI 配置目录

2. **影响面地图绘制**：
   - 按文件分类：核心代码文件、测试文件、配置文件、文档文件、CI/CD 文件
   - 按依赖深度分类：直接引用（L0）、间接依赖（L1）、传递依赖（L2，最多 3 层）
   - 按模块/目录聚合：统计每个模块受影响的文件数和函数数

3. **依赖排序分析**：
   - 识别叶子节点（无下游依赖的文件/模块）和根节点（被多处引用的文件/模块）
   - 确定迁移顺序建议：先叶后根、先测试后代码、先配置后逻辑
   - 标注循环依赖（如有）

4. **测试覆盖分析**：
   - 搜索已有测试：`Grep("<源技术相关函数名>|<源技术相关类名>", path="**/test*/**")` 或 `Glob("**/*test*.*", "**/*spec*.*")`
   - 测试缺口标注：与源技术相关但没有测试覆盖的代码区域

### 步骤 4：输出影响面报告

Scanner 输出结构化影响面报告：
- 影响面摘要（涉及 X 个文件、Y 个函数、Z 个测试）
- 按模块/目录的影响面分布
- 高风险区域标注（测试覆盖薄弱的区域、核心业务逻辑密集的区域）
- 建议的迁移顺序（叶子节点→根节点、外围→核心）
- 循环依赖和特殊依赖标注

报告完成后 scanner 关闭。

确认方式：
- **标准模式**：向用户展示影响面报告摘要，AskUserQuestion 确认继续
- **单轮确认模式**：team lead 确认影响可控后继续（database/architecture 级除外，必须用户确认）
- **完全自主模式**：自动决策，不询问用户

## 阶段一：双路迁移规划

### 步骤 5：启动 planner-1 和 planner-2

同时启动两位 planner。传入迁移目标、影响面报告。两位 planner 各自独立工作，team lead 不让两位 planner 看到对方的方案，确保独立性。

每位 planner 输出：
1. **阶段划分**：将迁移拆分为有序的阶段（每阶段独立可编译可测试可回滚）
2. **每阶段内容**：具体要迁移的文件/模块/功能清单
3. **执行顺序**：阶段间的依赖关系和推荐执行顺序
4. **回滚点设计**：每阶段的回滚方式（git revert / 配置切换 / 数据回滚）
5. **兼容层策略**：迁移过程中新旧技术如何共存（适配器、Feature Flag、双写等）
6. **风险标注**：每阶段的风险等级和注意事项
7. **工作量估算**：每阶段的人力和 CC 工作量预估

### 步骤 6：合并评审

两份方案完成后，team lead 逐项对比合并评审：

| 对比结果 | 处理方式 |
|---------|---------|
| **一致方案**（阶段划分和顺序相同） | 直接采纳，标记为"共识" |
| **互补方案**（一方考虑了另一方未覆盖的场景） | 合并最优部分，标记为"互补" |
| **差异方案**（阶段粒度不同但方向一致） | 选择原子性更好的，标记为"共识" |
| **冲突方案**（对迁移路径有根本分歧） | 标注为"待仲裁" |

评估要点：
- 哪种方案的阶段划分更安全（每阶段更独立可回滚）
- 哪种方案的兼容层设计更稳健（新旧共存过渡更平滑）
- 哪种方案的依赖排序更合理

输出：
1. **共识清单**：双方一致的迁移阶段和执行顺序
2. **互补清单**：一方独有的有价值设计（额外的回滚策略、更好的兼容层等）
3. **分歧清单**：矛盾之处及双方方案对比
4. **共识度评估**：共识度 = (共识步骤数 + 互补步骤数) / 总步骤数(去重并集) x 100%
5. **最终迁移方案**：合并两方优点的完整方案

有分歧无法裁决时：
- **标准模式**：team lead 用 AskUserQuestion 交由用户决策
- **单轮确认模式**：team lead 综合两方观点裁决，在方案确认时附带争议摘要和裁决理由
- **完全自主模式**：自动决策，不询问用户

### 步骤 7：用户确认

Team lead 向用户输出：
- 两份独立方案的核心差异对比
- 合并后的最终迁移方案（阶段列表、每阶段内容、预估风险）
- 影响范围覆盖情况（对照影响面报告）
- 回滚策略说明
- 兼容层策略说明
- 预估的阶段总数、执行顺序和工作量

用 AskUserQuestion 等待用户确认后，方可进入下一阶段。

**database/architecture 级迁移**：无论标准/单轮确认模式，此步骤必须用户确认。完全自主模式下自动决策。

## 阶段二：方案合并与回滚验证

### 步骤 8：启动 validator

启动 validator，传入合并后的最终迁移方案和影响面报告。关闭 planner-1 和 planner-2（方案已确定，不再需要）。

Validator 执行以下验证：

**8a. 可行性验证**：
- 每个阶段的文件/模块清单是否完整覆盖影响面报告中的所有引用
- 阶段间的依赖排序是否正确（不存在先迁移被依赖方的情况）
- 每个阶段完成后系统是否处于可工作状态（新旧共存不冲突）
- 兼容层设计是否能正确桥接新旧技术

**8b. 回滚验证**：
- 每个阶段的回滚方式是否可执行（git revert 是否足够 or 需要额外操作）
- 回滚后是否能恢复到上一个稳定状态（特别关注数据迁移的可逆性）
- 多阶段连续回滚时是否有顺序依赖
- 回滚的时间窗口和影响范围评估

**8c. 风险标注**：
- 高风险阶段标注（涉及数据变更、核心业务逻辑、外部接口的阶段）
- 不可逆操作标注（数据库 schema 变更、数据格式转换等）
- 性能风险标注（兼容层是否引入性能开销）
- 遗漏覆盖标注（影响面报告中存在但方案中未覆盖的引用）

### 步骤 9：验证结果处理

| 验证结果 | 处理方式 |
|---------|---------|
| 全部通过 | 进入阶段三 |
| 覆盖遗漏 | Team lead 补充遗漏的文件/模块到对应阶段 |
| 依赖排序错误 | Team lead 调整阶段顺序 |
| 回滚不可行 | Team lead 重新设计该阶段的回滚策略；如无法设计可行回滚则标记为"不可逆阶段"，在 MIGRATION.md 中特殊标注 |
| 多处严重问题 | **标准模式**：AskUserQuestion 向用户确认处理方式；**单轮确认模式**：team lead 裁决；**完全自主模式**：自动决策 |

## 阶段三：生成 MIGRATION.md

### 步骤 10：生成迁移计划文档

Team lead 在项目根目录生成 `MIGRATION.md`，格式如下：

```markdown
# Migration: {from} → {to}
Generated: YYYY-MM-DD | Status: Planning

## Summary
- Source: {源技术}
- Target: {目标技术}
- Blast radius: X files, Y functions, Z tests
- Estimated phases: N
- Estimated effort: human X weeks / CC Y hours

## Phase 1: {阶段描述}
- [ ] Step 1.1: {具体步骤}
- [ ] Step 1.2: {具体步骤}
- Rollback: {回滚方式}
- Validation: {验证方式}

## Phase 2: {阶段描述}
- [ ] Step 2.1: {具体步骤}
- [ ] Step 2.2: {具体步骤}
- Rollback: {回滚方式}
- Validation: {验证方式}

...（更多阶段）

## Risks
- {风险 1：描述 + 缓解方式}
- {风险 2：描述 + 缓解方式}

## Compatibility Layer
- {兼容层策略描述}
- {兼容层清理时机}
```

### 步骤 11：生成跨技能数据文件

在 `~/.gstack/data/{slug}/` 目录下生成 `migration-{slug}.json`，包含结构化迁移数据供其他技能消费：

```json
{
  "from": "源技术",
  "to": "目标技术",
  "type": "迁移类型",
  "status": "planning | in_progress | complete",
  "blast_radius": { "files": 0, "functions": 0, "tests": 0 },
  "phases": [
    {
      "id": 1,
      "description": "阶段描述",
      "steps": ["步骤1", "步骤2"],
      "status": "pending | in_progress | complete",
      "rollback": "回滚方式",
      "risk": "low | medium | high | critical"
    }
  ],
  "generated": "YYYY-MM-DD",
  "dry_run": true
}
```

**如果是 `--dry-run` 模式**，在此步骤完成后跳转到收尾阶段（步骤 15），不执行阶段四。

## 阶段四：执行跟踪（非 dry-run 模式）

### 步骤 12：按阶段执行迁移

对每个阶段 N（从 Phase 1 开始），team lead 协调执行：

**12a. 阶段执行**：
- Team lead 按 MIGRATION.md 中该阶段的步骤逐一执行（使用 worktree 隔离）
- 每个步骤完成后勾选 MIGRATION.md 中对应的 checkbox
- 编译/构建通过后运行测试验证

**12b. 阶段验证**：
- Validator 验证该阶段完成后系统处于预期状态
- 新旧技术共存正常（兼容层工作正常）
- 测试全部通过
- 验证通过后更新 MIGRATION.md 状态

**12c. 阶段合并**：
- 合并该阶段的变更到主分支
- 更新 `migration-{slug}.json` 中对应阶段的 status 为 `complete`
- 更新 MIGRATION.md 顶部 Status 为 `In Progress`

**Worktree 分支命名规范**：`migration/<team-name>/phase-<N>`，例如 `migration/team-migration-20260311-100000/phase-1`。合并后分支自动清理。

### 步骤 13：阶段失败处理

| 失败类型 | 处理方式 |
|---------|---------|
| 编译失败 | Team lead 自行修复；无法修复则向用户汇报 |
| 测试失败（可定位） | 修复后重新验证（最多 2 次尝试） |
| 测试失败（2 次修复仍失败） | **熔断**：回滚该阶段，重新规划。标准模式问用户；单轮确认模式 team lead 裁决；完全自主模式自动决策 |
| Validator 验证不通过 | 修复后重新验证 |
| 合并冲突 | 解决冲突后重新提交，validator 重新验证冲突解决部分 |

**database/architecture 级迁移额外规则**：每完成一个阶段后，team lead 暂停，向用户汇报进展，AskUserQuestion 确认继续。

### 步骤 14：全量验证

所有阶段执行完成后：
- 运行完整测试套件（单测 + 集成测试 + E2E 测试，如有）
- 确认源技术的所有引用已迁移完毕（全局搜索确认无残留）
- 确认兼容层已清理（如方案要求在最后阶段清理）
- 更新 MIGRATION.md 顶部 Status 为 `Complete`
- 更新 `migration-{slug}.json` 中 status 为 `complete`

## 阶段五：收尾

### 步骤 15：最终总结

Team lead 按 `--lang` 指定的语言向用户输出：
- 迁移目标达成情况（逐项对照）
- 迁移类型和影响面
- 阶段执行摘要（共 N 阶段，每阶段概述）（dry-run 模式下输出"计划生成完毕，未执行"）

- 测试验证情况（通过率、覆盖率变化）（dry-run 模式跳过）

- 已知的遗留项或后续建议
- **方案共识说明**：共识度、planner-1 和 planner-2 的共识/互补/分歧汇总、仲裁结果

## 核心原则

- **影响先知**：先扫描全部源技术引用，再设计方案，再执行迁移
- **双路规划**：两位 planner 独立设计迁移方案，避免单一视角盲区
- **回滚为先**：每个阶段必须有可行的回滚方案，不可逆操作必须特殊标注
- **持续可用**：迁移过程中系统始终保持可工作状态，新旧技术可安全共存
- **逐阶段验证**：每阶段完成后立即验证，不积累风险
- **计划驱动**：MIGRATION.md 是唯一的进度跟踪源，所有进度更新反映在文档中
- **不确定就问**：标准模式下任何不确定事项必须交由用户确认；单轮确认模式下条件节点可自动决策；完全自主模式下全部自动决策
- **熔断保护**：阶段 2 次修复仍失败必须回滚重新规划；database/architecture 级每阶段必须用户确认
- **兼容层清理**：迁移完成后必须清理所有临时兼容层代码

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 编译/构建失败 | Team lead 自行修复；无法修复则向用户汇报 |
| 阶段测试失败（2 次修复仍失败） | 回滚该阶段，team lead 重新规划拆分方式 |
| Validator 发现回滚不可行 | Team lead 重新设计回滚策略；无法设计则标记"不可逆"并告知用户 |
| 源技术引用遗漏 | Scanner 补充扫描，更新影响面报告 |
| 合并冲突 | 解决冲突后重新提交，validator 审查冲突解决部分 |
| 兼容层引入 Bug | 修复兼容层后重新验证 |
| Scanner 发现影响面过大 | Team lead 与用户协商缩小迁移范围或分批次执行 |
| Planner 方案分歧过大 | Team lead 裁决；标准模式交由用户选择；单轮确认模式附带理由说明；完全自主模式自动决策 |
| Database/architecture 级迁移中途发现方案不可行 | 立即暂停，向用户汇报，协商调整方向或终止 |
| 全量验证失败超 3 轮 | 暂停，向用户汇报当前状态，由用户决定继续/调整/终止 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| 外部依赖方受影响 | Team lead 提醒用户通知下游依赖方，不自动推送变更 |
| 数据迁移涉及不可逆操作 | 方案必须包含数据备份步骤；validator 验证备份/恢复流程可行；MIGRATION.md 中特殊标注 |
| 无测试框架/测试基础设施 | 跳过自动化测试步骤，改为手动验证；validator 加强可行性审查；team lead 在总结中标注"无自动化测试保障"风险 |
| Git 仓库状态异常（脏工作区/未提交变更） | Team lead 在创建 worktree 前检测工作区状态，提示用户先 stash 或提交变更 |
| 项目过大无法完整扫描 | Scanner 根据迁移目标缩小扫描范围，聚焦直接引用（最多 3 层）；team lead 在总结中标注扫描范围限制 |
