## 关键问题清单

### Critical 问题
（无不可修复的 Critical 问题。Challenger 提出的 3 个 Critical 经评估降级为 Major/Minor。）

### Major 问题

1. **Agent 遵从性无硬约束** [§4.6] — 1000 字符校验可被绕过（含 /tmp/ 路径的超长消息通过检测）。建议：校验规则改为"消息总长度>1000 字符时，要求 agent 确认已写入文件"，而非仅检查路径存在。

2. **摘要误导风险** [§4.4] — agent 可能写好文件但产生不准确的摘要，导致 team lead 跳过关键发现。建议：在 SKILL.md 的关键决策节点（判断是否达标、是否继续迭代）强制要求 TL Read 合并报告文件，不仅依赖摘要。

3. **监控能力缺失** [§9] — 标记 N/A 不可接受。建议：Phase 2 试点期间创建 `bin/handoff-check` 诊断工具，检查文件完整性、命名合规性、遵从率。

4. **SKILL.md 瘦身应纳入范围** [§3 非目标#2] — Challenger 指出 SKILL.md 自身 15-30KB 是重要的上下文贡献者。建议：作为互补优化并行执行，或至少在 Phase 4 评估其影响。

### Minor 问题

1. /tmp 目录应 `chmod 700` 防止同机其他用户读取安全审计报告 [§4.2]
2. Summary Schema 应添加 `version` 字段 [§4.4]
3. 特殊流程 9 个 skill 应从 Phase 3 拆出单独排期 [§7]
4. 500 字符限制应明确为字符数（非字节数）[§4.4]
5. 文件命名应支持重试场景（添加 -v{N} 后缀）[§4.2]

## 改进建议

### 必须改进（Conditionally Approved 条件）

1. **关键决策点强制 Read**：在 SKILL.md 中明确标注哪些步骤 TL 必须 Read 文件（判断达标、继续迭代、合并报告），不可仅依赖摘要
2. **加强遵从性校验**：改为"消息>1000字符 → 要求确认已写文件"，不仅检查路径
3. **创建诊断工具**：`bin/handoff-check` 在试点阶段验证效果
4. **评估 SKILL.md 瘦身**：在 Phase 4 量化 SKILL.md 对上下文的贡献

### 建议改进

5. /tmp 目录 chmod 700
6. 添加 version 字段到 Summary Schema
7. 特殊流程 skill 组单独排期
8. 关键报告持久化（final-report.md 复制到项目目录而非仅 /tmp）

## 方案亮点

1. **扇出效应消除**：路径转发取代内容中转，架构上优雅地解决了最大的上下文膨胀源
2. **渐进式推广+向下兼容**：改造失败仅回退到当前模式，零破坏风险
3. **诚实的量化分析**：主动将 90% 修正为 50-60%，含 Read 回补详细表格，可信度高
4. **完善的回滚方案**：per-skill 回滚、git revert、两种模式共存
5. **Summary Schema 设计**：吸收方案 B 的结构化摘要，为未来工具化奠基
