**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--scope=full,api,auth,data,deps,infra`：审计范围（可选，默认 `full` 全面审计）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从审计需求描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 审计范围确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **漏洞定级不确定** → 对应 auditor 自行判断，报告中说明
- **两位 auditor 对同一问题定级差异过大（相差 2 级及以上）** → **不可跳过，必须暂停问用户**（熔断机制）
- **Critical 漏洞超过 5 个** → **不可跳过，必须暂停问用户**（熔断机制）
- **扫描工具运行失败** → scanner 标注"该工具不可用"，不阻塞流程
- **审计范围超出预期（发现大量第三方组件）** → analyst 自行决定是否纳入

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（定级差异过大、Critical 漏洞超过 5 个时仍必须暂停问用户）。

使用 TeamCreate 创建 team（名称格式 `team-security-{YYYYMMDD-HHmmss}`，如 `team-security-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  审计范围确定 → 解析 scope，识别技术栈和安全边界，确认审计目标
         ↓
阶段一  自动化扫描 + 双路手工审计（并行）
         ├─ scanner：SAST/SCA/Secret/Config/API 自动化扫描
         ├─ auditor-1：认证授权、访问控制、会话管理、API 安全
         └─ auditor-2：输入验证、数据保护、加密实现、注入防护
         ↓
阶段二  合并分析 → analyst 合并去重、分组定级、差异校验、输出统一漏洞清单
         → 熔断检查（Critical > 5 / 审计差异过大）
         ↓
阶段三  报告生成 → reporter 生成结构化安全审计报告（含修复建议、优先级、影响评估）
         ↓
阶段四  收尾 → 保存报告 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 识别项目技术栈和安全工具链，运行自动化安全扫描（SAST/DAST/SCA/Secret Scanning/Config Audit），输出客观漏洞数据。**仅在扫描阶段工作，完成后关闭。** |
| auditor-1 | **认证授权维度审计**：认证与授权、访问控制、会话管理、API 安全。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-2 交流。** |
| auditor-2 | **数据安全维度审计**：输入验证、数据保护与加密、注入防护、配置安全。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-1 交流。** |
| analyst | 合并两份审计报告和 scanner 扫描结果，去重、分组、交叉校验、按 CVSS 对齐定级（Critical/High/Medium/Low/Info），输出统一漏洞清单。**不直接阅读代码，只对比和合并报告。完成后关闭。** |
| reporter | 基于统一漏洞清单生成安全审计报告（含修复建议、优先级矩阵、影响评估、修复时间线），按 `--lang` 语言输出。**完成后关闭。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段一（步骤 3） | 阶段一扫描完成后（步骤 3） | 扫描报告交付后即释放 |
| auditor-1 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| auditor-2 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| analyst | 阶段二（步骤 6） | 阶段二合并完成后（步骤 9） | 统一漏洞清单和评分输出后释放 |
| reporter | 阶段三（步骤 10） | 阶段三报告生成后（步骤 12） | 报告确认后释放 |

## 漏洞严重程度定级（对齐 CVSS）

每个漏洞按严重程度定级：

| 级别 | 含义 | 是否必须修复 |
|------|------|-------------|
| **Critical** | RCE、SQL 注入、认证绕过、数据泄露、未授权访问核心数据 | 必须立即修复 |
| **High** | XSS（存储型）、SSRF、权限提升、敏感信息泄露、不安全反序列化 | 必须修复 |
| **Medium** | CSRF、不安全配置、弱加密算法、会话固定、路径遍历 | 建议修复 |
| **Low** | 缺少安全头、冗余错误信息、Cookie 未设 Secure/HttpOnly | 可选修复 |
| **Info** | 最佳实践建议、安全加固方向、合规性提示 | 可选 |

## Scanner 自动化扫描工具

| 扫描类别 | 常用工具 | 输出 |
|---------|---------|------|
| SAST（静态应用安全测试） | Semgrep / CodeQL / Bandit / gosec / eslint-plugin-security | 代码级漏洞发现 |
| SCA（软件成分分析） | npm audit / pip-audit / cargo audit / Snyk / trivy | 依赖组件 CVE 列表 |
| Secret Scanning（敏感信息扫描） | gitleaks / trufflehog | 暴露的凭证、密钥、Token |
| Config Audit（配置审计） | checkov / tfsec / kube-bench | IaC 配置缺陷 |
| API Security（API 安全检查） | 基于 OWASP API Top 10 的自定义检查 | API 级漏洞 |

**如果某工具未安装或无法运行**：scanner 标注"该工具不可用"，不阻塞流程。

## 审计维度分配

双 auditor 按各自负责的维度独立审计，analyst 按权重合并为安全评估总分。

| 维度 | 权重 | 负责人 | 审计内容 |
|------|------|--------|---------|
| 认证与授权 | 20% | auditor-1 | 认证机制强度、JWT/Session 安全、OAuth 实现、密码策略、MFA |
| 访问控制 | 15% | auditor-1 | RBAC/ABAC 实现、越权检查、资源隔离、最小权限原则 |
| 会话管理 | 10% | auditor-1 | 会话生命周期、Session Fixation、Token 刷新、并发会话 |
| API 安全 | 10% | auditor-1 | OWASP API Top 10、速率限制、输入输出校验、API 认证 |
| 输入验证 | 15% | auditor-2 | 参数校验、白名单/黑名单、编码处理、文件上传安全 |
| 数据保护与加密 | 15% | auditor-2 | 传输加密（TLS）、存储加密、密钥管理、敏感数据脱敏 |
| 注入防护 | 10% | auditor-2 | SQL/NoSQL/LDAP/OS 注入、参数化查询、ORM 使用安全 |
| 配置安全 | 5% | auditor-2 | 安全头、CORS、CSP、错误处理配置、调试模式、默认凭证 |

**auditor-1 总权重：55%**（认证与授权 20% + 访问控制 15% + 会话管理 10% + API 安全 10%）
**auditor-2 总权重：45%**（输入验证 15% + 数据保护与加密 15% + 注入防护 10% + 配置安全 5%）

**`--scope` 参数处理**：如果用户指定了 `--scope`，仅审计对应范围的维度。例如 `--scope=api,auth` 仅审计 API 安全和认证与授权。未指定则默认 `full` 全面审计。

## 阶段零：审计范围确定

### 步骤 1：解析审计范围

Team lead 分析项目和参数：

1. 解析 `--scope` 参数，确定审计范围
2. 阅读项目结构，识别技术栈（语言、框架、数据库、中间件）
3. 识别安全边界：
   - 对外暴露的入口（HTTP API、gRPC、WebSocket、消息队列）
   - 认证/授权机制（JWT、OAuth、Session、API Key）
   - 数据存储（数据库、缓存、文件系统、对象存储）
   - 第三方集成（支付、短信、邮件、云服务 SDK）
   - 基础设施（Docker、K8s、Terraform、CI/CD）
4. 输出审计范围清单

### 步骤 2：用户确认审计范围

**标准模式**：向用户展示审计范围清单，AskUserQuestion 确认：
- 确认审计范围
- 调整审计范围（增删维度）
- 补充安全关注点

**单轮确认模式**：跳过确认，直接进入阶段一。
**完全自主模式**：自动决策，不询问用户，直接进入阶段一。

## 阶段一：自动化扫描 + 双路手工审计

### 步骤 3：启动 scanner 和双 auditor

三者并行启动。

**Scanner 自动化扫描**：
1. 阅读项目结构，识别技术栈和包管理器
2. 识别项目中已配置的安全工具（.semgrep.yml、.gitleaks.toml 等）
3. 按审计范围运行对应扫描工具，收集客观数据：

| 扫描范围 | 执行的工具类别 | 输出内容 |
|---------|-------------|---------|
| full | 全部工具 | 全量扫描结果 |
| api | SAST + API Security | API 相关漏洞 |
| auth | SAST（认证相关规则） | 认证实现缺陷 |
| data | SAST + Secret Scanning | 数据保护与泄露 |
| deps | SCA | 依赖组件 CVE |
| infra | Config Audit | 基础设施配置缺陷 |

4. 汇总输出**自动化扫描报告**发送给 team lead
5. Scanner 完成后关闭（不参与后续阶段）

**双 auditor 同时阅读项目**：
- 阅读项目结构（目录布局、入口文件、配置文件）
- 理解技术栈、框架和安全机制
- 梳理数据流（从用户输入到数据存储的完整链路）
- 各自输出项目安全概况给 team lead

### 步骤 4：分发 scanner 报告

Team lead 收到 scanner 报告后，将自动化扫描结果分发给两位 auditor，作为手工审计的参考数据。

Auditor 在审计时必须参考 scanner 结果——例如 scanner 发现依赖中存在已知 CVE，auditor-2 在数据保护维度应当纳入评估。

### 步骤 5：独立并行手工审计

两位 auditor 各自审计负责的维度，**互不交流**。

**Auditor-1（认证授权维度）** — 对应 OWASP Top 10 和 OWASP API Top 10 类别：

审计时必须逐项检查以下 OWASP 类别，每个类别输出"已检查 + 发现/无发现"：

| OWASP 类别 | 审计内容 | 具体检查项 |
|-----------|---------|-----------|
| **A01:2021 Broken Access Control** | 越权漏洞 | IDOR（通过修改 ID 访问他人资源）、水平越权、垂直越权、RBAC/ABAC 实现完整性、资源隔离、`@RequireRole`/中间件覆盖度 |
| **A07:2021 Identification and Authentication Failures** | 认证缺陷 | 密码存储算法（必须 bcrypt/scrypt/argon2，禁止 MD5/SHA1）、Token 生成随机性、JWT 签名验证（是否验证 alg）、OAuth 流程（state 参数/redirect_uri 校验）、MFA 实现、暴力破解防护（登录限速） |
| **OWASP API1:2023 Broken Object Level Authorization** | API 对象级授权 | 每个 API 端点是否检查当前用户对目标对象的访问权限 |
| **OWASP API2:2023 Broken Authentication** | API 认证 | API Key/Token 传输安全、认证端点速率限制、Token 过期策略 |
| **OWASP API4:2023 Unrestricted Resource Consumption** | API 速率限制 | 全局/per-user 限速配置、批量操作限制、文件上传大小限制 |
| **OWASP API5:2023 Broken Function Level Authorization** | API 功能级授权 | 管理接口是否有独立权限检查、普通用户能否调用管理 API |

- 审计会话管理：Session 生命周期（超时设置）、Session Fixation 防护、Token 刷新机制（refresh token rotation）、并发会话控制、登出处理（Token 吊销/黑名单）
- 审计 API 安全：OWASP API Top 10 全部 10 项逐一检查、GraphQL introspection 是否禁用（如适用）、WebSocket 认证
- 参考 scanner 的 SAST 结果，验证自动化发现的准确性
- 记录每个发现：文件路径 + 行号 + 严重程度 + OWASP 类别编号 + 漏洞描述 + PoC/利用路径 + 修复建议

**Auditor-2（数据安全维度）** — 对应 OWASP Top 10 类别：

审计时必须逐项检查以下 OWASP 类别，每个类别输出"已检查 + 发现/无发现"：

| OWASP 类别 | 审计内容 | 具体检查项 |
|-----------|---------|-----------|
| **A03:2021 Injection** | 注入防护 | SQL 注入（检查原始 SQL 拼接 vs 参数化查询）、NoSQL 注入（MongoDB `$where`/`$regex`）、LDAP 注入、OS Command 注入（`exec`/`system` 调用）、ORM 安全（raw query 使用） |
| **A02:2021 Cryptographic Failures** | 加密缺陷 | TLS 版本（≥1.2）、存储加密算法（AES-256 等）、密钥管理（硬编码密钥检测、环境变量 vs KMS）、敏感数据明文存储（密码、Token、PII） |
| **A04:2021 Insecure Design** | 不安全设计 | 业务逻辑漏洞、缺少速率限制的敏感操作、批量操作未做限制 |
| **A05:2021 Security Misconfiguration** | 安全配置 | 安全头（CSP/HSTS/X-Frame-Options/X-Content-Type-Options）、CORS 策略（`Access-Control-Allow-Origin: *`）、错误信息泄露（stack trace）、调试模式（`DEBUG=true`）、默认凭证 |
| **A06:2021 Vulnerable and Outdated Components** | 组件漏洞 | 依赖组件已知 CVE（参考 scanner SCA 结果）、过期依赖、不安全的依赖源 |
| **A08:2021 Software and Data Integrity Failures** | 完整性 | 反序列化安全（不安全的 `pickle`/`yaml.load`/`JSON.parse` + eval）、CI/CD 管道安全、子资源完整性 |
| **A09:2021 Security Logging and Monitoring Failures** | 日志监控 | 敏感数据是否出现在日志中、安全事件是否有审计日志、日志注入防护 |

- 审计输入验证：参数校验覆盖度（每个用户输入是否经过校验）、白名单 vs 黑名单、编码处理（XSS 防护：HTML/JS/URL 编码）、文件上传类型/大小限制、路径遍历防护
- 审计数据保护：敏感数据脱敏（日志中的密码/Token/PII、API 响应中的敏感字段）
- 参考 scanner 的 SCA 和 Secret Scanning 结果，验证并补充
- 记录每个发现：文件路径 + 行号 + 严重程度 + OWASP 类别编号 + 漏洞描述 + PoC/利用路径 + 修复建议

## 阶段二：合并分析

### 步骤 6：启动 analyst

两位 auditor 审计完成后，启动 analyst。Team lead 将以下材料发送给 analyst：
- Scanner 自动化扫描报告
- Auditor-1 审计报告
- Auditor-2 审计报告
- 项目安全边界信息

### 步骤 7：合并去重与分组

Analyst 执行合并分析：

1. **去重**：识别 scanner 结果与 auditor 手工发现的重叠项，合并为单条记录
2. **交叉校验**：检查两位 auditor 是否对同一问题有不同定级
3. **分组**：按攻击向量/漏洞类型分组（如"注入类"、"认证绕过类"、"配置缺陷类"）
4. **定级**：按 CVSS 对齐标准统一定级，输出每个漏洞的严重程度
5. **影响评估**：评估每个漏洞的实际业务影响（数据泄露范围、影响用户数、合规风险）

**共识度计算公式**：
```
两位 auditor 审计有部分维度交叉（如 auditor-1 的 API 安全中的输入校验 与 auditor-2 的输入验证可能重叠）。
对于重叠区域的发现：

共识发现数 = 两位 auditor 都识别出的漏洞数量（指向相同文件/行号且描述相同漏洞类型）
总重叠区域发现数 = 重叠区域去重后的漏洞总数

共识度 = 共识发现数 / 总重叠区域发现数 × 100%（如无重叠区域发现则为 100%）

定级一致性 = 两位 auditor 对同一漏洞定级完全一致的比例
争议率 = 定级差异 ≥ 2 级的漏洞数 / 两位 auditor 都识别的漏洞数 × 100%

争议率 > 50% → 触发熔断，必须暂停问用户
```

### 步骤 8：差异校验与熔断检查

**差异校验**：
- 如果两位 auditor 对同一问题的定级差异 ≥ 2 级（如一方定 Critical、另一方定 Medium），analyst 标注为"争议项"
- 争议项占比 > 50% → **不可跳过，必须暂停问用户**（熔断机制）

**Critical 漏洞熔断**：
- 统一漏洞清单中 Critical 漏洞超过 5 个 → **不可跳过，必须暂停问用户**（熔断机制）
- 向用户展示 Critical 漏洞列表，确认是否继续生成完整报告

**无熔断触发**时，analyst 输出**统一漏洞清单**：

```
## 统一漏洞清单

### 漏洞统计
- Critical: X 个 | High: X 个 | Medium: X 个 | Low: X 个 | Info: X 个
- 总计: X 个漏洞

### 漏洞分组

#### 组 1：[攻击向量/漏洞类型] — 最高级别：Critical
```

### 步骤 9：安全评估评分

Team lead 根据统一漏洞清单计算各维度安全评分：

评分规则：
- 基础分 10.0，按漏洞扣分
- Critical 漏洞：每个扣 2.0 分（该维度最多扣至 0 分）
- High 漏洞：每个扣 1.0 分
- Medium 漏洞：每个扣 0.5 分
- Low 漏洞：每个扣 0.2 分
- Info：不扣分

按权重计算总分，输出各维度和总体安全评分。

## 核心原则

- **工具先行**：先用自动化工具收集客观漏洞数据，再人工审计高价值安全问题
- **双路独立**：两位 auditor 从不同维度独立审计，互不交流，避免盲区和偏见
- **CVSS 对齐**：漏洞定级对齐 CVSS 标准，确保严重程度评估一致性
- **攻击者视角**：审计时从攻击者角度思考利用路径，不只看代码实现
- **纵深防御**：关注安全机制是否分层部署，单点失效不应导致完全沦陷
- **最小权限**：检查系统各层级是否遵循最小权限原则
- **合规意识**：审计时关注合规性要求（等保、GDPR、PCI DSS 等）
- **可操作性**：修复建议必须具体到代码级别，可直接执行，避免空泛建议

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 扫描工具未安装/无法运行 | Scanner 标注"该工具不可用"，不阻塞流程，auditor 基于手工审计评估 |
| 项目无法构建/运行 | 仅做静态审计（SAST + 手工代码审查），报告中标注"未验证运行时行为" |
| 代码仓库过大（>100 万行） | Scanner 按模块分批扫描；auditor 聚焦安全边界和高风险模块，标注未覆盖区域 |
| 两位 auditor 定级差异过大（争议项 > 50%） | 触发熔断，暂停问用户裁决争议项 |
| Critical 漏洞超过 5 个 | 触发熔断，暂停向用户确认是否继续完整报告 |
| 发现疑似 0day 或正在被利用的漏洞 | 立即通知用户（无论任何模式），建议紧急处置 |
| 加密代码/混淆代码 | Auditor 标注"无法审计"，建议用户提供源码或说明 |
| 项目使用不常见语言/框架 | Scanner 跳过不支持的工具，auditor 基于安全原则进行通用审计 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前步骤恢复 |
| 审计范围与实际项目不匹配 | Team lead 在阶段零发现后调整范围，标准模式需用户确认 |
