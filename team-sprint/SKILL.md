**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--period=1w|2w|1m`：Sprint 周期（默认 `2w`）
- `--team-size=N`：团队人数（影响容量计算，默认从 git log 中推断活跃贡献者数量）

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 速率基线确认 + 过载风险仲裁 + 最终计划确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终计划确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **Sprint 周期不确定** → team lead 根据历史速率自行判断，在最终计划中说明
- **planner 与 reviewer 对容量评估分歧** → team lead 综合论证后裁决，收尾时汇总
- **过载率超过 120%** → **不可跳过，必须暂停问用户**（熔断机制）
- **高不确定性条目超过 Sprint 容量 30%** → **不可跳过，必须暂停问用户**（熔断机制）
- **历史数据不足（<2 个 Sprint 周期）** → analyst 标注数据置信度低，planner 采用保守估算

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（过载率超过 120%、高不确定性条目超 30% 时仍必须暂停问用户）。

Sprint 周期说明：

| 周期 | 含义 | 速率采样窗口 |
|------|------|-------------|
| `1w` | 1 周 Sprint | 过去 4 周的周粒度数据 |
| `2w` | 2 周 Sprint（默认） | 过去 8 周的双周粒度数据 |
| `1m` | 1 月 Sprint | 过去 4 个月的月粒度数据 |

使用 TeamCreate 创建 team（名称格式 `team-sprint-{YYYYMMDD-HHmmss}`，如 `team-sprint-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  速率基线 → analyst 读取 git log（最近 4 个 Sprint 周期） → 计算速率指标
         ↓
阶段一  跨技能数据采集 → 读取 techdebt.json、capacity.json（如有） → 检查新鲜度
         ↓
阶段二  Sprint 规划 → planner 基于速率 + 待办条目推荐 Sprint 范围
         ↓
阶段三  风险评审 → reviewer 验证计划 → 标记过载 → 评估不确定性 → 检查熔断
         ↓
阶段四  输出 Sprint 计划 + 速率报告 → 保存 velocity.json → 用户确认
```

## 角色定义

| 角色 | 职责 |
|------|------|
| analyst | 从 git 历史中计算速率指标（吞吐量、周期时间、前置时间、变更返工率、聚焦分数）。**只做数据采集和计算，不做规划决策。** |
| planner | 基于速率数据 + TODOS.md/待办列表 + 跨技能数据，推荐 Sprint 范围和优先级排序。**只做规划推荐，不做速率计算。** |
| reviewer | 验证 planner 的计划是否符合历史速率，检测过载风险、高不确定性条目、依赖阻塞。**只做验证和风险评估，不修改计划。** |

## 阶段零：速率基线

### 步骤 1：启动 analyst

Team lead 启动 analyst，指示其执行 git 历史分析：
- 读取 git log（最近 4 个 Sprint 周期的数据）
- 识别活跃贡献者（如未指定 `--team-size`，以此推断团队规模）
- 统计 PR/合并提交数据
- 分析提交频率和模式

### 步骤 2：计算速率指标

Analyst 计算以下核心指标：

| 指标 | 计算方法 | 说明 |
|------|---------|------|
| **吞吐量（Throughput）** | 每个 Sprint 合并的 PR 数 | 按 Sprint 周期分桶统计 |
| **周期时间（Cycle Time）** | 从首次提交到合并的时间 | 取中位数和 P85 |
| **前置时间（Lead Time）** | 从 issue 创建到合并的时间 | 如无 issue 追踪，退化为周期时间 |
| **变更返工率（Churn Rate）** | Sprint 内被修改后又再次修改的代码行比例 | 高返工率表示需求不稳定或设计返工 |
| **聚焦分数（Focus Score）** | 每位开发者每周触及的不同模块数均值 | 低聚焦分数表示上下文切换过多 |

附加统计：
- 每个 Sprint 的提交数分布
- 活跃天数（有提交的工作日比例）
- 速率趋势方向（上升/稳定/下降）

**如果 git 历史数据不足（<2 个 Sprint 周期）**：analyst 标注"数据置信度低"，输出可用数据的最佳估算。

### 步骤 3：输出速率基线报告

Analyst 输出**速率基线报告**，包含：
- 团队规模（指定或推断）
- 各指标的逐 Sprint 数值和趋势
- 速率稳定性评估（标准差/变异系数）
- 数据置信度评级（高/中/低）

### 步骤 4：确认速率基线

**标准模式**：向用户展示速率基线报告，AskUserQuestion 确认数据是否符合预期
**单轮确认模式**：team lead 自行判断，收尾汇总时说明
**完全自主模式**：自动决策，不询问用户

## 阶段一：跨技能数据采集

### 步骤 5：读取跨技能数据

Team lead 指示 analyst 检查并读取以下跨技能数据文件：

| 数据源 | 路径 | 用途 |
|--------|------|------|
| 技术债务数据 | `~/.gstack/data/{slug}/techdebt.json` | 将高优先级债务条目纳入 Sprint 候选 |
| 容量风险数据 | `~/.gstack/data/{slug}/capacity.json` | 根据 bus factor 风险调整容量 |

### 步骤 6：新鲜度检查

对每个数据文件执行新鲜度检查：
- 文件修改时间 > 7 天前 → 标注"数据可能过时"，在最终计划中警告
- 文件不存在 → 标注"不可用"，不阻塞流程
- 文件存在且 ≤ 7 天 → 正常使用

### 步骤 7：整合跨技能数据

Analyst 整合跨技能数据到速率报告：
- 从 `techdebt.json` 提取高优先级技术债务条目，作为 Sprint 候选项
- 从 `capacity.json` 提取 bus factor 数据，评估容量风险：
  - bus factor = 1 的模块 → 标记为"关键人员风险"，建议分配知识传递任务
  - bus factor ≤ 2 的核心模块 → 标记为"需关注"

## 阶段二：Sprint 规划

### 步骤 8：启动 planner

Team lead 启动 planner，传递以下内容：
- Analyst 的速率基线报告
- 跨技能数据整合结果
- Sprint 周期（`--period` 参数值）
- 团队人数（`--team-size` 参数值或推断值）

### 步骤 9：计算 Sprint 容量

Planner 计算当前 Sprint 容量：

```
基础容量 = 历史平均吞吐量（PR/Sprint）
调整系数 = 1.0
  - 如速率趋势下降 → 调整系数 × 0.85
  - 如有 bus factor = 1 模块需知识传递 → 调整系数 × 0.9
  - 如上个 Sprint 变更返工率 > 20% → 调整系数 × 0.9
最终容量 = 基础容量 × 调整系数（向下取整）
```

### 步骤 10：收集和评估待办条目

Planner 收集待办条目来源：
- 项目中的 `TODOS.md`（如存在）
- GitHub/GitLab Issues（通过 CLI 工具获取，如可用）
- `techdebt.json` 中的高优先级债务条目

对每个待办条目评估：

| 维度 | 评估方法 |
|------|---------|
| **优先级** | 从 issue 标签/TODOS.md 标记推断，分为 P0/P1/P2/P3 |
| **规模估算** | 基于历史类似 PR 的周期时间估算，分为 S/M/L/XL |
| **不确定性** | 需求是否明确、是否有技术探索、是否依赖外部 → 高/中/低 |
| **依赖关系** | 是否阻塞/被阻塞于其他条目 |

### 步骤 11：推荐 Sprint 范围

Planner 按以下规则推荐 Sprint 范围：

1. 按优先级排序（P0 > P1 > P2 > P3）
2. 同优先级内，优先选择低不确定性条目
3. 累积规模不超过最终容量
4. 标记依赖关系链
5. 建议预留 15-20% 缓冲容量应对意外

Planner 输出**Sprint 计划草案**：
- 推荐条目列表（含优先级、规模、不确定性标记）
- 容量使用率 = 推荐条目总规模 / 最终容量 × 100%
- 未纳入条目列表及原因
- 技术债务分配建议（建议 Sprint 容量的 10-20% 用于偿还技术债务）

---

## 阶段三：风险评审

### 步骤 12：启动 reviewer

Team lead 启动 reviewer，传递以下内容：
- Analyst 的速率基线报告
- Planner 的 Sprint 计划草案
- 跨技能数据整合结果

### 步骤 13：过载检测

Reviewer 检查以下过载指标：

| 检查项 | 阈值 | 处理 |
|--------|------|------|
| 容量使用率 > 100% | 过载 | 标记为"过载"，建议削减 |
| 容量使用率 > 120% | 严重过载 | **触发熔断**，必须暂停 |
| 高不确定性条目占比 > 30% | 高风险 | **触发熔断**，必须暂停 |
| 单人承担 > 40% 的计划条目 | 集中风险 | 标记为"负载不均"，建议重新分配 |
| 依赖链长度 > 3 | 依赖风险 | 标记为"长依赖链"，建议拆分或并行化 |

### 步骤 14：检查熔断条件

如果触发熔断：
- **必须暂停**，team lead 向用户报告过载情况或高不确定性风险
- 建议：削减低优先级条目 / 拆分大条目 / 将不确定条目移至下个 Sprint
- AskUserQuestion 让用户决定调整方案

**标准模式**：所有风险点均报告用户确认
**单轮确认模式**：仅熔断时暂停，其他风险自动处理并在收尾汇总
**完全自主模式**：仅熔断时暂停，其他风险自动处理

### 步骤 15：Reviewer 输出风险评审报告

Reviewer 输出**风险评审报告**：
- 速率一致性验证（计划是否符合历史速率范围）
- 过载检测结果
- 风险条目清单（含风险类型和建议）
- 整体风险等级（低/中/高/极高）
- 调整建议（如有）

### 步骤 16：Planner 更新计划

如果 reviewer 提出调整建议且被采纳：
- Planner 根据调整建议更新 Sprint 计划
- Reviewer 再次验证更新后的计划（轻量复查，仅验证调整项）

## 阶段五（隐含）：收尾

### 步骤 21：最终总结

Team lead 向用户输出：
- 分析了什么（项目名称、Sprint 周期、团队规模）
- 速率核心指标（吞吐量、周期时间、趋势方向）
- Sprint 计划要点（计划条目数、容量使用率、风险等级）
- 跨技能数据使用情况（techdebt.json / capacity.json 是否可用及新鲜度）
- velocity.json 保存位置
- **（单轮确认模式/完全自主模式）自动决策汇总**：列出所有自动决策的节点、决策内容和理由

关闭所有 teammate，用 TeamDelete 清理 team。

## 核心原则

- **数据驱动**：所有规划决策以 git 历史速率数据为锚点，避免主观臆测
- **职责分离**：analyst 只做数据计算，planner 只做规划推荐，reviewer 只做验证评审
- **保守估算**：数据不足时采用保守策略，宁可少排不可过载
- **过载保护**：多层过载检测机制，熔断条件下必须暂停确认
- **跨技能协同**：消费 techdebt.json 和 capacity.json，产出 velocity.json，形成数据闭环
- **可追溯**：速率指标逐 Sprint 记录，支持长期趋势分析
- **缓冲预留**：始终建议预留 15-20% 容量应对意外

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| Git 历史数据不足（<2 个 Sprint 周期） | Analyst 标注"数据置信度低"，planner 采用保守估算（基础容量 × 0.7） |
| 无法获取 PR/合并数据 | 退化为提交数统计，标注"粗粒度指标" |
| TODOS.md / Issue 列表为空 | 提示用户提供待办条目，或从近期未关闭 issue 中推断 |
| techdebt.json / capacity.json 不存在 | 标注"不可用"，不阻塞流程，仅基于 git 数据规划 |
| 跨技能数据过时（>7 天） | 在计划中警告"跨技能数据可能过时"，降低其权重 |
| 过载率超过 120% | 触发熔断，暂停问用户确认调整方案 |
| 高不确定性条目超过 30% | 触发熔断，建议将不确定条目移至 spike 或下个 Sprint |
| Planner 与 reviewer 对容量评估分歧 | Team lead 综合双方论证裁决，收尾汇总 |
| 团队规模推断不准确 | 在速率报告中说明推断逻辑，建议用户通过 `--team-size` 显式指定 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前阶段恢复 |
