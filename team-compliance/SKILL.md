**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--framework=soc2|gdpr|hipaa|pci`：合规框架（可选，默认 `soc2`）
- `--evidence`：生成证据文件（可选，启用后在 `docs/compliance/evidence/` 目录生成证据制品）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从审计需求描述中移除。

**V1 范围说明**：当前版本仅支持 SOC2 代码级控制子集。如用户指定 `--framework=gdpr|hipaa|pci`，提示"V1 仅支持 SOC2 代码级控制子集，其他框架将在后续版本支持"，并回退到 SOC2 审计。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 审计范围确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **控制评级不确定** → 对应 auditor 自行判断，报告中说明
- **两位 auditor 对同一控制项评级差异过大（相差 2 级及以上）** → **不可跳过，必须暂停问用户**（熔断机制）
- **"不符合"控制项超过 5 个** → **不可跳过，必须暂停问用户**（熔断机制）
- **扫描工具运行失败** → scanner 标注"该工具不可用"，不阻塞流程
- **审计范围超出预期（发现大量第三方组件）** → analyst 自行决定是否纳入

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（评级差异过大、"不符合"控制项超过 5 个时仍必须暂停问用户）。

使用 TeamCreate 创建 team（名称格式 `team-compliance-{YYYYMMDD-HHmmss}`，如 `team-compliance-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

**文件命名规范**：

| 角色输出 | 文件名 |
|---------|--------|
| Scanner 报告 | `scanner-report.md` |
| Reviewer-N 第R轮 | `reviewer-{N}-round-{R}.md` |
| 合并报告第R轮 | `merged-report-round-{R}.md` |
| 根因分组 | `root-cause-groups-round-{R}.md` |
| Fixer 第R轮 | `fixer-round-{R}.md` |
| Tester 第R轮 | `tester-round-{R}.md` |
| Architect-N 方案 | `architect-{N}-design.md` |
| 任务拆解 | `task-breakdown.md` |
| Coder-N 任务T | `coder-{N}-task-{T}.md` |
| 审查任务T | `review-task-{T}.md` |
| 集成测试第R轮 | `integration-test-round-{R}.md` |
| 最终报告 | `final-report.md` |

> 仅当角色存在于当前 skill 时使用对应命名。未列出的角色用 `{role}-{context}.md` 格式。

## 流程概览

```
阶段零  框架选择 + 项目扫描 → 选定合规框架，识别技术栈和相关代码模式，确认审计目标
         ↓
阶段一  自动化控制检查 + 双路独立代码级审计（并行）
         ├─ scanner：自动化控制检查（认证模式、日志记录、输入验证、密钥管理、CI/CD 配置）
         ├─ auditor-1：访问控制实现审计（认证、RBAC、会话管理、API 认证）
         └─ auditor-2：监控与变更管理审计（日志记录、审计追踪、Git 工作流、CI/CD）
         ↓
阶段二  差距分析 → analyst 合并发现，识别缺失控制项，输出统一合规评估
         → 熔断检查（"不符合"控制项 > 5 / 审计差异过大）
         ↓
阶段三  报告生成 → reporter 生成差距分析报告 + 证据制品 + 整改计划
         ↓
阶段四  可选证据导出 → 保存报告 + 证据文件 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 识别项目技术栈和合规相关配置，运行自动化控制检查（认证模式扫描、日志覆盖率检测、密钥泄露扫描、CI/CD 配置审计、依赖漏洞扫描），输出客观控制数据。**仅在扫描阶段工作，完成后关闭。** |
| auditor-1 | **访问控制维度审计**：逻辑访问控制（CC6.1）、边界保护（CC6.6）、漏洞管理（CC6.8）。审查认证实现、RBAC、会话管理、输入验证、CORS/CSP、依赖安全。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-2 交流。** |
| auditor-2 | **监控与变更管理维度审计**：系统监控（CC7.2）、变更管理（CC8.1）。审查日志记录、审计追踪、告警机制、Git 工作流、PR 审核流程、CI/CD 管道安全。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-1 交流。** |
| analyst | 合并两份审计报告和 scanner 扫描结果，去重、交叉校验、识别缺失控制项，输出统一合规评估清单。**不直接阅读代码，只对比和合并报告。完成后关闭。** |
| reporter | 基于统一合规评估清单生成差距分析报告（含证据制品、整改计划、优先级矩阵、时间线），按 `--lang` 语言输出。**完成后关闭。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段一（步骤 3） | 阶段一扫描完成后（步骤 3） | 扫描报告交付后即释放 |
| auditor-1 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| auditor-2 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| analyst | 阶段二（步骤 6） | 阶段二合并完成后（步骤 9） | 统一合规评估清单输出后释放 |
| reporter | 阶段三（步骤 10） | 阶段三报告生成后（步骤 12） | 报告确认后释放 |

## SOC2 代码级控制项（V1 范围）

每个控制项按合规状态定级：

| 状态 | 含义 | 是否需要整改 |
|------|------|-------------|
| **符合** | 控制项已完整实现，有充分证据 | 无需整改 |
| **部分符合** | 控制项部分实现，存在缺口 | 需要整改 |
| **不符合** | 控制项未实现或实现严重不足 | 必须整改 |
| **不适用** | 控制项不适用于当前项目 | 无需整改 |

### V1 支持的 SOC2 控制项

| 控制项 | 名称 | 代码级检查范围 |
|--------|------|---------------|
| **CC6.1** | 逻辑访问控制 | 认证模式（JWT/Session/OAuth）、RBAC/ABAC 实现、会话管理、密码策略、MFA |
| **CC6.6** | 边界保护 | 输入验证、CORS 策略、CSP 配置、API 网关、请求过滤 |
| **CC6.8** | 漏洞管理 | 依赖扫描、已知 CVE 检测、安全补丁状态、依赖更新策略 |
| **CC7.2** | 系统监控 | 日志记录覆盖率、审计追踪实现、告警机制、安全事件日志 |
| **CC8.1** | 变更管理 | Git 工作流（分支保护）、PR 审核要求、CI/CD 管道配置、部署流程 |

## 审计维度分配

双 auditor 按各自负责的 SOC2 控制项独立审计，analyst 按权重合并为合规评估总分。

| 控制项 | 权重 | 负责人 | 审计内容 |
|--------|------|--------|---------|
| CC6.1 逻辑访问控制 | 25% | auditor-1 | 认证机制强度、JWT/Session 安全、RBAC/ABAC 实现、密码策略、MFA、最小权限原则 |
| CC6.6 边界保护 | 15% | auditor-1 | 输入验证覆盖度、CORS 策略、CSP 配置、API 请求过滤、文件上传限制 |
| CC6.8 漏洞管理 | 10% | auditor-1 | 依赖 CVE 检测、安全补丁状态、依赖更新策略、已知漏洞修复时效 |
| CC7.2 系统监控 | 25% | auditor-2 | 日志记录覆盖率、审计追踪完整性、安全事件日志、告警机制、日志保护（防篡改） |

**auditor-1 总权重：50%**（CC6.1 逻辑访问控制 25% + CC6.6 边界保护 15% + CC6.8 漏洞管理 10%）
**auditor-2 总权重：50%**（CC7.2 系统监控 25% + CC8.1 变更管理 25%）

## 阶段零：框架选择 + 项目扫描

### 步骤 1：解析合规框架与审计范围

Team lead 分析项目和参数：

1. 解析 `--framework` 参数，确定合规框架（V1 仅支持 SOC2，其他框架提示并回退）
2. 阅读项目结构，识别技术栈（语言、框架、数据库、中间件）
3. 识别合规相关代码模式：
   - 认证与授权实现（登录、Token、权限中间件）
   - 日志与审计追踪（日志框架、日志级别、审计日志）
   - 输入验证与边界保护（校验库、CORS 配置、CSP 头）
   - CI/CD 配置（GitHub Actions、GitLab CI、Jenkins 等）
   - 依赖管理（package.json、requirements.txt、go.mod 等）
   - 密钥管理（环境变量、配置文件、KMS 集成）
4. 输出审计范围清单（含控制项与项目代码的映射关系）

### 步骤 2：用户确认审计范围

**标准模式**：向用户展示审计范围清单，AskUserQuestion 确认：
- 确认合规框架和审计范围
- 调整控制项（增删特定控制项）
- 补充合规关注点

**单轮确认模式**：跳过确认，直接进入阶段一。
**完全自主模式**：自动决策，不询问用户，直接进入阶段一。

## 阶段一：自动化控制检查 + 双路代码级审计

### 步骤 3：启动 scanner 和双 auditor

三者并行启动。

**Scanner 自动化控制检查**：
1. 阅读项目结构，识别技术栈和包管理器
2. 识别项目中已配置的安全/合规工具（.semgrep.yml、.gitleaks.toml、CI 配置等）
3. 按控制项运行对应检查工具，收集客观数据：

| 控制项 | 执行的检查类别 | 输出内容 |
|--------|-------------|---------|
| CC6.1 逻辑访问控制 | 认证模式扫描 + 密钥泄露扫描 | 认证实现模式、暴露的凭证 |
| CC6.6 边界保护 | 认证模式扫描（输入验证规则） | 输入验证覆盖率、CORS/CSP 配置 |
| CC6.8 漏洞管理 | 依赖漏洞扫描 | 依赖 CVE 列表、补丁状态 |
| CC7.2 系统监控 | 日志覆盖率检测 | 日志记录覆盖率、缺失的审计点 |
| CC8.1 变更管理 | CI/CD 配置审计 + Git 工作流检查 | CI/CD 安全配置、分支保护状态 |

4. 汇总输出**自动化控制检查报告**发送给 team lead
5. Scanner 完成后关闭（不参与后续阶段）

**双 auditor 同时阅读项目**：
- 阅读项目结构（目录布局、入口文件、配置文件）
- 理解技术栈、框架和合规相关机制
- 梳理各自负责控制项的代码实现链路
- 各自输出项目合规概况给 team lead

### 步骤 4：分发 scanner 报告

Team lead 收到 scanner 报告后，将自动化检查结果分发给两位 auditor，作为代码级审计的参考数据。

Auditor 在审计时必须参考 scanner 结果——例如 scanner 发现依赖中存在已知 CVE，auditor-1 在漏洞管理维度应当纳入评估。

### 步骤 5：独立并行代码级审计

两位 auditor 各自审计负责的控制项，**互不交流**。

**Auditor-1（访问控制维度）** — 对应 SOC2 控制项 CC6.1 / CC6.6 / CC6.8：

审计时必须逐项检查以下控制项，每个控制项输出"合规状态 + 发现/无发现 + 证据"：

| 控制项 | 审计内容 | 具体检查项 |
|--------|---------|-----------|
| **CC6.1 逻辑访问控制** | 认证与授权实现 | 认证机制类型和强度（JWT 签名验证、Session 安全配置）、密码存储算法（必须 bcrypt/scrypt/argon2）、RBAC/ABAC 实现完整性、最小权限原则执行、MFA 实现（如适用）、Token 过期和刷新策略、登出处理（Token 吊销） |
| **CC6.6 边界保护** | 输入验证与边界安全 | 输入参数校验覆盖度、白名单/黑名单策略、CORS 策略（`Access-Control-Allow-Origin` 配置）、CSP 头配置、文件上传类型/大小限制、API 速率限制、请求过滤（SQL 注入/XSS 防护） |
| **CC6.8 漏洞管理** | 依赖安全与补丁管理 | 依赖组件已知 CVE（参考 scanner SCA 结果）、过期依赖数量和风险、安全补丁应用时效、依赖更新策略（Dependabot/Renovate 配置）、不安全的依赖源 |

- 参考 scanner 的自动化检查结果，验证发现的准确性
- 记录每个发现：控制项编号 + 文件路径 + 行号 + 合规状态（符合/部分符合/不符合/不适用） + 发现描述 + 证据引用 + 整改建议

**Auditor-2（监控与变更管理维度）** — 对应 SOC2 控制项 CC7.2 / CC8.1：

审计时必须逐项检查以下控制项，每个控制项输出"合规状态 + 发现/无发现 + 证据"：

| 控制项 | 审计内容 | 具体检查项 |
|--------|---------|-----------|
| **CC7.2 系统监控** | 日志记录与审计追踪 | 日志框架使用和配置、关键操作日志覆盖率（登录/登出/权限变更/数据修改）、审计追踪完整性（谁/何时/做了什么/结果）、安全事件日志（失败登录/权限拒绝/异常访问）、日志保护机制（防篡改/访问控制）、告警机制（异常行为触发告警）、敏感数据是否出现在日志中（密码/Token/PII 不应明文记录） |
| **CC8.1 变更管理** | Git 工作流与 CI/CD | Git 分支保护策略（main/master 分支保护规则）、PR 审核要求（最少审核人数、必须审核通过才能合并）、CI/CD 管道配置（自动化测试/lint/安全扫描）、部署流程（手动审批/自动部署/回滚机制）、环境隔离（dev/staging/prod 分离）、密钥管理（CI/CD 中的 secrets 管理方式）、构建制品完整性验证 |

- 参考 scanner 的 CI/CD 配置审计和日志覆盖率检测结果，验证并补充
- 记录每个发现：控制项编号 + 文件路径 + 行号 + 合规状态（符合/部分符合/不符合/不适用） + 发现描述 + 证据引用 + 整改建议

## 阶段二：差距分析

### 步骤 6：启动 analyst

两位 auditor 审计完成后，启动 analyst。Team lead 将以下材料发送给 analyst：
- Scanner 自动化控制检查报告
- Auditor-1 审计报告
- Auditor-2 审计报告
- 项目合规边界信息

### 步骤 7：合并去重与差距识别

Analyst 执行合并分析：

1. **去重**：识别 scanner 结果与 auditor 手工发现的重叠项，合并为单条记录
2. **交叉校验**：检查两位 auditor 是否对同一控制项有不同评级
3. **差距识别**：逐个控制项标注合规状态，识别缺失的控制实现
4. **统一评级**：按 SOC2 标准统一评级（符合/部分符合/不符合/不适用）
5. **影响评估**：评估每个不符合项的实际业务影响和合规风险

**共识度计算公式**：
```
两位 auditor 审计的控制项无直接重叠，但部分检查点可能交叉（如 auditor-1 的 CC6.6 输入验证与 auditor-2 的 CC7.2 日志中的输入校验日志）。
对于重叠区域的发现：

共识发现数 = 两位 auditor 都识别出的问题数量（指向相同文件/行号且描述相同问题类型）
总重叠区域发现数 = 重叠区域去重后的问题总数

共识度 = 共识发现数 / 总重叠区域发现数 × 100%（如无重叠区域发现则为 100%）

评级一致性 = 两位 auditor 对同一问题评级完全一致的比例
争议率 = 评级差异 ≥ 2 级的问题数 / 两位 auditor 都识别的问题数 × 100%

争议率 > 50% → 触发熔断，必须暂停问用户
```

### 步骤 8：差异校验与熔断检查

**差异校验**：
- 如果两位 auditor 对同一问题的评级差异 ≥ 2 级（如一方定"不符合"、另一方定"符合"），analyst 标注为"争议项"
- 争议项占比 > 50% → **不可跳过，必须暂停问用户**（熔断机制）

**"不符合"控制项熔断**：
- 统一评估清单中"不符合"控制项超过 5 个 → **不可跳过，必须暂停问用户**（熔断机制）
- 向用户展示"不符合"控制项列表，确认是否继续生成完整报告

**无熔断触发**时，analyst 输出**统一合规评估清单**：

```
## 统一合规评估清单

### 控制项统计
- 符合: X 个 | 部分符合: X 个 | 不符合: X 个 | 不适用: X 个
- 总计: X 个控制项

### 控制项评估

#### CC6.1 逻辑访问控制 — 状态：符合/部分符合/不符合
  ├─ [符合] 认证机制实现 - 文件:行号 - 证据描述 - 来源：scanner/auditor-1
  ├─ [部分符合] RBAC 实现 - 文件:行号 - 缺失描述 - 来源：auditor-1
  └─ 影响评估：[合规风险描述]

#### CC6.6 边界保护 — 状态：部分符合
  └─ ...

#### CC6.8 漏洞管理 — 状态：不符合
  └─ ...

#### CC7.2 系统监控 — 状态：部分符合
  └─ ...

#### CC8.1 变更管理 — 状态：符合
  └─ ...

### 争议项（如有）
1. 控制项:检查点 - auditor-1 评级: X / auditor-2 评级: Y - analyst 最终评级: Z - 理由
```

Analyst 完成后关闭。

### 步骤 9：合规评估评分

Team lead 根据统一合规评估清单计算各控制项合规评分：

评分规则：
- 基础分 10.0，按发现扣分
- "不符合"发现：每个扣 2.0 分（该控制项最多扣至 0 分）
- "部分符合"发现：每个扣 1.0 分
- "符合"：不扣分
- "不适用"：不扣分，不计入权重

按权重计算总分，输出各控制项和总体合规评分。

## 核心原则

- **控制导向**：以 SOC2 控制项为核心，逐项检查代码级实现是否满足控制要求
- **双路独立**：两位 auditor 从不同控制维度独立审计，互不交流，避免盲区和偏见
- **证据驱动**：每个合规评估必须有代码级证据支撑，不做主观判断
- **差距聚焦**：重点识别缺失的控制实现，而非罗列已有的安全措施
- **可操作性**：整改建议必须具体到代码级别，可直接执行，避免空泛建议
- **持续合规**：不仅识别当前差距，还建议持续监控机制防止合规退化
- **最小范围**：V1 仅覆盖 SOC2 代码级控制子集，不扩大范围，确保深度优于广度

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 扫描工具未安装/无法运行 | Scanner 标注"该工具不可用"，不阻塞流程，auditor 基于手工审计评估 |
| 项目无法构建/运行 | 仅做静态审计（代码审查 + 配置检查），报告中标注"未验证运行时行为" |
| 代码仓库过大（>100 万行） | Scanner 按模块分批扫描；auditor 聚焦合规相关模块，标注未覆盖区域 |
| 两位 auditor 评级差异过大（争议项 > 50%） | 触发熔断，暂停问用户裁决争议项 |
| "不符合"控制项超过 5 个 | 触发熔断，暂停向用户确认是否继续完整报告 |
| 用户指定非 SOC2 框架 | 提示"V1 仅支持 SOC2 代码级控制子集"，回退到 SOC2 审计 |
| CI/CD 配置不可访问（私有仓库设置） | Auditor-2 标注"无法审计 CI/CD 配置"，基于可用信息评估 |
| 无 Git 历史记录 | Auditor-2 标注"无法审计变更管理历史"，仅审计当前配置 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前步骤恢复 |
| 审计范围与实际项目不匹配 | Team lead 在阶段零发现后调整范围，标准模式需用户确认 |
