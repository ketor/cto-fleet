**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）
- `--db=postgres|mysql|sqlite|mongo|auto`：数据库类型（默认 `auto` 自动检测）
- `--action=design|migrate|audit|optimize`：操作类型（默认根据需求描述自动推断）
  - `design`：新 Schema 设计或现有 Schema 改进
  - `migrate`：生成迁移脚本
  - `audit`：Schema 健康审计（规范性、性能、安全）
  - `optimize`：索引和查询性能优化

解析后将标志从需求描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 扫描结果确认、设计方案确认、迁移脚本确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅迁移脚本确认（因为数据库变更有高风险） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **扫描数据不完整** → team lead 自行判断可用数据范围，在迁移脚本确认时告知用户
- **designer-1/designer-2 设计分歧** → team lead 综合两方设计裁决，在方案确认时附带分歧摘要和裁决理由
- **兼容性审查发现破坏性变更** → reviewer 标记风险等级，在迁移脚本确认时重点展示
- **分歧度 > 50%** → **不可跳过，必须暂停问用户**（熔断机制）
- **迁移脚本验证失败超 3 轮** → **不可跳过，必须暂停问用户**（熔断机制）

完全自主模式下：所有节点（包括扫描确认、设计确认、迁移脚本确认等）均自动决策，不询问用户。收尾时汇总所有自动决策。唯一例外：验证失败超 3 轮仍自动停止（但不询问用户，而是 team lead 自行裁决是否继续）。

使用 TeamCreate 创建 team（名称格式 `team-schema-{YYYYMMDD-HHmmss}`，如 `team-schema-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  Schema 扫描 → scanner 收集 Schema 文件、迁移历史、查询模式
         ↓
阶段一  双路独立设计 → designer-1（Schema 演进 + 范式化）
                     + designer-2（索引优化 + 查询性能）
         → team lead merge 两份设计
         ↓
阶段二  共识合并 → 共识/分歧/仲裁 → 输出合并后的 Schema 方案 + 兼容性分析
         ↓
阶段三  兼容性审查 → reviewer 验证向后兼容性 + 回滚安全性
         ↓
阶段四  迁移脚本生成 → writer 生成迁移脚本 + 文档
         ↓
阶段五  验证收尾 → 脚本语法校验 + dry-run + 回滚测试 + 输出报告
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 扫描项目数据库相关文件，收集现有 Schema 定义、迁移历史、ORM 模型、查询模式。**只收集数据，不做设计。** |
| designer-1 | 独立设计，**从 Schema 演进和范式化角度**：表结构设计、字段类型选择、关系定义、范式化/反范式化权衡、Schema 版本演进路径。输出 Schema 设计方案和演进说明。**独立工作阶段不与 designer-2 交流。** |
| designer-2 | 独立设计，**从索引优化和查询性能角度**：索引策略、查询计划分析、分区策略、读写分离考量、热点数据处理。输出索引方案和性能优化建议。**独立工作阶段不与 designer-1 交流。** |
| reviewer | 审查合并后的 Schema 方案，验证向后兼容性、回滚安全性、数据迁移风险。**只做审查，不修改设计。** |
| writer | 根据审查通过的方案生成迁移脚本、回滚脚本和文档。**严格按审查结论执行。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段零（步骤 1） | 阶段零完成后（步骤 2 数据收集完毕） | 数据交付后即释放 |
| designer-1 | 阶段一（步骤 3） | 阶段二完成后（步骤 6 共识合并完成） | 共识确认后释放 |
| designer-2 | 阶段一（步骤 3） | 阶段二完成后（步骤 6 共识合并完成） | 共识确认后释放 |
| reviewer | 阶段三（步骤 7） | 阶段三完成后（步骤 8 审查通过） | 审查报告交付后释放 |
| writer | 阶段四（步骤 9） | 阶段五验证通过后（步骤 11） | 验证失败时保留用于迭代 |

## 阶段零：Schema 扫描

### 步骤 1：启动 scanner

启动 scanner，传入项目路径和 `--db` 参数。

Scanner 执行以下数据收集工作：

1. **数据库类型检测**（`--db=auto` 时）：
   - 检查配置文件中的数据库连接信息（`DATABASE_URL`、`database.yml`、`settings.py`、`.env`）
   - 检查 ORM 框架配置（SQLAlchemy、TypeORM、GORM、Prisma、ActiveRecord、Django ORM）
   - 检查依赖文件中的数据库驱动（`pg`、`mysql2`、`sqlite3`、`mongoose`）
   - 检查迁移工具配置（`alembic`、`flyway`、`knex`、`goose`、`dbmate`、`prisma migrate`）

2. **Schema 文件收集**：
   - SQL Schema 文件（`schema.sql`、`*.sql` DDL 文件）
   - ORM 模型文件（`models/`、`entities/`、`schema.prisma`）
   - 迁移文件目录（`migrations/`、`db/migrate/`、`alembic/versions/`）
   - Schema 定义文件（`schema.prisma`、`schema.graphql`、`*.proto`）
   - 种子数据文件（`seeds/`、`fixtures/`）

3. **迁移历史分析**：
   - 收集所有迁移文件，按时间排序
   - 分析迁移模式：主要是 ADD/ALTER/DROP 哪类操作
   - 识别已回滚/失败的迁移（如有记录）
   - 统计迁移频率和复杂度趋势

4. **查询模式收集**：
   - 扫描代码中的数据库查询（ORM 查询、raw SQL、Repository 模式）
   - 识别频繁使用的查询模式（JOIN、子查询、聚合）
   - 识别潜在的 N+1 查询模式
   - 收集查询日志/慢查询日志（如有）

5. **现有索引和约束分析**：
   - 收集所有索引定义（单列、复合、唯一、部分索引）
   - 收集所有约束定义（主键、外键、唯一约束、检查约束）
   - 识别缺失的索引（有外键但无索引的情况）

### 步骤 2：扫描结果汇总

Scanner 输出结构化的扫描报告，包含：
- 数据库类型和版本信息
- 表/集合清单（名称、字段数、行数估算、关系）
- 迁移历史摘要（总数、最近变更、变更趋势）
- 查询模式统计（频繁查询、慢查询、N+1 候选）
- 索引覆盖情况
- 现有约束和完整性规则
- Schema 规范性初步评估（命名约定、类型一致性）

确认方式：
- **标准模式**：向用户展示扫描报告，AskUserQuestion 确认数据是否准确、是否需要补充
- **单轮确认模式**：team lead 自行确认，在迁移脚本确认时一并展示
- **完全自主模式**：自动决策，不询问用户

## 阶段一：双路独立设计

### 步骤 3：启动 designer-1 和 designer-2

同时启动两位 designer，分别传入扫描报告和 `--action` 参数。**两位 designer 独立工作，team lead 不让两位 designer 看到对方的设计，确保独立性。**

**designer-1（Schema 演进 + 范式化设计）**：
- 分析现有 Schema 的范式化程度（1NF/2NF/3NF/BCNF）
- 识别数据冗余和更新异常风险
- 设计 Schema 演进方案：
  - 新增表/字段的设计（数据类型、默认值、约束）
  - 表拆分/合并建议（规范化 vs 反范式化的 trade-off）
  - 字段类型优化（varchar 长度、数值精度、时间类型）
  - 关系设计（外键、关联表、嵌入式文档 for NoSQL）
- 评估命名规范一致性（表名、字段名、索引名的命名约定）
- 设计数据完整性约束（NOT NULL、UNIQUE、CHECK、DEFAULT）
- 考虑多租户/软删除/审计字段等通用模式
- 输出：Schema 演进方案 + 范式化评估 + ER 图（文本描述）+ 置信度

**designer-2（索引优化 + 查询性能设计）**：
- 分析现有查询模式，识别缺失索引和冗余索引
- 设计索引策略：
  - 单列索引 vs 复合索引选择
  - 覆盖索引设计（避免回表查询）
  - 部分索引/条件索引优化（WHERE 子句索引）
  - 全文搜索索引（如需要）
- 分析查询计划，识别全表扫描和低效 JOIN
- 设计分区策略（时间分区、范围分区、哈希分区，如适用）
- 评估读写比例，建议读写分离方案（如适用）
- 识别热点数据和缓存策略
- 估算存储空间和索引大小影响
- 输出：索引方案 + 查询优化建议 + 性能预期 + 置信度

### 步骤 4：设计质量要求

每位 designer 的输出必须包含：
1. **变更清单**：按优先级排序（Critical / High / Medium / Low）
2. **设计理由**：每个变更有明确的技术理由和业务依据
3. **置信度**：高/中/低，附带理由
4. **影响评估**：变更对应用代码、数据迁移、查询性能的影响
5. **风险标注**：潜在的数据丢失、停机时间、锁表风险

## 阶段二：共识合并

### 步骤 5：Team lead 合并两份设计

两份设计完成后，team lead 执行 merge：

1. **对比两份设计方案**，计算共识度和分歧度：

**共识度计算公式**：
```
对比维度（共 5 项，每项 20%）：
  ① 表结构（对表/字段的设计方向一致）
  ② 索引策略（索引选择和优先级一致）
  ③ 性能评估（对性能影响的判断一致）
  ④ 迁移复杂度（对迁移难度的评估一致）
  ⑤ 风险认知（对潜在风险的识别一致）

共识度 = 一致维度数 × 20%
分歧度 = 100% - 共识度
```

| 一致性结果 | 分歧度 | 处理方式 |
|-----------|--------|---------|
| **完全一致** | 0% | 直接合并为统一的 Schema 方案 |
| **互补型** | < 30%（≤1 项分歧） | 两者从不同角度设计，合并为完整方案 |
| **部分分歧** | 30%-50%（2-3 项分歧） | Team lead 综合设计理由判断，选择更优方案或融合两者 |
| **严重分歧** | > 50%（≥3 项分歧） | 触发熔断：必须暂停，AskUserQuestion 向用户展示两份设计，请用户提供额外信息或判断方向 |

**分歧处理策略**：
- **共识部分**：直接纳入合并方案，标记为"双方共识"
- **互补部分**：一方设计了另一方未覆盖的维度，合并纳入，标记为"互补设计"
- **冲突部分**：两方对同一表/字段/索引的设计不一致，team lead 比较设计理由和性能数据，采纳更优方案，标记为"仲裁结论"并记录被否决方的设计
- **盲区检查**：team lead 检查是否存在两方都未覆盖的设计角度（如安全性、审计需求），标注为"未覆盖风险"

2. **整合为统一的 Schema 方案**：包含表结构 + 索引策略 + 约束定义
3. **输出合并报告**：变更总数 + 按优先级分布 + 分歧处理说明

### 步骤 6：设计方案确认

确认方式：
- **标准模式**：向用户展示合并方案（含两份独立设计摘要、merge 结论、分歧处理说明），AskUserQuestion 确认设计方案
- **单轮确认模式**：team lead 自行确认（分歧度 ≤ 50% 时）；分歧度 > 50% 时必须问用户
- **完全自主模式**：自动决策，不询问用户

## 阶段三：兼容性审查

### 步骤 7：启动 reviewer

启动 reviewer，传入合并后的 Schema 方案和现有 Schema 信息。

Reviewer 执行兼容性审查：

1. **向后兼容性检查**：
   - 是否有破坏性变更（DROP COLUMN/TABLE、类型缩窄、NOT NULL 无默认值）
   - 是否有隐式破坏（重命名字段导致应用查询失败）
   - 是否影响现有 API/ORM 映射
   - 是否影响现有的视图、存储过程、触发器

2. **回滚安全性验证**：
   - 每个迁移步骤是否可回滚
   - 回滚后数据是否完整（不可逆操作标记：如 DROP COLUMN 数据丢失）
   - 回滚脚本是否覆盖所有变更
   - 部分执行失败时的处理方案

3. **数据迁移风险评估**：
   - 大表 ALTER 的锁表风险和预计时间
   - 数据回填操作的耗时和内存估算
   - 并发影响评估（在线 DDL vs 停机维护）
   - 数据完整性验证方案

4. **安全性检查**：
   - 敏感字段是否有加密/脱敏处理
   - 权限变更是否合理
   - 是否暴露了不应暴露的数据

5. **输出审查报告**：
   - 兼容性评级（安全/有风险/破坏性）
   - 风险项清单（按严重性排序）
   - 每个风险项的缓解方案
   - 推荐的迁移策略（在线迁移 / 维护窗口 / 蓝绿部署）
   - 审查结论（通过 / 有条件通过 / 不通过）

### 步骤 8：审查结果处理

| 审查结论 | 处理方式 |
|---------|---------|
| **通过** | 进入阶段四生成迁移脚本 |
| **有条件通过** | Reviewer 列出需要满足的条件（如添加默认值、分步执行），team lead 调整方案后进入阶段四 |
| **不通过** | Team lead 根据审查意见调整设计方案，重新提交审查。最多迭代 2 轮 |

确认方式（审查报告）：
- **标准模式**：向用户展示审查报告，AskUserQuestion 确认
- **单轮确认模式**：team lead 自行处理，在迁移脚本确认时一并展示审查结论
- **完全自主模式**：自动决策，不询问用户

## 阶段四：迁移脚本生成

### 步骤 9：启动 writer

启动 writer，传入审查通过的 Schema 方案、审查报告和现有迁移工具信息。

**Writer 生成迁移脚本和文档**：

1. **迁移脚本（up）**：
   - 按迁移工具的格式生成（Alembic、Flyway、Knex、goose、Prisma 等）
   - 每个变更一个独立的迁移步骤（原子化）
   - 包含事务控制（BEGIN/COMMIT/ROLLBACK）
   - 大表操作添加在线 DDL 注释（如 `pt-online-schema-change`、`gh-ost`）
   - 数据回填操作添加批次处理逻辑
   - 添加迁移前置检查（如检查表是否存在、字段是否已添加，保证幂等性）

2. **回滚脚本（down）**：
   - 每个 up 迁移对应一个 down 回滚
   - 标注不可逆操作（如 DROP COLUMN）并添加警告注释
   - 数据回填的回滚策略

3. **验证脚本**：
   - 迁移后数据完整性验证 SQL
   - 索引有效性验证查询
   - 行数/数据一致性检查

4. **迁移文档**：
   - 迁移执行顺序和依赖关系
   - 预估执行时间（按表大小估算）
   - 所需的维护窗口（如有停机需求）
   - 紧急回滚步骤
   - 相关应用代码变更提示（需要同步修改的 ORM 模型/查询）

### 步骤 10：Writer 自检 + 方案确认

迁移脚本生成后，writer 执行自检清单：

1. **SQL 语法校验**：使用对应数据库的语法检查（如有工具）
2. **幂等性检查**：重复执行迁移是否安全（IF NOT EXISTS、IF EXISTS）
3. **回滚完整性**：每个 up 是否有对应 down
4. **命名规范**：迁移文件命名符合项目约定
5. **依赖顺序**：外键引用的表先创建、后引用

自检全部通过后，通知 team lead。

确认方式（迁移脚本）：
- **标准模式和单轮确认模式均需确认**：向用户展示迁移脚本（含 Schema 方案摘要、审查结论、迁移步骤、回滚方案、风险提示），AskUserQuestion 确认后再应用。这是唯一在单轮确认模式下仍需用户确认的节点（因为数据库变更有高风险，需要用户知情同意）
- **完全自主模式**：自动决策，不询问用户

## 阶段五：验证收尾

### 步骤 11：验证与报告

1. **语法验证**：对生成的 SQL/迁移文件做语法校验
2. **Dry-run 验证**（如环境支持）：在测试数据库上试运行迁移，确认无报错
3. **回滚测试**（如环境支持）：执行 up 后执行 down，确认回滚无报错
4. **数据完整性验证**：运行验证脚本，确认数据迁移后的完整性

| 验证结果 | 处理方式 |
|---------|---------|
| **全部通过** | 进入步骤 12 总结 |
| **SQL 语法错误** | 打回 writer 修复语法 |
| **迁移执行失败** | Writer 分析失败原因，调整迁移脚本 |
| **回滚失败** | Writer 修复回滚脚本 |
| **数据完整性问题** | Writer 添加数据修复步骤 |

**验证失败迭代**：writer 修复后重新验证，循环直到通过。**最大迭代轮次：3 轮。** 超出后触发熔断，team lead 向用户汇报当前状态（已通过/未通过的验证项、失败原因分析），由用户决定继续、调整方向或终止。

## 核心原则

- **数据安全第一**：任何 Schema 变更必须评估数据丢失风险，破坏性变更必须明确标注
- **独立设计**：两位 designer 独立工作，避免锚定偏差，确保设计多样性
- **向后兼容**：优先选择向后兼容的变更方案，最小化对现有应用的影响
- **可回滚**：每个迁移步骤都必须有回滚方案，标注不可逆操作
- **原子迁移**：每个迁移文件做一件事，保持原子性和幂等性
- **性能意识**：Schema 设计必须考虑查询性能，索引策略必须基于实际查询模式
- **分歧即信号**：两路设计分歧度高意味着问题复杂度高，必须谨慎处理
- **有限迭代**：验证失败最多重试 3 轮，超出则暂停问用户
- **不确定就问**：标准模式下关键节点必须用户确认；单轮确认模式下迁移脚本仍需确认；完全自主模式下全部自动决策

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 无法检测数据库类型 | Scanner 提示 team lead，team lead 用 AskUserQuestion 向用户确认数据库类型和版本 |
| 无 Schema 文件（纯 ORM 项目） | Scanner 从 ORM 模型文件反推 Schema 定义，标注为"推断 Schema" |
| 无迁移工具 | Writer 生成原生 SQL 迁移脚本，并建议用户选择迁移工具 |
| 两位 designer 设计严重分歧（>50%） | 触发熔断，暂停问用户。用户可提供额外信息或指定设计方向 |
| 迁移脚本语法校验失败 | Writer 修复语法问题；3 轮未解决则暂停问用户 |
| 大表 ALTER 风险过高 | Reviewer 标记为"需要在线 DDL 工具"，writer 生成对应工具的命令（pt-osc、gh-ost） |
| 数据回填量过大 | Writer 设计分批回填方案，添加进度监控和可中断机制 |
| Schema 与 ORM 模型不一致 | Scanner 标记不一致项，designer 在设计中优先修复不一致 |
| 外键约束导致迁移顺序复杂 | Writer 分析依赖图，自动排序迁移顺序，处理循环依赖 |
| 多数据库项目（如读写分离） | Scanner 分别收集各数据库 Schema，designer 分别设计，writer 生成同步迁移方案 |
| NoSQL Schema-less 场景 | Designer 转为文档结构设计，生成 Schema 验证规则（JSON Schema、Mongoose Schema） |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| Git 仓库状态异常（脏工作区） | Team lead 在阶段零检测工作区状态，提示用户先 stash 或提交变更 |
