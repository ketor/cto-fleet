**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）
- `--platform=github-actions|gitlab-ci|jenkins|circleci|auto`：CI/CD 平台（默认 `auto` 自动检测）
- `--fix`：自动应用优化配置（默认仅输出分析报告和建议）

解析后将标志从需求描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 扫描结果确认、分析结果确认、优化方案确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅优化方案确认（因为修改流水线配置有风险） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **扫描数据不完整** → team lead 自行判断可用数据范围，在优化方案确认时告知用户
- **analyzer-1/analyzer-2 分析分歧** → team lead 综合两方分析裁决，在优化方案确认时附带分歧摘要和裁决理由
- **优化后验证失败** → writer 自行调整配置重试
- **分歧度 > 50%** → **不可跳过，必须暂停问用户**（熔断机制）
- **优化后验证失败超 3 轮** → **不可跳过，必须暂停问用户**（熔断机制）

完全自主模式下：所有节点（包括扫描确认、分析确认、优化方案确认等）均自动决策，不询问用户。收尾时汇总所有自动决策。唯一例外：验证失败超 3 轮仍自动停止（但不询问用户，而是 team lead 自行裁决是否继续）。

使用 TeamCreate 创建 team（名称格式 `team-cicd-{YYYYMMDD-HHmmss}`，如 `team-cicd-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  流水线扫描 → scanner 收集流水线配置、构建日志、执行时间数据
         ↓
阶段一  双路独立分析 → analyzer-1（构建时间 + 瓶颈分析）
                     + analyzer-2（Flakiness + 失败模式分析）
         → team lead merge 两份分析
         ↓
阶段二  共识合并 → 共识/分歧/仲裁 → 输出优先级排序的问题清单
         ↓
阶段三  优化设计 → optimizer 设计流水线改进方案（并行化、缓存、步骤精简）
         ↓
阶段四  配置生成 → writer 生成优化后的流水线配置文件（仅 --fix 模式）
         ↓
阶段五  验证收尾 → 配置语法校验 + dry-run 验证 + 输出报告
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 扫描项目 CI/CD 配置文件、构建日志、执行历史，收集时间线数据和失败记录。**只收集数据，不做分析。** |
| analyzer-1 | 独立分析，**从构建时间和瓶颈角度**：各步骤耗时分布、关键路径、串行瓶颈、缓存命中率、依赖安装耗时、构建产物大小。输出瓶颈清单和优化潜力评估。**独立工作阶段不与 analyzer-2 交流。** |
| analyzer-2 | 独立分析，**从 Flakiness 和失败模式角度**：频繁失败的步骤/测试、失败的时间模式（特定时段/特定分支）、超时配置合理性、重试机制有效性、环境依赖稳定性。输出失败模式清单和稳定性评估。**独立工作阶段不与 analyzer-1 交流。** |
| optimizer | 根据合并后的分析结果，设计流水线优化方案：并行化策略、缓存方案、步骤合并/精简、条件执行优化。**只设计方案，不写配置。** |
| writer | 根据优化方案生成具体的流水线配置文件和说明文档。**只在 `--fix` 模式下启动。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段零（步骤 1） | 阶段零完成后（步骤 2 数据收集完毕） | 数据交付后即释放 |
| analyzer-1 | 阶段一（步骤 3） | 阶段二完成后（步骤 6 共识合并完成） | 共识确认后释放 |
| analyzer-2 | 阶段一（步骤 3） | 阶段二完成后（步骤 6 共识合并完成） | 共识确认后释放 |
| optimizer | 阶段三（步骤 7） | 阶段三完成后（步骤 8 方案确认） | 方案交付后释放 |
| writer | 阶段四（步骤 9） | 阶段五验证通过后（步骤 11） | 仅 --fix 模式启动；验证失败时保留用于迭代 |

## 阶段零：流水线扫描

### 步骤 1：启动 scanner

启动 scanner，传入项目路径和 `--platform` 参数。

Scanner 执行以下数据收集工作：

1. **CI/CD 平台检测**（`--platform=auto` 时）：
   - 检查 `.github/workflows/` → GitHub Actions
   - 检查 `.gitlab-ci.yml` → GitLab CI
   - 检查 `Jenkinsfile` / `jenkins/` → Jenkins
   - 检查 `.circleci/config.yml` → CircleCI
   - 检查 `Dockerfile`、`docker-compose*.yml` → Docker 构建相关
   - 检查 `Makefile`、`Taskfile.yml` → 本地构建工具
   - 如果存在多个平台配置，全部收集

2. **配置文件收集**：
   - 收集所有 CI/CD 配置文件的完整内容
   - 解析 workflow/pipeline 结构：触发条件、job 定义、step 列表、依赖关系
   - 识别缓存配置（dependency cache、build cache、layer cache）
   - 识别 artifact/产物配置（上传/下载/共享）
   - 识别 secret/环境变量引用

3. **构建日志和历史收集**：
   - 如有 CI API 访问权限（gh CLI 等），收集最近 20-50 次构建记录
   - 提取每次构建的：总耗时、各步骤耗时、成功/失败状态、触发分支/事件
   - 如无 API 访问，基于配置文件静态分析估算
   - 检查 CI 日志文件（如本地有缓存）

4. **依赖和环境信息**：
   - 识别使用的 base image 和版本
   - 识别安装的依赖和工具
   - 识别并行度配置（matrix、parallel、concurrency）
   - 识别条件执行逻辑（paths-filter、if 条件、skip 标记）

### 步骤 2：扫描结果汇总

Scanner 输出结构化的扫描报告，包含：
- CI/CD 平台类型和版本
- 配置文件清单和结构摘要
- Workflow/Pipeline 拓扑图（job 依赖关系）
- 历史构建数据摘要（平均耗时、P50/P90/P99 耗时、成功率）
- 缓存配置状态
- 已知问题/告警（如：deprecated action、过期 image）

确认方式：
- **标准模式**：向用户展示扫描报告，AskUserQuestion 确认数据是否准确、是否需要补充
- **单轮确认模式**：team lead 自行确认，在优化方案确认时一并展示
- **完全自主模式**：自动决策，不询问用户

## 阶段一：双路独立分析

### 步骤 3：启动 analyzer-1 和 analyzer-2

同时启动两位 analyzer，分别传入扫描报告。**两位 analyzer 独立工作，team lead 不让两位 analyzer 看到对方的分析，确保独立性。**

**analyzer-1（构建时间 + 瓶颈分析）**：
- 分析各步骤耗时分布，识别关键路径（最长执行链）
- 识别串行瓶颈：可以并行但当前串行执行的步骤
- 评估缓存有效性：dependency cache 命中率、build cache 利用率、layer cache 效果
- 分析依赖安装耗时：是否每次全量安装、lock file 是否利用缓存
- 评估构建产物大小和传输耗时
- 识别冗余步骤：重复执行的操作、可合并的步骤
- 检查资源配置：runner 规格是否匹配需求、并行度是否合理
- 输出：瓶颈清单（按节省时间潜力排序）+ 优化建议 + 预期时间节省

**analyzer-2（Flakiness + 失败模式分析）**：
- 分析失败频率和分布：哪些步骤/测试最常失败
- 识别 Flaky 失败模式：
  - 时间相关（超时配置不合理、网络波动）
  - 环境相关（runner 状态不一致、依赖版本漂移）
  - 并发相关（资源竞争、端口冲突、数据库竞态）
  - 顺序相关（job 依赖不正确、artifact 传递失败）
- 分析重试策略有效性：当前重试是否解决问题还是掩盖根因
- 评估超时配置：过短导致假失败、过长浪费资源
- 检查分支策略影响：特定分支/事件触发的失败率差异
- 识别安全隐患：secret 暴露风险、权限过宽、未固定版本的 action/image
- 输出：失败模式清单（按影响频率排序）+ 稳定性评分 + 修复建议

### 步骤 4：分析质量要求

每位 analyzer 的输出必须包含：
1. **问题清单**：按优先级排序（Critical / High / Medium / Low）
2. **证据链**：每个问题有配置/日志级证据（文件路径、行号、日志片段）
3. **置信度**：高/中/低，附带理由
4. **影响评估**：该问题对构建速度/稳定性/安全性的具体影响
5. **改进建议**：具体的改进方向（但不涉及具体配置生成）

## 阶段二：共识合并

### 步骤 5：Team lead 合并两份分析

两份分析完成后，team lead 执行 merge：

1. **对比两份分析结果**，计算共识度和分歧度：

**共识度计算公式**：
```
对比维度（共 5 项，每项 20%）：
  ① 优先瓶颈（哪些步骤/配置最需要优化）
  ② 问题类型（识别出的主要问题类别一致）
  ③ 严重程度（对问题严重性的评估一致）
  ④ 改进方向（建议的改进方向一致）
  ⑤ 风险评估（对优化风险的评估一致）

共识度 = 一致维度数 × 20%
分歧度 = 100% - 共识度
```

| 一致性结果 | 分歧度 | 处理方式 |
|-----------|--------|---------|
| **完全一致** | 0% | 直接合并为统一的问题清单 |
| **互补型** | < 30%（≤1 项分歧） | 两者发现的是不同维度的问题，合并为完整清单 |
| **部分分歧** | 30%-50%（2-3 项分歧） | Team lead 综合证据链判断优先级，融合两者 |
| **严重分歧** | > 50%（≥3 项分歧） | 触发熔断：必须暂停，AskUserQuestion 向用户展示两份分析结果，请用户提供额外信息或判断方向 |

**分歧处理策略**：
- **共识部分**：直接纳入合并清单，标记为"双方共识"
- **互补部分**：一方发现另一方未覆盖的问题，合并纳入，标记为"互补发现"
- **冲突部分**：两方对同一问题的严重性或处理方式不一致，team lead 比较证据链强度，采纳更强的一方，标记为"仲裁结论"
- **盲区检查**：team lead 检查是否存在两方都未覆盖的分析角度，标注为"未覆盖风险"

2. **整合为优先级排序的问题清单**：按 Critical > High > Medium > Low 排序
3. **输出合并报告**：问题总数 + 按优先级分布 + 预期总优化效果 + 分歧处理说明

### 步骤 6：分析结果确认

确认方式：
- **标准模式**：向用户展示合并报告（含两份独立分析摘要、merge 结论、分歧处理说明），AskUserQuestion 确认分析结果
- **单轮确认模式**：team lead 自行确认（分歧度 ≤ 50% 时）；分歧度 > 50% 时必须问用户
- **完全自主模式**：自动决策，不询问用户

## 阶段三：优化设计

### 步骤 7：启动 optimizer

启动 optimizer，传入合并后的分析报告和平台信息。

Optimizer 设计流水线优化方案：

1. **并行化优化**：
   - 识别可以并行执行的 job/step（无依赖关系的任务）
   - 设计 matrix 策略（多平台/多版本并行测试）
   - 优化 job 依赖图，缩短关键路径
   - 评估并行度上限（runner 资源限制、API rate limit）

2. **缓存优化**：
   - 设计 dependency cache 策略（lock file hash key、fallback key）
   - 设计 build cache 策略（增量构建、构建产物缓存）
   - 设计 Docker layer cache 策略（如适用）
   - 评估缓存空间和命中率 trade-off

3. **步骤精简**：
   - 合并可合并的步骤（减少 step 开销）
   - 删除冗余步骤（重复安装、无效检查）
   - 条件执行优化（paths-filter 跳过不相关的检查）
   - 提前失败策略（fail-fast，将快速检查前置）

4. **稳定性优化**：
   - 修复 Flaky 步骤的根因（超时调整、重试策略、环境隔离）
   - 固定依赖版本（action 版本、base image 版本）
   - 优化 secret 管理和权限配置
   - 添加健康检查和告警机制

5. **实施路线图**：
   - 分阶段实施计划（第一批：quick win 缓存优化；第二批：并行化重构；第三批：深度优化）
   - 各阶段预期效果（时间节省、稳定性提升）
   - 回滚方案（每个优化步骤的回滚方法）

### 步骤 8：优化方案确认

确认方式：
- **标准模式和单轮确认模式均需确认**：向用户展示优化方案（含分析摘要、优化清单、预期效果、回滚方案），AskUserQuestion 确认后再执行。这是唯一在单轮确认模式下仍需用户确认的节点（因为修改流水线配置有风险，需要用户知情同意）
- **完全自主模式**：自动决策，不询问用户

## 阶段四：配置生成

> **注意**：此阶段仅在 `--fix` 模式下执行。无 `--fix` 标志时跳过此阶段，直接进入阶段五输出分析报告。

### 步骤 9：启动 writer

启动 writer，传入确认的优化方案和原始配置文件。

**Writer 生成优化后的配置文件**（严格按优化方案执行）：

1. **配置文件生成**：
   - 按优化方案逐项修改/重写 CI/CD 配置
   - 保留原始配置中与优化无关的部分（不做超出方案范围的改动）
   - 添加清晰的注释说明每处优化的目的
   - 遵循目标平台的最佳实践和语法规范

2. **辅助文件生成**（如需要）：
   - 缓存配置文件（`.cache-config` 等）
   - Makefile/Taskfile 优化（如构建步骤有变化）
   - Docker 构建优化（多阶段构建、layer 优化）
   - 环境变量模板更新

3. **变更说明文档**：
   - 逐项说明每处变更的内容、目的和预期效果
   - 标注需要用户额外操作的配置（如 secret 设置、runner 配置）
   - 提供回滚命令/步骤

### 步骤 10：Writer 自检

配置生成后，writer 执行自检清单：

1. **语法校验**：使用平台对应的校验工具检查配置语法
   - GitHub Actions: `actionlint`（如有）或 YAML 语法检查
   - GitLab CI: `gitlab-ci-lint`（如有）或 YAML 语法检查
   - Jenkins: Groovy 语法检查
   - CircleCI: `circleci config validate`（如有）
2. **结构完整性**：检查 job 依赖关系是否有环、artifact 引用是否正确
3. **Secret 引用**：确认 secret 引用未被破坏或泄露
4. **格式规范**：YAML 格式化、缩进一致性
5. **与原配置对比**：生成 diff，确认修改范围在方案范围内

自检全部通过后，通知 team lead。自检失败则自行修复，无法修复则升级 team lead。

## 阶段五：验证收尾

### 步骤 11：验证与报告

**如果是 `--fix` 模式**：

1. **语法验证**：运行平台配置校验工具，确认配置无语法错误
2. **Dry-run 验证**（如平台支持）：模拟执行流水线，确认逻辑正确
3. **Diff 审查**：生成原始配置 vs 优化配置的完整 diff
4. **回归检查**：确认原有的 CI 功能（测试、构建、部署）均未丢失

| 验证结果 | 处理方式 |
|---------|---------|
| **全部通过** | 进入步骤 12 总结 |
| **语法错误** | 打回 writer 修复语法问题 |
| **逻辑错误（依赖环、缺失引用）** | Writer 检查配置逻辑，修复依赖关系 |
| **功能回归（丢失 CI 步骤）** | Writer 回退相关修改，确保功能完整 |
| **安全问题（secret 泄露风险）** | Writer 立即修复安全配置 |

**验证失败迭代**：writer 修复后重新验证，循环直到通过。**最大迭代轮次：3 轮。** 超出后触发熔断，team lead 向用户汇报当前状态（已通过/未通过的验证项、失败原因分析），由用户决定继续、调整方向或终止。

**如果非 `--fix` 模式**：直接进入步骤 12 输出分析报告（不生成配置文件）。

## 核心原则

- **数据驱动**：分析基于实际构建日志和历史数据，不凭直觉
- **独立分析**：两位 analyzer 独立工作，避免锚定偏差，确保分析多样性
- **优先级导向**：优化方案按投入产出比排序，先解决 quick win
- **安全第一**：不引入 secret 泄露风险，不降低安全性
- **可回滚**：每个优化步骤都有明确的回滚方案
- **分歧即信号**：两路分析分歧度高意味着问题复杂度高，必须谨慎处理
- **有限迭代**：配置生成失败最多重试 3 轮，超出则暂停问用户
- **不确定就问**：标准模式下关键节点必须用户确认；单轮确认模式下优化方案仍需确认；完全自主模式下全部自动决策

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 无法识别 CI/CD 平台 | Scanner 提示 team lead，team lead 用 AskUserQuestion 向用户确认使用的 CI/CD 平台 |
| CI 配置文件语法已损坏 | Scanner 标记语法错误，analyzer 在分析中将其列为 Critical 问题 |
| 无构建历史数据 | 跳过历史分析部分，仅基于配置文件静态分析，在报告中标注数据局限性 |
| 两位 analyzer 分析严重分歧（>50%） | 触发熔断，暂停问用户。用户可提供额外信息或指定分析方向 |
| 生成的配置语法校验失败 | Writer 修复语法问题；3 轮未解决则暂停问用户 |
| 优化后丢失 CI 功能 | Writer 回退修改，确保功能完整后重新优化 |
| Secret 引用被破坏 | Writer 立即修复，确保 secret 配置正确 |
| 平台特性不支持某优化（如旧版本无 cache 功能） | Optimizer 标注平台限制，提供替代方案或跳过该优化 |
| 多平台配置（同时有 GitHub Actions + GitLab CI） | Scanner 全部收集，analyzer 分别分析，optimizer 分别优化 |
| Runner 资源限制（并行度上限、存储配额） | Optimizer 在方案中标注资源约束，不超出限制设计方案 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| Git 仓库状态异常（脏工作区） | Team lead 在阶段零检测工作区状态，提示用户先 stash 或提交变更 |
