**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--target-locales=en,zh,ja,ko`：目标语言列表（可选，逗号分隔，默认从项目已有 locale 文件推断）
- `--fix`：自动修复模式（可选，启用后 fixer 将自动提取硬编码字符串到 locale 文件）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从审计需求描述中移除。

**目标语言推断**：当未指定 `--target-locales` 时，scanner 从项目中检测已有 locale 配置（`i18n/`、`locales/`、`messages/`、`translations/` 目录，或框架配置文件），自动确定目标语言列表。若项目无任何 i18n 配置，默认 `en`。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 审计范围确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **字符串分类不确定（是否需要翻译）** → 对应 auditor 自行判断，报告中说明
- **两位 auditor 对同一文件的 i18n 问题评级差异过大（相差 2 级及以上）** → **不可跳过，必须暂停问用户**（熔断机制）
- **硬编码字符串超过 100 个** → **不可跳过，必须暂停问用户**（熔断机制）
- **locale 文件解析失败** → scanner 标注"该文件不可用"，不阻塞流程
- **发现大量第三方库内的硬编码字符串** → auditor 自行决定是否纳入（通常排除 node_modules 等）

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（评级差异过大、硬编码字符串超过 100 个时仍必须暂停问用户）。

使用 TeamCreate 创建 team（名称格式 `team-i18n-{YYYYMMDD-HHmmss}`，如 `team-i18n-20260308-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 流程概览

```
阶段零  项目扫描 + 语言检测 → 识别技术栈、i18n 框架、已有 locale 配置，确认审计目标
         ↓
阶段一  代码扫描 + 双路独立审计（并行）
         ├─ scanner：自动化 i18n 扫描（硬编码字符串检测、locale 文件解析、翻译键使用统计）
         ├─ auditor-1：字符串与翻译审计（硬编码字符串、翻译覆盖率、缺失键、废弃键、占位符一致性）
         └─ auditor-2：编码与渲染审计（字符编码、RTL 支持、文本膨胀/收缩、日期/数字/货币格式、复数规则）
         ↓
阶段二  合并分析 → fixer 合并发现，生成统一 i18n 评估 + 可选自动修复
         → 熔断检查（评级差异过大 / 硬编码字符串 > 100）
         ↓
阶段三  报告生成 → reporter 生成 i18n 合规报告 + 改进方案
         ↓
阶段四  保存报告 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 识别项目技术栈和 i18n 框架（react-intl、vue-i18n、i18next、gettext、Android strings.xml、iOS Localizable.strings 等），扫描所有源代码文件检测硬编码字符串，解析 locale 文件统计翻译键覆盖率，输出客观扫描数据。**仅在扫描阶段工作，完成后关闭。** |
| auditor-1 | **字符串与翻译维度审计**：检查硬编码用户可见字符串、翻译键覆盖率（每个 locale 的缺失键/多余键）、翻译质量（占位符一致性、变量格式、截断风险）、翻译键命名规范。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-2 交流。** |
| auditor-2 | **编码与渲染维度审计**：检查字符编码处理（UTF-8 一致性、BOM 处理）、RTL 布局支持、文本膨胀/收缩（德语膨胀约 30%、CJK 字符宽度）、日期/时间/数字/货币格式化、复数规则（CLDR）、排序/比较（Collation）。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-1 交流。** |
| fixer | 合并两份审计报告和 scanner 扫描结果，输出统一 i18n 评估清单。如启用 `--fix`，自动提取硬编码字符串到 locale 文件（生成翻译键、更新代码引用、补充缺失 locale 条目）。**完成后关闭。** |
| reporter | 基于统一 i18n 评估清单生成合规报告（含问题清单、覆盖率矩阵、改进方案、优先级排序），按 `--lang` 语言输出。**完成后关闭。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段一（步骤 3） | 阶段一扫描完成后（步骤 3） | 扫描报告交付后即释放 |
| auditor-1 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| auditor-2 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| fixer | 阶段二（步骤 6） | 阶段二合并/修复完成后（步骤 9） | 统一评估清单输出后释放 |
| reporter | 阶段三（步骤 10） | 阶段三报告生成后（步骤 12） | 报告确认后释放 |

## i18n 审计维度

双 auditor 按各自负责的维度独立审计，fixer 按权重合并为 i18n 评估总分。

| 审计维度 | 权重 | 负责人 | 审计内容 |
|---------|------|--------|---------|
| 硬编码字符串 | 30% | auditor-1 | 用户可见的未国际化字符串（UI 文本、错误消息、提示信息、日志中的用户可见部分） |
| 翻译覆盖率 | 20% | auditor-1 | 每个目标 locale 的翻译键覆盖率、缺失键、废弃键、占位符一致性 |
| 翻译键规范 | 10% | auditor-1 | 键命名一致性、层级结构、命名空间、重复键 |
| 字符编码 | 10% | auditor-2 | UTF-8 一致性、文件 BOM、数据库编码、API 响应编码 |
| RTL 支持 | 10% | auditor-2 | 双向文本处理、CSS 逻辑属性、布局镜像、图标方向 |
| 文本适应性 | 10% | auditor-2 | 文本膨胀/收缩空间、UI 布局弹性、截断处理、换行策略 |
| 格式化规则 | 10% | auditor-2 | 日期/时间格式、数字/货币格式、复数规则（CLDR）、排序规则 |

**auditor-1 总权重：60%**（硬编码字符串 30% + 翻译覆盖率 20% + 翻译键规范 10%）
**auditor-2 总权重：40%**（字符编码 10% + RTL 支持 10% + 文本适应性 10% + 格式化规则 10%）

## i18n 问题分级

| 级别 | 含义 | 示例 |
|------|------|------|
| **P0 - 阻断** | 功能无法在目标 locale 使用 | 字符编码错误导致乱码、RTL 布局完全错乱、关键流程缺失翻译 |
| **P1 - 严重** | 明显影响用户体验 | 大量硬编码字符串、翻译覆盖率 < 80%、日期格式不符合 locale 习惯 |
| **P2 - 中等** | 有问题但不影响功能 | 个别缺失翻译、占位符不一致、文本截断 |
| **P3 - 轻微** | 优化项 | 翻译键命名不规范、废弃翻译键未清理、排序规则未本地化 |

## 阶段零：项目扫描 + 语言检测

### 步骤 1：解析审计范围

Team lead 分析项目和参数：

1. 解析 `--target-locales` 参数，确定目标语言列表
2. 阅读项目结构，识别技术栈（语言、UI 框架、i18n 库）
3. 识别 i18n 相关配置和文件：
   - i18n 框架配置（react-intl、vue-i18n、i18next、ngx-translate、gettext 等）
   - locale 文件目录（`locales/`、`i18n/`、`messages/`、`translations/`、`lang/`）
   - locale 文件格式（JSON、YAML、PO/POT、XLIFF、ARB、strings.xml 等）
   - 已支持的语言列表
   - 翻译管理平台集成（Crowdin、Lokalise、Phrase、Transifex）
4. 输出审计范围清单（含已有 locale、目标 locale、i18n 框架信息）

### 步骤 2：用户确认审计范围

**标准模式**：向用户展示审计范围清单，AskUserQuestion 确认：
- 确认目标语言列表和审计范围
- 调整审计维度（增删特定检查项）
- 补充特殊 i18n 要求（如特定 locale 优先级）

**单轮确认模式**：跳过确认，直接进入阶段一。
**完全自主模式**：自动决策，不询问用户，直接进入阶段一。

## 阶段一：代码扫描 + 双路 i18n 审计

### 步骤 3：启动 scanner 和双 auditor

三者并行启动。

**Scanner 自动化 i18n 扫描**：
1. 扫描所有源代码文件，检测硬编码用户可见字符串：
   - 排除范围：测试文件、注释、日志（非用户可见）、常量定义（技术值如 HTTP 方法）、node_modules/vendor
   - 检测范围：JSX/TSX 文本节点、模板字面量、HTML 文本、Android XML、iOS Storyboard
2. 解析所有 locale 文件，统计翻译键：
   - 每个 locale 的键数量
   - 缺失键（某 locale 有而其他 locale 没有）
   - 废弃键（代码中未引用的翻译键）
3. 检测 i18n 框架使用模式：
   - 翻译函数调用统计（`t()`、`$t()`、`intl.formatMessage()`、`gettext()` 等）
   - 动态键使用（`t(variable)` 无法静态分析的情况）
4. 输出**i18n 扫描报告**
5. Scanner 完成后关闭

**双 auditor 同时阅读项目**：
- 阅读项目结构（UI 组件、页面、布局、样式）
- 理解 i18n 框架使用方式和翻译流程
- 梳理各自负责维度的审计要点
- 各自输出项目 i18n 概况给 team lead

### 步骤 4：分发 scanner 报告

Team lead 收到 scanner 报告后，将扫描结果分发给两位 auditor，作为审计的参考数据。

Auditor 在审计时必须参考 scanner 结果——例如 scanner 检测到的硬编码字符串列表是 auditor-1 审计的基础数据。

### 步骤 5：独立并行 i18n 审计

两位 auditor 各自审计负责的维度，**互不交流**。

**Auditor-1（字符串与翻译维度）**：

逐维度输出"合规状态 + 发现 + 证据"：

| 维度 | 审计内容 | 具体检查项 |
|------|---------|-----------|
| **硬编码字符串** | 用户可见的未国际化字符串 | UI 文本（按钮、标签、标题、提示）、错误消息（用户可见的 validation/error）、占位符文本（placeholder、tooltip）、确认对话框文本、邮件/通知模板、PDF/导出文本 |
| **翻译覆盖率** | 每个 locale 的翻译完整性 | 缺失键清单（按 locale 分组）、多余键（已不在代码中引用）、占位符一致性（`{name}` 在所有 locale 中存在）、变量格式一致性（ICU MessageFormat 语法）、翻译为空字符串的键 |
| **翻译键规范** | 键命名和组织 | 命名一致性（驼峰/蛇形/点分隔）、层级结构合理性、命名空间划分、重复键检测、键名是否有意义（避免 `label1`、`text2`） |

- 参考 scanner 的硬编码字符串列表，逐一确认并补充遗漏
- 记录每个发现：维度 + 文件路径 + 行号 + 级别（P0-P3） + 描述 + 建议修复

**Auditor-2（编码与渲染维度）**：

逐维度输出"合规状态 + 发现 + 证据"：

| 维度 | 审计内容 | 具体检查项 |
|------|---------|-----------|
| **字符编码** | 编码一致性和正确性 | 源文件编码（UTF-8 without BOM）、HTML meta charset、HTTP Content-Type header、数据库连接编码设置、API 请求/响应编码、文件读写编码指定 |
| **RTL 支持** | 从右到左语言支持 | HTML `dir` 属性设置、CSS 逻辑属性（`margin-inline-start` vs `margin-left`）、Flexbox/Grid 方向处理、图标和图片方向、文本对齐、双向文本（BiDi）混合处理 |
| **文本适应性** | 不同语言长度和显示 | 德语/芬兰语文本膨胀（约 30-50% 更长）、CJK 字符宽度（等宽 vs 比例）、UI 容器弹性（固定宽度 vs 自适应）、文本截断策略（ellipsis 处理）、换行规则（CJK 无空格换行） |
| **格式化规则** | 区域相关的格式化 | 日期格式（`MM/DD/YYYY` vs `DD.MM.YYYY` vs `YYYY年MM月DD日`）、数字格式（千位分隔符：逗号/点/空格）、货币格式（符号位置/小数位）、复数规则（CLDR categories: zero/one/two/few/many/other）、排序规则（locale-aware collation） |

- 如果目标 locale 不含 RTL 语言（如阿拉伯语、希伯来语），RTL 审计可标注"不适用"
- 记录每个发现：维度 + 文件路径 + 行号 + 级别（P0-P3） + 描述 + 建议修复

## 阶段二：合并分析 + 可选修复

### 步骤 6：启动 fixer

两位 auditor 审计完成后，启动 fixer。Team lead 将以下材料发送给 fixer：
- Scanner i18n 扫描报告
- Auditor-1 审计报告
- Auditor-2 审计报告
- `--fix` 标志状态

### 步骤 7：合并去重与评估

Fixer 执行合并分析：

1. **去重**：识别 scanner 结果与 auditor 手工发现的重叠项，合并为单条记录
2. **交叉校验**：检查两位 auditor 是否对同一文件有不同问题级别评定
3. **统一评级**：按 i18n 问题分级标准统一评级（P0-P3）
4. **覆盖率计算**：计算每个目标 locale 的翻译覆盖率
5. **影响评估**：评估每个问题对各目标 locale 的实际影响

**共识度计算公式**：
```
两位 auditor 审计的维度不同，但部分检查点可能交叉（如 auditor-1 的翻译覆盖率与 auditor-2 的格式化规则中的占位符）。
对于重叠区域的发现：

共识发现数 = 两位 auditor 都识别出的问题数量（指向相同文件/行号且描述相同问题类型）
总重叠区域发现数 = 重叠区域去重后的问题总数

共识度 = 共识发现数 / 总重叠区域发现数 × 100%（如无重叠区域发现则为 100%）

评级一致性 = 两位 auditor 对同一问题评级完全一致的比例
争议率 = 评级差异 ≥ 2 级的问题数 / 两位 auditor 都识别的问题数 × 100%

争议率 > 50% → 触发熔断，必须暂停问用户
```

### 步骤 8：差异校验与熔断检查

**差异校验**：
- 如果两位 auditor 对同一问题的级别差异 ≥ 2 级（如一方定 P0、另一方定 P2），fixer 标注为"争议项"
- 争议项占比 > 50% → **不可跳过，必须暂停问用户**（熔断机制）

**硬编码字符串熔断**：
- 统一评估清单中硬编码字符串超过 100 个 → **不可跳过，必须暂停问用户**（熔断机制）
- 向用户展示硬编码字符串统计和示例，确认是否继续（及是否启用 `--fix`）

**无熔断触发**时，fixer 输出**统一 i18n 评估清单**：

```
## 统一 i18n 评估清单

### 总览统计
- 硬编码字符串: X 个（P0: X / P1: X / P2: X / P3: X）
- 翻译覆盖率: en XX% / zh XX% / ja XX% / ko XX%
- 缺失翻译键: X 个 | 废弃翻译键: X 个
- 编码问题: X 个 | RTL 问题: X 个 | 格式化问题: X 个

### 维度评估

#### 硬编码字符串 — 评分：X/10
  ├─ [P0] src/components/Login.tsx:42 — "登录失败" 未国际化
  ├─ [P1] src/pages/Dashboard.tsx:88 — "欢迎回来" 硬编码
  └─ ...

#### 翻译覆盖率 — 评分：X/10
  ├─ [P1] locale/ja.json — 缺失 15 个键（覆盖率 85%）
  └─ ...

#### 编码与渲染 — 评分：X/10
  └─ ...

### 争议项（如有）
```

### 步骤 8.5：可选自动修复（`--fix` 模式）

如启用 `--fix`，fixer 执行自动修复：

1. **字符串提取**：将硬编码字符串提取为翻译键
   - 生成有意义的键名（基于上下文，如 `login.error.failed`）
   - 将原始字符串作为默认 locale 的翻译值
   - 替换源代码中的硬编码字符串为翻译函数调用
2. **缺失键补充**：在目标 locale 文件中补充缺失键（值留空或标注 `[NEEDS_TRANSLATION]`）
3. **废弃键清理**：移除代码中未引用的翻译键（生成独立 commit 或标注供人工确认）

每次修复生成独立的代码变更记录，供用户审查。

Fixer 完成后关闭。

### 步骤 9：i18n 评估评分

Team lead 根据统一 i18n 评估清单计算各维度评分：

评分规则：
- 基础分 10.0，按发现扣分
- P0 发现：每个扣 2.0 分（该维度最多扣至 0 分）
- P1 发现：每个扣 1.0 分
- P2 发现：每个扣 0.5 分
- P3 发现：每个扣 0.2 分

按权重计算总分，输出各维度和总体 i18n 评分。

## 核心原则

- **双路独立**：两位 auditor 从不同维度独立审计，互不交流，确保覆盖字符串和渲染两个层面
- **用户可见优先**：硬编码字符串检测聚焦用户可见文本，排除内部日志和技术常量
- **覆盖率驱动**：以翻译覆盖率为核心指标，量化每个 locale 的国际化程度
- **可操作性**：每个问题必须给出具体修复建议（文件、行号、修复方式），`--fix` 模式直接生成修复代码
- **locale 感知**：审计结果按目标 locale 分组，不同语言可能有不同的问题（如仅 RTL 语言才有 RTL 问题）
- **渐进式**：从 P0 阻断问题开始，逐步解决到 P3 优化项，不强求一次全覆盖
- **最小侵入**：`--fix` 模式生成的修复代码保持项目现有 i18n 风格和命名规范

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 项目无任何 i18n 配置 | Scanner 标注"项目未配置 i18n 框架"，按硬编码字符串审计为主，报告中建议 i18n 框架选型 |
| locale 文件格式无法解析 | Scanner 标注"该文件格式不支持"，不阻塞流程，auditor 基于代码审计 |
| 源文件数量过大（> 5000 文件） | Scanner 按模块分批扫描；auditor 聚焦 UI 相关模块，标注未覆盖区域 |
| 两位 auditor 评级差异过大（争议率 > 50%） | 触发熔断，暂停问用户裁决争议项 |
| 硬编码字符串超过 100 个 | 触发熔断，暂停向用户确认是否启用 --fix 自动修复 |
| 目标 locale 无对应 CLDR 数据 | Auditor-2 标注"该 locale 无 CLDR 覆盖"，复数/格式化规则审计标注"不适用" |
| --fix 模式修改导致测试失败 | Fixer 回滚该修改，标注"自动修复失败，需人工处理" |
| 翻译管理平台配置不可访问 | Scanner 标注"无法连接翻译平台"，仅基于本地 locale 文件审计 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前步骤恢复 |
| 混合 i18n 方案（多个框架共存） | Scanner 分别扫描各框架的 locale 配置，auditor 按框架分组审计 |
