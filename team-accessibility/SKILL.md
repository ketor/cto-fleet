**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--level=A|AA|AAA`：WCAG 合规等级（可选，默认 `AA`）
- `--fix`：自动修复模式（自动生成并应用修复代码）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从审计需求描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 审计范围确认 + 修复方案确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **WCAG 条款适用性不确定** → 对应 auditor 自行判断，报告中说明
- **两位 auditor 对同一问题的严重程度定级差异过大（相差 2 级及以上）** → **不可跳过，必须暂停问用户**（熔断机制）
- **Critical 无障碍问题超过 10 个** → **不可跳过，必须暂停问用户**（熔断机制）
- **扫描工具运行失败** → scanner 标注"该工具不可用"，不阻塞流程
- **组件使用了自定义 UI 框架** → auditor 自行基于 ARIA 规范审计

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（定级差异过大、Critical 问题超过 10 个时仍必须暂停问用户）。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

**文件命名规范**：

| 角色输出 | 文件名 |
|---------|--------|
| Scanner 报告 | `scanner-report.md` |
| Reviewer-N 第R轮 | `reviewer-{N}-round-{R}.md` |

| 根因分组 | `root-cause-groups-round-{R}.md` |
| Fixer 第R轮 | `fixer-round-{R}.md` |

| 最终报告 | `final-report.md` |

> 仅当角色存在于当前 skill 时使用对应命名。未列出的角色用 `{role}-{context}.md` 格式。

## 流程概览

```
阶段零  审计范围确定 → 解析 level，识别技术栈和页面清单，确认审计目标
         ↓
阶段一  自动化扫描 + 双路独立审计（并行）
         ├─ scanner：axe-core/Lighthouse/WAVE 自动化扫描
         ├─ auditor-1：视觉无障碍审计（色彩对比、文本可读性、视觉层次、媒体替代文本）
         └─ auditor-2：交互无障碍审计（键盘导航、屏幕阅读器、焦点管理、ARIA 属性）
         ↓
阶段二  合并分析 → fixer 合并去重、分组定级、差异校验、输出统一问题清单
         → 熔断检查（Critical > 10 / 审计差异过大）
         → 修复实施（如启用 --fix）
         ↓
阶段三  报告生成 → reporter 生成结构化 WCAG 合规报告（含修复建议、合规矩阵）
         ↓
阶段四  收尾 → 保存报告 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 识别项目前端技术栈和页面清单，运行自动化无障碍扫描（axe-core/Lighthouse/WAVE/pa11y），输出客观问题数据。**仅在扫描阶段工作，完成后关闭。** |
| auditor-1 | **视觉无障碍维度审计**：色彩对比度、文本可读性、视觉层次、图片替代文本、视频字幕、动画安全。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-2 交流。** |
| auditor-2 | **交互无障碍维度审计**：键盘导航、屏幕阅读器兼容性、焦点管理、ARIA 属性、表单标签、错误提示。输出结构化审计发现。**不修改代码。独立审计，不与 auditor-1 交流。** |
| fixer | 合并两份审计报告和 scanner 扫描结果，去重、分组、交叉校验、按 WCAG 条款对齐定级，输出统一问题清单。如启用 `--fix`，生成并应用修复代码。**完成后关闭。** |
| reporter | 基于统一问题清单生成 WCAG 合规报告（含修复建议、合规矩阵、影响评估），按 `--lang` 语言输出。**完成后关闭。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段一（步骤 3） | 阶段一扫描完成后（步骤 3） | 扫描报告交付后即释放 |
| auditor-1 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| auditor-2 | 阶段一（步骤 3，与 scanner 并行） | 阶段一审计完成后（步骤 5） | 审计报告交付后释放 |
| fixer | 阶段二（步骤 6） | 阶段二合并/修复完成后（步骤 9） | 统一问题清单和修复输出后释放 |
| reporter | 阶段三（步骤 10） | 阶段三报告生成后（步骤 12） | 报告确认后释放 |

## 问题严重程度定级（对齐 WCAG）

每个问题按严重程度定级：

| 级别 | 含义 | WCAG 对应 |
|------|------|----------|
| **Critical** | 完全阻断特定用户群体的访问，无替代方案 | Level A 违规（核心功能不可用） |
| **High** | 严重影响用户体验，需额外努力才能完成任务 | Level A 违规（非核心功能） |
| **Medium** | 影响部分用户效率，但有替代方案 | Level AA 违规 |
| **Low** | 轻微不便，不影响核心功能使用 | Level AAA 违规或最佳实践 |
| **Info** | 增强建议，提升无障碍体验 | 超出 WCAG 的最佳实践 |

## Scanner 自动化扫描工具

| 扫描类别 | 常用工具 | 输出 |
|---------|---------|------|
| 静态分析 | axe-core / axe-linter / eslint-plugin-jsx-a11y | 代码级无障碍问题 |
| 运行时分析 | Lighthouse Accessibility / WAVE | 页面级无障碍评分和问题 |
| 命令行扫描 | pa11y / pa11y-ci | 批量页面扫描结果 |
| 色彩对比 | color-contrast-checker / axe color rules | 色彩对比度不达标项 |
| HTML 验证 | html-validate / Nu HTML Checker | 语义化 HTML 问题 |

**如果某工具未安装或无法运行**：scanner 标注"该工具不可用"，不阻塞流程。

## 审计维度分配

双 auditor 按各自负责的维度独立审计，fixer 按权重合并为无障碍评估总分。

| 维度 | 权重 | 负责人 | 审计内容 |
|------|------|--------|---------|
| 色彩与对比度 | 15% | auditor-1 | 文本对比度（≥4.5:1 正文 / ≥3:1 大文本）、非文本元素对比度（≥3:1）、不仅依赖颜色传达信息 |
| 文本可读性 | 10% | auditor-1 | 字体大小可缩放（至 200%）、行高/字间距可调、文本不被截断、语言标识正确 |
| 图片与媒体 | 15% | auditor-1 | 图片 alt 文本（描述性/装饰性区分）、视频字幕/音频描述、SVG 可访问性、背景图片信息 |
| 动画与运动 | 5% | auditor-1 | prefers-reduced-motion 支持、自动播放控制、闪烁频率限制（<3Hz）、视差滚动 |
| 视觉布局 | 10% | auditor-1 | 响应式布局（320px 最小宽度）、内容重排不丢失信息、焦点可见指示器、间距合理 |
| 键盘导航 | 15% | auditor-2 | Tab 顺序逻辑性、所有交互元素可键盘操作、无键盘陷阱、快捷键不冲突 |
| ARIA 属性 | 15% | auditor-2 | ARIA role/state/property 正确使用、动态内容 live region、自定义组件 ARIA 模式 |
| 表单与错误 | 10% | auditor-2 | 表单 label 关联、必填字段标识、错误提示关联和描述性、表单验证无障碍 |
| 屏幕阅读器 | 5% | auditor-2 | 标题层级结构（h1-h6）、地标区域（landmark）、阅读顺序、跳过导航链接 |

**auditor-1 总权重：55%**（色彩与对比度 15% + 文本可读性 10% + 图片与媒体 15% + 动画与运动 5% + 视觉布局 10%）
**auditor-2 总权重：45%**（键盘导航 15% + ARIA 属性 15% + 表单与错误 10% + 屏幕阅读器 5%）

**`--level` 参数处理**：
- `A`：仅审计 WCAG 2.1 Level A 条款
- `AA`（默认）：审计 Level A + AA 条款
- `AAA`：审计 Level A + AA + AAA 条款

## 阶段零：审计范围确定

### 步骤 1：解析审计范围

Team lead 分析项目和参数：

1. 解析 `--level` 参数，确定审计等级
2. 阅读项目结构，识别前端技术栈（React/Vue/Angular/Svelte/原生 HTML）
3. 识别审计边界：
   - 页面/路由清单（公共页面、认证后页面、模态框、弹窗）
   - UI 组件库（Material UI、Ant Design、自定义组件）
   - 交互模式（SPA/MPA、SSR/CSR、PWA）
   - 多媒体内容（图片、视频、音频、SVG、Canvas）
   - 国际化（多语言支持、RTL 布局）
4. 输出审计范围清单

### 步骤 2：用户确认审计范围

**标准模式**：向用户展示审计范围清单，AskUserQuestion 确认：
- 确认审计范围
- 调整审计范围（增删页面/组件）
- 补充无障碍关注点（如特定残障用户群体）

**单轮确认模式**：跳过确认，直接进入阶段一。
**完全自主模式**：自动决策，不询问用户，直接进入阶段一。

## 阶段一：自动化扫描 + 双路独立审计

### 步骤 3：启动 scanner 和双 auditor

三者并行启动。

**Scanner 自动化扫描**：
1. 阅读项目结构，识别前端框架和构建工具
2. 识别项目中已配置的无障碍工具（.axerc、lighthouse 配置等）
3. 按审计范围运行对应扫描工具，收集客观数据：

| 扫描范围 | 执行的工具类别 | 输出内容 |
|---------|-------------|---------|
| 全页面扫描 | axe-core + Lighthouse | 页面级无障碍评分和问题 |
| 组件级扫描 | eslint-plugin-jsx-a11y / vue-a11y | 组件代码级问题 |
| 色彩扫描 | axe color rules | 对比度不达标项 |
| HTML 语义 | html-validate | 语义化 HTML 问题 |
| 批量扫描 | pa11y-ci | 多页面批量结果 |

4. 汇总输出**自动化扫描报告**发送给 team lead
5. Scanner 完成后关闭（不参与后续阶段）

**双 auditor 同时阅读项目**：
- 阅读项目结构（目录布局、组件文件、样式文件）
- 理解前端框架、组件库和样式方案
- 梳理用户交互流程（核心用户旅程）
- 各自输出项目无障碍概况给 team lead

### 步骤 4：分发 scanner 报告

Team lead 收到 scanner 报告后，将自动化扫描结果分发给两位 auditor，作为手工审计的参考数据。

Auditor 在审计时必须参考 scanner 结果——例如 scanner 发现某页面 Lighthouse 无障碍评分仅 40 分，auditor 应重点审计该页面。

### 步骤 5：独立并行手工审计

两位 auditor 各自审计负责的维度，**互不交流**。

**Auditor-1（视觉无障碍维度）** — 对应 WCAG 2.1 感知原则（Perceivable）：

审计时必须逐项检查以下 WCAG 条款，每个条款输出"已检查 + 合规/不合规"：

| WCAG 条款 | 审计内容 | 具体检查项 |
|-----------|---------|-----------|
| **1.1.1 Non-text Content** | 非文本内容替代 | img alt 属性（描述性 vs 装饰性 `alt=""`）、icon 的 aria-label、SVG title/desc、CSS 背景图片信息替代、Canvas 替代文本 |
| **1.3.1 Info and Relationships** | 信息和关系 | 标题层级正确（h1→h2→h3 不跳级）、表格 th/scope、列表语义（ul/ol/dl）、表单 label 关联、地标区域 |
| **1.3.2 Meaningful Sequence** | 有意义的序列 | DOM 顺序与视觉顺序一致、CSS flexbox/grid order 不影响阅读顺序 |
| **1.4.1 Use of Color** | 颜色的使用 | 不仅依赖颜色传达信息（错误状态需有图标/文本）、链接需有下划线或其他非颜色区分 |
| **1.4.3 Contrast (Minimum)** | 对比度（最低） | 正文文本 ≥4.5:1、大文本（≥18pt/14pt bold）≥3:1 |
| **1.4.4 Resize Text** | 文本缩放 | 页面在 200% 缩放下无内容丢失/重叠 |
| **1.4.5 Images of Text** | 文本图片 | 避免使用图片代替文本（logo 除外） |
| **1.4.10 Reflow** | 内容重排 | 320px 宽度下内容不需要水平滚动 |
| **1.4.11 Non-text Contrast** | 非文本对比度 | UI 组件和图形元素对比度 ≥3:1 |
| **1.4.12 Text Spacing** | 文本间距 | 行高 1.5x、段间距 2x、字间距 0.12em、词间距 0.16em 下无内容丢失 |
| **2.3.1 Three Flashes** | 闪烁限制 | 无每秒闪烁超过 3 次的内容 |

- 审计视频/音频内容：字幕（1.2.2）、音频描述（1.2.5）、手语（1.2.6，AAA）
- 参考 scanner 的色彩对比度扫描结果，验证自动化发现的准确性
- 记录每个发现：文件路径 + 行号/选择器 + 严重程度 + WCAG 条款 + 问题描述 + 修复建议

**Auditor-2（交互无障碍维度）** — 对应 WCAG 2.1 可操作/可理解/健壮原则：

审计时必须逐项检查以下 WCAG 条款，每个条款输出"已检查 + 合规/不合规"：

| WCAG 条款 | 审计内容 | 具体检查项 |
|-----------|---------|-----------|
| **2.1.1 Keyboard** | 键盘可操作 | 所有交互元素可 Tab 到达、可 Enter/Space 激活、自定义组件键盘支持 |
| **2.1.2 No Keyboard Trap** | 无键盘陷阱 | 模态框可 Esc 关闭、焦点不会被困在组件内、下拉菜单可 Esc 收起 |
| **2.4.1 Bypass Blocks** | 跳过块 | 提供"跳到主内容"链接、地标区域标记 |
| **2.4.3 Focus Order** | 焦点顺序 | Tab 顺序与视觉顺序一致、模态框焦点陷阱（焦点留在模态框内）、关闭后焦点返回触发元素 |
| **2.4.6 Headings and Labels** | 标题和标签 | 标题描述性、表单 label 描述性、链接文本有意义（非"点击这里"） |
| **2.4.7 Focus Visible** | 焦点可见 | 所有可交互元素有可见焦点指示器、焦点样式对比度足够 |
| **2.5.1 Pointer Gestures** | 指针手势 | 多点/路径手势有单点替代方案 |
| **3.1.1 Language of Page** | 页面语言 | html lang 属性正确设置 |
| **3.1.2 Language of Parts** | 部分语言 | 多语言内容正确标记 lang 属性 |
| **3.2.1 On Focus** | 聚焦时 | 聚焦不触发上下文变化（页面跳转/弹窗） |
| **3.3.1 Error Identification** | 错误识别 | 表单错误以文本描述、错误关联到对应字段（aria-describedby） |
| **3.3.2 Labels or Instructions** | 标签或说明 | 输入字段有清晰标签、复杂表单有说明 |
| **4.1.1 Parsing** | 解析 | HTML 无重复 ID、标签正确嵌套 |
| **4.1.2 Name, Role, Value** | 名称、角色、值 | 自定义组件有正确 ARIA role、aria-label/labelledby、状态通过 aria-expanded/checked/selected 传达 |
| **4.1.3 Status Messages** | 状态消息 | Toast/通知使用 aria-live region、加载状态通过 aria-busy 传达 |

- 审计 ARIA 使用模式：widget role 使用正确性、composite widget 键盘模式（combobox/listbox/tree/grid）
- 参考 scanner 的 axe-core 结果，验证自动化发现的准确性
- 记录每个发现：文件路径 + 行号/选择器 + 严重程度 + WCAG 条款 + 问题描述 + 修复建议

## 阶段二：合并分析与修复

### 步骤 6：启动 fixer

两位 auditor 审计完成后，启动 fixer。Team lead 将以下材料发送给 fixer：
- Scanner 自动化扫描报告
- Auditor-1 审计报告
- Auditor-2 审计报告
- 项目前端技术栈信息

### 步骤 7：合并去重与分组

Fixer 执行合并分析：

1. **去重**：识别 scanner 结果与 auditor 手工发现的重叠项，合并为单条记录
2. **交叉校验**：检查两位 auditor 是否对同一问题有不同定级
3. **分组**：按 WCAG 原则分组（感知/可操作/可理解/健壮）
4. **定级**：按 WCAG 条款级别统一定级，输出每个问题的严重程度
5. **影响评估**：评估每个问题对不同残障用户群体的影响

**共识度计算公式**：
```
两位 auditor 审计有部分维度交叉（如 auditor-1 的视觉布局中的焦点指示器 与 auditor-2 的焦点可见可能重叠）。
对于重叠区域的发现：

共识发现数 = 两位 auditor 都识别出的问题数量（指向相同文件/选择器且描述相同问题类型）
总重叠区域发现数 = 重叠区域去重后的问题总数

共识度 = 共识发现数 / 总重叠区域发现数 × 100%（如无重叠区域发现则为 100%）

定级一致性 = 两位 auditor 对同一问题定级完全一致的比例
争议率 = 定级差异 ≥ 2 级的问题数 / 两位 auditor 都识别的问题数 × 100%

争议率 > 50% → 触发熔断，必须暂停问用户
```

### 步骤 8：差异校验与熔断检查

**差异校验**：
- 如果两位 auditor 对同一问题的定级差异 ≥ 2 级（如一方定 Critical、另一方定 Medium），fixer 标注为"争议项"
- 争议项占比 > 50% → **不可跳过，必须暂停问用户**（熔断机制）

**Critical 问题熔断**：
- 统一问题清单中 Critical 问题超过 10 个 → **不可跳过，必须暂停问用户**（熔断机制）
- 向用户展示 Critical 问题列表，确认是否继续

**无熔断触发**时，fixer 输出**统一问题清单**：

```
## 统一无障碍问题清单

### 问题统计
- Critical: X 个 | High: X 个 | Medium: X 个 | Low: X 个 | Info: X 个
- 总计: X 个问题

### 问题分组

#### 组 1：[WCAG 原则/条款] — 最高级别：Critical
  ├─ [Critical] 文件:行号 - 问题描述 - WCAG 条款 - 来源：scanner/auditor-1/auditor-2
  ├─ [High] 文件:行号 - 问题描述 - WCAG 条款 - 来源：scanner/auditor-1/auditor-2
  └─ 影响评估：[受影响用户群体]

#### 组 2：[WCAG 原则/条款] — 最高级别：High
  └─ ...

### 争议项（如有）
1. 文件:行号 - auditor-1 定级: X / auditor-2 定级: Y - fixer 最终定级: Z - 理由
```

### 步骤 8.5：自动修复（如启用 --fix）

如果用户指定了 `--fix`，fixer 对可自动修复的问题生成并应用修复代码：

| 可自动修复类别 | 修复方式 |
|--------------|---------|
| 缺少 alt 属性 | 添加描述性 alt 或 `alt=""` |
| 缺少 aria-label | 添加 aria-label/aria-labelledby |
| 缺少 lang 属性 | 添加 html lang 属性 |
| 对比度不足（CSS） | 调整颜色值达标 |
| 缺少表单 label | 添加 label 关联或 aria-label |
| 缺少跳过导航链接 | 添加 skip-to-main-content 链接 |
| 缺少 landmark | 添加语义化 HTML5 标签（header/nav/main/footer） |
| 缺少 aria-live | 添加 aria-live region |
| 焦点样式缺失 | 添加 :focus-visible 样式 |

不可自动修复的问题（需人工判断）标注为"建议手动修复"。

Fixer 完成后关闭。

### 步骤 9：无障碍评估评分

Team lead 根据统一问题清单计算各维度无障碍评分：

评分规则：
- 基础分 10.0，按问题扣分
- Critical 问题：每个扣 1.5 分（该维度最多扣至 0 分）
- High 问题：每个扣 1.0 分
- Medium 问题：每个扣 0.5 分
- Low 问题：每个扣 0.2 分
- Info：不扣分

按权重计算总分，输出各维度和总体无障碍评分。

## 核心原则

- **工具先行**：先用自动化工具收集客观问题数据，再人工审计需要理解上下文的问题
- **双路独立**：两位 auditor 从不同维度独立审计，互不交流，避免盲区和偏见
- **WCAG 对齐**：所有发现对齐 WCAG 2.1 条款，确保合规性评估的权威性
- **用户视角**：审计时从残障用户角度体验，不只看代码实现是否符合条款
- **渐进增强**：优先确保 Level A 合规，再追求 AA 和 AAA
- **可修复性**：修复建议必须具体到代码级别，可自动修复的尽量自动修复
- **持续合规**：建议将无障碍检查集成到 CI/CD，避免回归
- **包容设计**：关注所有残障用户群体，不遗漏小众需求

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 扫描工具未安装/无法运行 | Scanner 标注"该工具不可用"，不阻塞流程，auditor 基于手工审计评估 |
| 项目无法构建/运行 | 仅做静态审计（代码级 ARIA/HTML 检查），报告中标注"未验证运行时行为" |
| 页面数量过多（>100 个路由） | Scanner 按模块分批扫描；auditor 聚焦核心用户旅程页面，标注未覆盖页面 |
| 两位 auditor 定级差异过大（争议项 > 50%） | 触发熔断，暂停问用户裁决争议项 |
| Critical 问题超过 10 个 | 触发熔断，暂停向用户确认是否继续完整报告 |
| 使用 Canvas/WebGL 渲染 | Auditor 标注"Canvas 内容无法自动审计"，建议提供替代文本 |
| 自定义 UI 框架（非标准 HTML） | Auditor 基于 ARIA 规范进行通用审计，标注框架特殊性 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前步骤恢复 |
| --fix 修复导致视觉回归 | Fixer 标注"需人工验证视觉效果"，不强制应用 |
