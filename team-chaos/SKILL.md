**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--target=service|network|database|infra`：实验目标类型（可选，默认全部类型）
- `--dry-run`：仅生成实验方案，不执行任何故障注入操作
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从混沌工程需求描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 拓扑确认 + 实验方案确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **拓扑识别不确定** → scanner 自行补充，报告中说明
- **两位 designer 对同一故障场景的爆炸半径评估差异过大（相差 2 级及以上）** → **不可跳过，必须暂停问用户**（熔断机制）
- **识别到生产环境关键路径且无降级方案** → **不可跳过，必须暂停问用户**（熔断机制）
- **依赖服务不可达或拓扑信息缺失** → scanner 标注"该服务拓扑不完整"，不阻塞流程
- **实验范围超出预期（发现大量微服务依赖）** → analyst 自行决定是否纳入

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（爆炸半径评估差异过大、关键路径无降级方案时仍必须暂停问用户）。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

## 流程概览

```
阶段零  系统拓扑识别 → 解析 target，识别系统架构和依赖关系，确认实验边界
         ↓
阶段一  拓扑扫描 + 双路实验设计（并行）
         ├─ scanner：系统拓扑扫描、依赖链分析、关键路径识别
         ├─ designer-1：故障注入实验设计（服务宕机、进程崩溃、资源耗尽）
         └─ designer-2：负载异常实验设计（流量激增、网络延迟、数据异常）
         ↓
阶段二  合并分析 → analyst 合并去重、爆炸半径评估、韧性评分、输出统一实验方案
         → 熔断检查（爆炸半径差异过大 / 关键路径无降级方案）
         ↓
阶段三  实验执行/方案输出 → executor 按方案执行实验或输出 dry-run 方案
         ↓
阶段四  收尾 → 保存报告 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| scanner | 识别系统架构和拓扑关系，扫描服务依赖链、网络拓扑、数据库连接、消息队列关系，输出系统拓扑图和关键路径。**仅在扫描阶段工作，完成后关闭。** |
| designer-1 | **故障注入维度设计**：服务宕机、进程崩溃、资源耗尽（CPU/内存/磁盘）、容器驱逐、Pod 重启。输出结构化实验方案。**不执行实验。独立设计，不与 designer-2 交流。** |
| designer-2 | **负载异常维度设计**：流量激增、网络延迟/丢包/分区、DNS 故障、数据库慢查询注入、消息队列积压。输出结构化实验方案。**不执行实验。独立设计，不与 designer-1 交流。** |
| analyst | 合并两份实验设计和 scanner 拓扑数据，去重、评估爆炸半径、交叉校验、按影响级别排序，输出统一实验方案。**不直接阅读代码，只对比和合并方案。完成后关闭。** |
| executor | 基于统一实验方案执行混沌实验或输出 dry-run 方案（含实验脚本、回滚步骤、监控指标），按 `--lang` 语言输出。**完成后关闭。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| scanner | 阶段一（步骤 3） | 阶段一扫描完成后（步骤 3） | 拓扑报告交付后即释放 |
| designer-1 | 阶段一（步骤 3，与 scanner 并行） | 阶段一设计完成后（步骤 5） | 实验方案交付后释放 |
| designer-2 | 阶段一（步骤 3，与 scanner 并行） | 阶段一设计完成后（步骤 5） | 实验方案交付后释放 |
| analyst | 阶段二（步骤 6） | 阶段二合并完成后（步骤 9） | 统一实验方案输出后释放 |
| executor | 阶段三（步骤 10） | 阶段三执行/输出后（步骤 12） | 实验执行或方案确认后释放 |

## 爆炸半径等级定义

每个实验场景按爆炸半径定级：

| 级别 | 含义 | 影响范围 |
|------|------|---------|
| **Critical** | 全局级联故障，所有用户受影响，数据可能丢失 | 核心数据库宕机、全局网络分区、认证服务不可用 |
| **High** | 多服务级联故障，大量用户受影响 | 核心微服务宕机、消息队列不可用、缓存全量失效 |
| **Medium** | 单服务故障，部分用户受影响 | 单一微服务宕机、局部网络延迟、非核心数据库慢查询 |
| **Low** | 局部降级，极少用户感知 | 非关键服务异常、日志服务不可用、监控告警延迟 |
| **Info** | 无用户感知，仅内部观测变化 | 后台任务延迟、非实时数据同步延迟、缓存命中率下降 |

## 混沌实验类型矩阵

| 实验类别 | 故障注入方式 | 适用目标 |
|---------|------------|---------|
| 服务级故障 | 进程 kill、容器驱逐、Pod 删除、OOM 模拟 | service |
| 网络级故障 | 延迟注入、丢包模拟、网络分区、DNS 劫持、带宽限制 | network |
| 数据库级故障 | 连接池耗尽、慢查询注入、主从延迟、磁盘满模拟 | database |
| 基础设施故障 | CPU 压力、内存压力、磁盘 IO 压力、节点宕机 | infra |
| 流量级故障 | 流量激增、请求风暴、热点 Key、大对象请求 | service/network |
| 依赖级故障 | 第三方 API 超时、证书过期、配置中心不可用 | service/infra |

**如果某些故障注入工具未安装或无法运行**：executor 标注"该工具不可用，建议手动执行"，不阻塞流程。

## 实验维度分配

双 designer 按各自负责的维度独立设计实验，analyst 按权重合并为韧性评估总分。

| 维度 | 权重 | 负责人 | 设计内容 |
|------|------|--------|---------|
| 服务容错 | 25% | designer-1 | 单服务宕机恢复、多服务级联影响、容器自愈能力、健康检查有效性 |
| 资源韧性 | 15% | designer-1 | CPU/内存/磁盘极端场景、资源限制（limits/requests）有效性、OOM 处理 |
| 降级策略 | 15% | designer-1 | 熔断器配置验证、降级开关有效性、兜底方案覆盖度 |
| 网络韧性 | 20% | designer-2 | 网络延迟/丢包/分区场景、超时配置合理性、重试策略验证 |
| 数据韧性 | 15% | designer-2 | 数据库故障切换、缓存雪崩/穿透/击穿、数据一致性验证 |
| 流量韧性 | 10% | designer-2 | 限流策略验证、流量突增应对、排队机制有效性 |

**designer-1 总权重：55%**（服务容错 25% + 资源韧性 15% + 降级策略 15%）
**designer-2 总权重：45%**（网络韧性 20% + 数据韧性 15% + 流量韧性 10%）

**`--target` 参数处理**：如果用户指定了 `--target`，仅设计对应目标类型的实验。例如 `--target=network` 仅设计网络级故障实验。未指定则默认全部类型。

## 阶段零：系统拓扑识别

### 步骤 1：解析实验范围

Team lead 分析项目和参数：

1. 解析 `--target` 参数，确定实验目标类型
2. 阅读项目结构，识别技术栈（语言、框架、部署方式、编排工具）
3. 识别系统边界：
   - 微服务列表（服务名、端口、协议）
   - 服务间依赖关系（同步调用、异步消息）
   - 数据存储组件（数据库、缓存、对象存储、消息队列）
   - 外部依赖（第三方 API、CDN、DNS、证书）
   - 基础设施（K8s、Docker Compose、VM、Serverless）
4. 输出实验范围清单

### 步骤 2：用户确认实验范围

**标准模式**：向用户展示系统拓扑和实验范围清单，AskUserQuestion 确认：
- 确认实验范围
- 调整实验目标（增删服务/组件）
- 补充系统关注点（如已知薄弱环节）

**单轮确认模式**：跳过确认，直接进入阶段一。
**完全自主模式**：自动决策，不询问用户，直接进入阶段一。

## 阶段一：拓扑扫描 + 双路实验设计

### 步骤 3：启动 scanner 和双 designer

三者并行启动。

**Scanner 系统拓扑扫描**：
1. 阅读项目结构，识别部署配置（docker-compose.yml、k8s manifests、terraform 等）
2. 解析服务依赖关系：
   - 服务发现配置（Consul、Eureka、K8s Service）
   - API 网关路由（Nginx、Kong、Envoy、Istio）
   - 消息队列拓扑（Kafka topics、RabbitMQ exchanges）
   - 数据库连接配置（连接池、主从、分片）
3. 识别关键路径：

| 分析维度 | 扫描内容 | 输出 |
|---------|---------|------|
| 服务依赖链 | 调用链深度、扇出度、单点依赖 | 依赖拓扑图 |
| 数据流向 | 读写路径、缓存策略、一致性要求 | 数据流图 |
| 容错配置 | 熔断器、重试、超时、降级开关 | 容错配置清单 |
| 资源配置 | CPU/内存 limits、副本数、HPA 配置 | 资源配置清单 |
| 监控覆盖 | 健康检查、metrics、告警规则 | 可观测性清单 |

4. 汇总输出**系统拓扑报告**发送给 team lead
5. Scanner 完成后关闭（不参与后续阶段）

**双 designer 同时阅读项目**：
- 阅读项目结构（目录布局、部署配置、容错机制）
- 理解技术栈、框架和容错策略
- 梳理服务间通信链路和数据流
- 各自输出项目架构概况给 team lead

### 步骤 4：分发 scanner 报告

Team lead 收到 scanner 报告后，将系统拓扑数据分发给两位 designer，作为实验设计的参考数据。

Designer 在设计时必须参考 scanner 结果——例如 scanner 发现某服务是单点部署，designer-1 应设计针对该单点的故障注入实验。

### 步骤 5：独立并行实验设计

两位 designer 各自设计负责维度的实验，**互不交流**。

**Designer-1（故障注入维度）** — 设计以下类别的实验：

| 实验类别 | 设计内容 | 具体实验项 |
|---------|---------|-----------|
| **服务宕机** | 单服务故障恢复能力 | Kill 进程后自动恢复时间、K8s Pod 驱逐后重建时间、多实例下单实例故障影响范围、健康检查探针有效性验证 |
| **级联故障** | 多服务连锁反应 | 核心服务宕机后下游影响链、熔断器是否正确触发、降级方案是否生效、故障传播深度 |
| **资源耗尽** | 极端资源场景 | CPU 100% 下服务响应能力、内存逼近 OOM 行为、磁盘满场景处理、文件描述符耗尽 |
| **容器编排** | 编排平台韧性 | Pod 驱逐策略验证、节点故障后 Pod 迁移、HPA 扩容响应时间、PDB 保护有效性 |

- 每个实验输出：实验名称 + 故障注入方式 + 预期行为 + 观测指标 + 回滚步骤 + 爆炸半径评估
- 参考 scanner 的拓扑数据，优先设计关键路径上的实验
- 标注每个实验的前置条件和风险提示

**Designer-2（负载异常维度）** — 设计以下类别的实验：

| 实验类别 | 设计内容 | 具体实验项 |
|---------|---------|-----------|
| **网络故障** | 网络层面异常 | 延迟注入（50ms/200ms/1s/5s）、丢包模拟（1%/5%/20%/50%）、网络分区（脑裂场景）、DNS 解析失败、TLS 证书过期 |
| **流量激增** | 流量异常场景 | 正常流量 2x/5x/10x 突增、热点 Key 集中访问、大对象请求风暴、慢客户端（Slowloris） |
| **数据库异常** | 数据存储层故障 | 连接池耗尽、慢查询注入、主从复制延迟、缓存全量失效（雪崩）、缓存穿透/击穿 |
| **依赖异常** | 外部依赖故障 | 第三方 API 超时/错误、配置中心不可用、消息队列积压、对象存储不可达 |

- 每个实验输出：实验名称 + 故障注入方式 + 预期行为 + 观测指标 + 回滚步骤 + 爆炸半径评估
- 参考 scanner 的拓扑数据，优先设计高扇出服务的实验
- 标注每个实验的前置条件和风险提示

## 阶段二：合并分析

### 步骤 6：启动 analyst

两位 designer 设计完成后，启动 analyst。Team lead 将以下材料发送给 analyst：
- Scanner 系统拓扑报告
- Designer-1 故障注入实验方案
- Designer-2 负载异常实验方案
- 系统架构和边界信息

### 步骤 7：合并去重与爆炸半径评估

Analyst 执行合并分析：

1. **去重**：识别两位 designer 方案中的重叠实验（如都涉及数据库连接池），合并为单条记录
2. **交叉校验**：检查两位 designer 对同一场景的爆炸半径评估是否一致
3. **分组**：按实验目标分组（服务级、网络级、数据库级、基础设施级）
4. **排序**：按爆炸半径从小到大排序（先执行低风险实验，逐步升级）
5. **依赖分析**：标注实验间的依赖关系（如"网络分区实验需在服务容错实验之后"）

**共识度计算公式**：
```
两位 designer 设计有部分维度交叉（如 designer-1 的级联故障 与 designer-2 的依赖异常可能重叠）。
对于重叠区域的实验：

共识实验数 = 两位 designer 都设计的实验数量（针对相同组件且描述相同故障类型）
总重叠区域实验数 = 重叠区域去重后的实验总数

共识度 = 共识实验数 / 总重叠区域实验数 × 100%（如无重叠区域则为 100%）

爆炸半径一致性 = 两位 designer 对同一实验爆炸半径评估完全一致的比例
争议率 = 爆炸半径差异 ≥ 2 级的实验数 / 两位 designer 都设计的实验数 × 100%

争议率 > 50% → 触发熔断，必须暂停问用户
```

### 步骤 8：差异校验与熔断检查

**差异校验**：
- 如果两位 designer 对同一场景的爆炸半径评估差异 ≥ 2 级（如一方定 Critical、另一方定 Medium），analyst 标注为"争议项"
- 争议项占比 > 50% → **不可跳过，必须暂停问用户**（熔断机制）

**关键路径熔断**：
- 发现关键路径（核心交易链路、认证链路等）上没有降级方案 → **不可跳过，必须暂停问用户**（熔断机制）
- 向用户展示关键路径风险，确认是否继续实验设计

**无熔断触发**时，analyst 输出**统一实验方案**：

```
## 统一混沌实验方案

### 实验统计
- Critical 爆炸半径: X 个 | High: X 个 | Medium: X 个 | Low: X 个 | Info: X 个
- 总计: X 个实验场景

### 实验分组（按执行顺序）

#### 第一轮：低风险实验（Info/Low）
  ├─ [Low] 实验名称 - 目标组件 - 故障类型 - 来源：designer-1/designer-2
  └─ 预期行为 + 观测指标 + 回滚步骤

#### 第二轮：中等风险实验（Medium）
  └─ ...

#### 第三轮：高风险实验（High/Critical）
  └─ ...

### 争议项（如有）
1. 实验名称 - designer-1 评估: X / designer-2 评估: Y - analyst 最终评估: Z - 理由
```

Analyst 完成后关闭。

### 步骤 9：韧性评估评分

Team lead 根据统一实验方案计算各维度韧性评分：

评分规则：
- 基础分 10.0，按风险发现扣分
- 关键路径无降级方案：每个扣 2.0 分
- 单点故障（无冗余）：每个扣 1.5 分
- 容错配置缺失（无熔断/重试/超时）：每个扣 1.0 分
- 监控覆盖不足：每个扣 0.5 分
- 最佳实践建议：不扣分

按权重计算总分，输出各维度和总体韧性评分。

## 核心原则

- **安全第一**：默认 dry-run 模式，真实执行需明确确认，所有实验必须有回滚步骤
- **双路独立**：两位 designer 从不同维度独立设计，互不交流，避免盲区和偏见
- **渐进式实验**：从低风险到高风险逐步升级，先验证基础容错再测试极端场景
- **爆炸半径可控**：每个实验必须明确爆炸半径，优先在非生产环境执行
- **可观测性**：每个实验必须定义观测指标和成功标准，实验结果必须可量化
- **回滚优先**：每个实验必须有明确的回滚步骤，确保快速恢复
- **韧性导向**：关注系统整体韧性而非单点，识别级联故障链和薄弱环节
- **持续改进**：混沌实验不是一次性活动，建议建立常态化实验机制

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 系统拓扑信息不完整 | Scanner 基于代码和配置文件尽力识别，标注"拓扑信息可能不完整" |
| 部署配置缺失（无 K8s/Docker 配置） | 基于代码层面进行容错分析，跳过基础设施级实验 |
| 项目规模过大（>50 个微服务） | Scanner 按业务域分批扫描；designer 聚焦核心链路，标注未覆盖区域 |
| 两位 designer 爆炸半径评估差异过大（争议项 > 50%） | 触发熔断，暂停问用户裁决争议项 |
| 关键路径无降级方案 | 触发熔断，暂停向用户确认是否继续 |
| 发现生产环境正在运行的故障 | 立即通知用户（无论任何模式），建议启动 incident 响应 |
| 混沌工具未安装（Chaos Monkey/Litmus/ChaosBlade） | Executor 输出手动执行方案，标注推荐工具 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前步骤恢复 |
| 实验范围与实际架构不匹配 | Team lead 在阶段零发现后调整范围，标准模式需用户确认 |

### 共识度计算

team lead 按五维度评估双路分析的共识度：

| 维度 | 权重 |
|------|------|
| 发现一致性（相同问题/结论） | 20% |
| 互补性（独有但不矛盾的发现） | 20% |
| 分歧程度（直接矛盾的结论） | 20% |
| 严重度一致性（同一问题的严重等级差异） | 20% |
| 覆盖完整性（两路合并后的覆盖面） | 20% |

共识度 = 各维度加权得分之和

- **≥ 60%**：自动合并，分歧项由 team lead 裁决
- **50-59%**：合并但标注分歧，收尾时汇总争议点
- **< 50%**：触发熔断，暂停并向用户确认方向
