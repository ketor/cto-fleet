**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--scope=module|package|system`：重构范围（默认 `module`）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从重构目标描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 固定节点 + 条件节点，全部 AskUserQuestion | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅固定节点（影响确认、方案确认、architecture 级每阶段确认） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **重构类型判断不确定** → team lead 自行判断，在方案确认时告知用户判定结果
- **planner-1/planner-2 方案分歧** → team lead 综合两方观点裁决，在方案确认时附带争议摘要和裁决理由
- **批次执行期间决策升级** → coder 先问 team lead，team lead 自行决策
- **批次测试失败** → coder 尝试修复（最多 2 次），仍失败则 revert 并重新规划
- **迭代超 3 轮** → **不可跳过，必须暂停问用户**（熔断机制）
- **architecture 级重构** → **不可跳过，每个阶段必须用户确认**（熔断机制）

完全自主模式下：所有节点（包括影响确认、方案确认等固定节点）均自动决策，不询问用户。收尾时汇总所有自动决策。唯一例外：迭代超 3 轮仍自动停止（但不询问用户，而是 team lead 自行裁决是否继续）。architecture 级重构在完全自主模式下也自动决策（但 team lead 需格外谨慎，收尾时详细汇总每个阶段的自动决策）。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

**文件命名规范**：

| 角色输出 | 文件名 |
|---------|--------|
| Scanner 报告 | `scanner-report.md` |
| Reviewer-N 第R轮 | `reviewer-{N}-round-{R}.md` |
| 合并报告第R轮 | `merged-report-round-{R}.md` |
| 根因分组 | `root-cause-groups-round-{R}.md` |
| Fixer 第R轮 | `fixer-round-{R}.md` |
| Tester 第R轮 | `tester-round-{R}.md` |
| Architect-N 方案 | `architect-{N}-design.md` |
| 任务拆解 | `task-breakdown.md` |
| Coder-N 任务T | `coder-{N}-task-{T}.md` |
| 审查任务T | `review-task-{T}.md` |
| 集成测试第R轮 | `integration-test-round-{R}.md` |
| 最终报告 | `final-report.md` |

> 仅当角色存在于当前 skill 时使用对应命名。未列出的角色用 `{role}-{context}.md` 格式。

## 重构类型分类

| 类型 | 描述 | 风险等级 |
|------|------|---------|
| **rename** | 重命名（变量/函数/类/模块） | Low |
| **extract** | 提取（函数/类/模块/接口） | Medium |
| **merge** | 合并（重复代码/相似模块） | Medium |
| **split** | 拆分（大模块/大类/大函数） | Medium |
| **interface** | 接口变更（参数/返回值/协议） | High |
| **pattern** | 设计模式迁移（如回调→Promise→async/await） | High |
| **architecture** | 架构迁移（如单体→微服务、同步→异步） | Critical |

## 重构范围

| 范围 | 说明 | 团队配置 |
|------|------|---------|
| **module** | 单模块内部重构 | analyzer + 2 planner + coder + reviewer（无 tester，依赖现有测试） |
| **package** | 跨多模块重构 | analyzer + 2 planner + coder + reviewer + tester |
| **system** | 跨系统/跨服务重构 | analyzer + 2 planner + coder + reviewer + tester |

## 流程概览

```
阶段零  重构目标分析 → team lead 解析目标 → 确定重构类型和范围 → 结构化重构目标
         ↓
阶段一  影响分析 → analyzer 深度分析影响范围（引用链、测试覆盖、外部依赖）→ 输出影响评估报告
         ↓
阶段二  双路方案设计 → planner-1 + planner-2 独立设计重构方案 → team lead 合并评审 → 用户确认
         ↓
阶段三  分批实施 → coder 按批次执行（每批 = 一个原子重构步骤，worktree 隔离）
         → reviewer 审查 → tester 验证 → 合并 → 循环下一批
         ↓
阶段四  全量验证 → 全部批次完成后整体验证（全量测试 + reviewer 整体审查）
         ↓
阶段五  收尾 → 总结 + 清理

module 范围精简：跳过 tester，依赖项目现有测试套件验证
architecture 级加固：每个阶段结束后必须用户确认，不可跳过
```

## 角色定义

| 角色 | 职责 |
|------|------|
| analyzer | 分析重构影响范围——依赖关系图、调用链追踪、测试覆盖分析、外部依赖识别。输出影响评估报告后关闭。**不参与方案设计和代码修改。** |
| planner-1 | 独立制定重构方案——步骤分解、中间状态设计、迁移路径规划、回滚策略。**不与 planner-2 交流，不编写实现代码。** |
| planner-2 | 同 planner-1 职责，独立制定重构方案。**不与 planner-1 交流，不编写实现代码。** |
| coder | 按批次执行重构（使用 worktree 隔离），每批完成后执行自检清单。**只做重构实现，不做方案设计。** |
| reviewer | 审查每批重构的正确性——接口兼容性、行为一致性、无遗漏引用。全量完成后做整体审查。**不编写或修改代码，只做审查。** |
| tester | 运行测试验证每批重构无回归，监控覆盖率无下降。全量完成后执行完整测试套件。**（module 范围不启动）** |

## 阶段零：重构目标分析

### 步骤 1：解析重构目标

Team lead 将用户的重构描述转化为结构化目标：

1. **重构类型**：从上表中识别（rename / extract / merge / split / interface / pattern / architecture）
2. **重构范围**：根据 `--scope` 或自动判断（module / package / system）
3. **目标描述**：具体的重构目标（"将 UserService 拆分为 AuthService 和 ProfileService"而非"拆分用户服务"）
4. **预期收益**：为什么要做这次重构（可维护性/性能/可测试性/解耦）
5. **安全约束**：不能破坏的行为、必须保持的接口、不能中断的服务

### 步骤 2：风险预评估

Team lead 根据重构类型和范围做初步风险评估：

| 组合 | 风险等级 | 处理方式 |
|------|---------|---------|
| rename + module | 低 | 正常流程 |
| extract/merge/split + module | 中 | 正常流程 |
| extract/merge/split + package | 中高 | 正常流程，加强审查 |
| interface + 任意范围 | 高 | 每批必须测试验证 |
| pattern + package/system | 高 | 每批必须测试验证 |
| architecture + 任意范围 | 严重 | 每阶段用户确认，不可自动决策 |

确认方式：
- **标准模式**：向用户展示结构化目标和风险评估，AskUserQuestion 确认
- **单轮确认模式**：team lead 自行制定，在方案确认时一并展示（architecture 级除外，必须确认）
- **完全自主模式**：自动决策，不询问用户

## 阶段一：影响分析

### 步骤 3：启动 analyzer

启动 analyzer，传入重构目标和范围。Analyzer 执行深度影响分析：

1. **依赖关系图**——具体操作方法：
   - 直接依赖：`Grep("import |from .* import|require\(|use |#include", path="<重构目标文件/目录>")` 提取所有 import 关系
   - 反向依赖：`Grep("<目标模块名>|<目标类名>|<目标函数名>")` 全局搜索谁在使用要重构的代码
   - 包依赖分析（按语言）：
     - JavaScript/TypeScript：分析 `package.json` 依赖 + `Grep("require\(|import .* from", type="ts")` 或 `type="js"`
     - Python：`Grep("import <模块>|from <模块>", type="py")` + 检查 `setup.py`/`pyproject.toml` 的依赖
     - Go：`Grep("import.*<包路径>", type="go")` + 分析 `go.mod` 依赖
     - Java/Kotlin：`Grep("import <包路径>", type="java")` + 分析 `build.gradle`/`pom.xml`
     - Rust：`Grep("use <crate>|extern crate", type="rust")` + 分析 `Cargo.toml`
   - 间接依赖：从直接依赖出发，递归搜索传递依赖（最多 3 层）
2. **调用链追踪**——具体操作方法：
   - `Grep("<函数名>\(|<方法名>\(|\.<函数名>|::<函数名>")` 搜索所有调用点
   - 入口点识别：`Grep("main\(|app\.(get|post|put)|@(Controller|RequestMapping)|func main|if __name__")` 定位 API/CLI/UI 入口
   - 从入口点到目标函数的调用深度评估
3. **测试覆盖分析**——具体操作方法：
   - 搜索已有测试：`Grep("<目标函数名>|<目标类名>", path="**/test*/**")` 或 `Glob("**/*test*.*", "**/*spec*.*")`
   - 测试类型分布：`Glob("**/unit/**", "**/integration/**", "**/e2e/**", "**/tests/**")` 识别测试目录结构
   - 运行覆盖率工具（如有）：根据语言运行 `jest --coverage`、`pytest --cov`、`go test -cover` 等
   - 测试缺口标注：没有被任何测试引用的公开函数/方法列为缺口
4. **外部依赖识别**——具体操作方法：
   - 配置文件引用：`Grep("<目标模块名>|<目标类名>", glob="*.{yaml,yml,json,toml,ini,conf}")`
   - 文档引用：`Grep("<目标模块名>|<目标类名>", glob="*.md")`
   - CI/CD 脚本引用：`Grep("<目标模块名>", path="**/.github/**")` 或其他 CI 配置目录
   - 环境变量引用：`Grep("<目标模块相关环境变量>", glob=".env*")`

### 步骤 4：输出影响评估报告

Analyzer 输出结构化影响评估报告：
- 影响范围摘要（涉及 N 个文件、M 个模块、K 个测试）
- 高风险区域标注（测试覆盖薄弱的区域、外部依赖密集的区域）
- 建议的重构顺序（先内后外、先叶后根）
- 风险缓解建议

报告完成后 analyzer 关闭。

确认方式：
- **标准模式**：向用户展示影响报告摘要，AskUserQuestion 确认继续
- **单轮确认模式**：team lead 确认影响可控后继续（architecture 级除外，必须用户确认）
- **完全自主模式**：自动决策，不询问用户

## 阶段二：双路方案设计

### 步骤 5：启动 planner-1 和 planner-2

同时启动两位 planner。传入重构目标、影响评估报告。两位 planner 各自独立工作，team lead 不让两位 planner 看到对方的方案，确保独立性。

每位 planner 输出：
1. **重构步骤分解**：将重构拆分为有序的原子步骤（每步独立可编译可测试）
2. **中间状态设计**：每步完成后系统应处于什么状态（旧接口和新接口如何共存）
3. **迁移路径**：从当前状态到目标状态的具体路径
4. **批次划分**：将步骤组织为批次（batch），每批 = 一个原子重构单元
5. **回滚策略**：每批的回滚方式（git revert 即可 or 需要额外清理）
6. **风险标注**：每批的风险等级和注意事项

### 步骤 6：合并评审

两份方案完成后，team lead 逐项对比合并评审：

| 对比结果 | 处理方式 |
|---------|---------|
| **一致方案**（步骤拆分和顺序相同） | 直接采纳，标记为"共识" |
| **互补方案**（一方考虑了另一方未覆盖的场景） | 合并最优部分，标记为"互补" |
| **差异方案**（批次粒度不同但方向一致） | 选择原子性更好的，标记为"共识" |
| **冲突方案**（对重构路径有根本分歧） | 标注为"待仲裁" |

评估要点：
- 哪种方案的原子性更好（每步更安全可回滚）
- 哪种方案的中间状态更稳定（不会出现半成品状态）
- 哪种方案的批次划分粒度更合理

输出：
1. **共识清单**：双方一致的重构步骤和批次划分
2. **互补清单**：一方独有的有价值设计（额外的安全检查、更好的中间状态等）
3. **分歧清单**：矛盾之处及双方方案对比
4. **共识度评估**：共识度 = (共识步骤数 + 互补步骤数) / 总步骤数(去重并集) x 100%
5. **最终重构方案**：合并两方优点的完整方案

有分歧无法裁决时：
- **标准模式**：team lead 用 AskUserQuestion 交由用户决策
- **单轮确认模式**：team lead 综合两方观点裁决，在方案确认时附带争议摘要和裁决理由
- **完全自主模式**：自动决策，不询问用户

### 步骤 7：用户确认

Team lead 向用户输出：
- 两份独立方案的核心差异对比
- 合并后的最终重构方案（批次列表、每批内容、预估风险）
- 影响范围覆盖情况（对照影响评估报告）
- 回滚策略说明
- 预估的批次总数和执行顺序

用 AskUserQuestion 等待用户确认后，方可进入下一阶段。

**architecture 级重构**：无论标准/单轮确认模式，此步骤必须用户确认。完全自主模式下自动决策。

## 阶段三：分批实施

### 步骤 8：启动 coder、reviewer 和 tester

启动 coder（使用独立 worktree），启动 reviewer。如果范围为 package 或 system，同时启动 tester。三者全程保持存活直到收尾。

关闭 planner-1 和 planner-2（方案已确定，不再需要）。

**Worktree 分支命名规范**：`refactor/<team-name>/<batch-N>`，例如 `refactor/team-refactor-20260311-100000/batch-1`。合并后分支自动清理。

### 步骤 9：批次执行循环

**每个批次是一个原子重构步骤。批次必须独立可编译可测试，不能破坏现有功能。**

对每个批次 N（从 batch-1 开始），执行以下循环：

**9a. Coder 执行重构**：
- 在 worktree 中执行当前批次的重构操作
- 严格按照方案中该批次的内容执行，不超出范围
- 修改后确保编译通过

**9b. Coder 自检清单**（提交审查前必须完成）：
1. 编译/构建通过（无新增错误或警告）
2. 运行 lint/format（如项目有配置）
3. 运行现有测试套件——全部通过
4. 自查修改是否与批次描述的范围一致（无超出范围的改动）
5. 自查是否有遗漏的引用（搜索旧名称/旧接口确认无残留）
6. 检查覆盖率无下降（如有覆盖率工具）

自检全部通过后，通知 reviewer 和 team lead。

**9c. Reviewer 审查**：

| 维度 | 审查内容 | 是否阻塞 |
|------|---------|---------|
| 行为一致性 | 重构前后外部可观察行为是否一致 | 是 |
| 接口兼容性 | 公开接口的签名、语义是否保持兼容（或按方案正确变更） | 是 |
| 引用完整性 | 是否有遗漏的旧引用/旧命名残留 | 是 |
| 中间状态 | 当前批次完成后系统是否处于方案描述的中间状态 | 是 |
| 回滚安全 | 该批次是否可通过 git revert 安全回滚 | 是 |
| 代码质量 | 重构后代码是否比重构前更清晰 | 不阻塞，建议改进 |

审查不通过则打回 coder 修复，循环直到通过。

**9d. Tester 验证**（module 范围跳过此步，由 coder 自检中的测试覆盖）：
- 运行全量测试套件，确认无回归
- 检查测试覆盖率无下降
- 如有新增代码路径，建议补充测试
- 测试失败则通知 team lead

**9e. 合并**：
- 审查和测试通过后，team lead 在主分支执行 `git merge <worktree-branch>`
- 如有合并冲突，让 coder 解决冲突后重新提交
- 合并成功后标记该批次完成，进入下一批次

### 步骤 10：批次失败处理

**如果批次测试失败或审查不通过：**

| 失败类型 | 处理方式 |
|---------|---------|
| 编译失败 | Coder 自行修复，无法修复则升级 team lead |
| 测试失败（可定位） | Coder 修复（最多 2 次尝试） |
| 测试失败（2 次修复仍失败） | **熔断**：revert 该批次，team lead 重新规划该批次的拆分方式。标准模式问用户；单轮确认模式 team lead 裁决；完全自主模式自动决策 |
| 审查不通过（行为不一致） | Coder 修复后重新审查 |
| 审查不通过（遗漏引用） | Coder 补全后重新审查 |
| 合并冲突 | Coder 解决冲突，reviewer 重新审查冲突解决部分 |

**architecture 级重构额外规则**：每完成 3 个批次后，team lead 暂停，向用户汇报进展，AskUserQuestion 确认继续。

---

## 阶段四：全量验证

### 步骤 11：全量测试

所有批次合并完成后，执行全量验证：

**Tester 全量测试**（module 范围由 coder 执行）：
- 运行完整测试套件（单测 + 集成测试 + E2E 测试，如有）
- 输出测试报告：通过率、失败列表、覆盖率对比（重构前 vs 重构后）
- 覆盖率不应下降；如有下降需说明原因

### 步骤 12：Reviewer 整体审查

Reviewer 对重构后的完整代码做整体审查：
- 重构目标是否达成（对照阶段零的目标描述）
- 所有旧引用/旧命名是否清理完毕（全局搜索确认）
- 模块间接口是否对齐
- 代码整体质量是否提升（对比重构前）
- 是否有遗留的临时兼容代码需要清理

### 步骤 13：验证失败迭代

**最大迭代轮次：3 轮。** 超出后暂停，team lead 向用户汇报当前状态，由用户决定继续、调整方向或终止。

每轮迭代：
- Reviewer 和 tester 协作分析失败原因：是单批次引入还是批次间交互问题？
- **单批次问题**：coder 在主分支修复 → reviewer 审查 → tester 回归测试
- **批次间交互问题**：team lead 评估是否需要 revert 部分批次重新规划
  - **标准模式**：AskUserQuestion 向用户确认处理方式
  - **单轮确认模式**：team lead 自行决策（architecture 级除外，必须用户确认）
  - **完全自主模式**：自动决策，不询问用户
- 循环直到全量验证通过，或达到最大迭代轮次

---

## 阶段五：收尾

### 步骤 14：最终总结

全量验证通过后，team lead 按 `--lang` 指定的语言向用户输出：
- 重构目标达成情况（逐项对照）
- 重构类型和范围
- 批次执行摘要（共 N 批，每批概述）
- 影响范围实际覆盖（对照影响评估报告）
- 测试验证情况（通过率、覆盖率变化）
- 关键决策点回顾
- 已知的遗留项或后续建议
- **方案共识说明**：共识度、planner-1 和 planner-2 的共识/互补/分歧汇总、仲裁结果
- **（单轮确认模式/完全自主模式）自动决策汇总**：列出所有自动决策的节点、决策内容和理由

### 步骤 15：跨团队衔接建议（可选）

Team lead 根据项目情况向用户建议后续动作：
- **interface/pattern/architecture 级重构**：建议运行 `/team-review` 对重构代码做全面审查
- **涉及 API 接口变更**：建议运行 `/team-api-design` 更新 API 定义和文档
- **重构完成后需要发布**：建议运行 `/team-release` 管理版本发布流程
- **涉及文档变更**：建议运行 `/team-onboard` 更新项目入职文档和架构概览
- **涉及依赖方**：建议通知下游依赖方适配变更
- **重构涉及基础设施变更**：建议运行 `/team-cost` 评估资源配置变化的成本影响
- 用户可选择执行或跳过，不强制。

### 步骤 16：清理

关闭所有 teammate，用 TeamDelete 清理 team。

## 核心原则

- **影响先知**：先分析影响范围，再设计方案，再动手重构
- **原子批次**：每个批次是独立可编译可测试可回滚的原子操作
- **持续可用**：重构过程中系统始终保持可工作状态，任何中间状态都是合法状态
- **双路设计**：两位 planner 独立设计方案，避免单一视角盲区
- **逐批验证**：每批完成后立即验证，不积累风险
- **安全回滚**：任何批次都可以通过 git revert 安全回滚
- **不确定就问**：标准模式下任何不确定事项必须交由用户确认；单轮确认模式下条件节点可自动决策；完全自主模式下全部自动决策
- **熔断保护**：批次 2 次修复仍失败必须 revert 重新规划；architecture 级每阶段必须用户确认
- **覆盖率守恒**：重构不应降低测试覆盖率

---

### 共识度计算

team lead 按五维度评估双路分析的共识度：

| 维度 | 权重 |
|------|------|
| 发现一致性（相同问题/结论） | 20% |
| 互补性（独有但不矛盾的发现） | 20% |
| 分歧程度（直接矛盾的结论） | 20% |
| 严重度一致性（同一问题的严重等级差异） | 20% |
| 覆盖完整性（两路合并后的覆盖面） | 20% |

共识度 = 各维度加权得分之和

- **≥ 60%**：自动合并，分歧项由 team lead 裁决
- **50-59%**：合并但标注分歧，收尾时汇总争议点
- **< 50%**：触发熔断，暂停并向用户确认方向

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 编译/构建失败 | Coder 自行修复；无法修复则升级 team lead |
| 批次测试失败（2 次修复仍失败） | Revert 该批次，team lead 重新规划拆分方式 |
| Reviewer 发现行为不一致 | Coder 修复后重新审查，仍不通过则升级 team lead |
| 遗漏旧引用/旧命名 | Coder 全局搜索补全，reviewer 重新审查 |
| 合并冲突 | Coder 解决冲突，reviewer 审查冲突解决部分 |
| 覆盖率下降 | Coder 补充测试用例覆盖新路径 |
| Analyzer 发现影响范围过大 | Team lead 与用户协商缩小重构范围或分阶段执行 |
| Planner 方案分歧过大 | Team lead 裁决；标准模式交由用户选择；单轮确认模式附带理由说明；完全自主模式自动决策 |
| Architecture 级重构中途发现方案不可行 | 立即暂停，向用户汇报，协商调整方向或终止 |
| 全量验证失败超 3 轮 | 暂停，向用户汇报当前状态，由用户决定继续/调整/终止 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| 外部依赖方受影响 | Team lead 提醒用户通知下游依赖方，不自动推送变更 |
| 重构涉及外部 API 兼容性 | Analyzer 在影响评估中标注外部 API 依赖；planner 方案必须包含 API 兼容层或版本迁移策略；coder 在重构期间保留旧 API 作为兼容适配 |
| 无测试框架/测试基础设施 | Coder 自检中跳过自动化测试步骤，改为手动验证行为一致性；reviewer 加强行为一致性审查；team lead 在总结中标注"无自动化测试保障"风险 |
| Git 仓库状态异常（脏工作区/未提交变更） | Team lead 在创建 worktree 前检测工作区状态，提示用户先 stash 或提交变更，确保 worktree 基于干净状态创建 |
| 项目过大无法完整分析 | Analyzer 根据重构目标缩小影响分析范围，聚焦直接依赖链（最多 3 层）；team lead 在总结中标注分析范围限制 |
