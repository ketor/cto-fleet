---
name: team-incident
description: 启动一个故障响应团队（triager/investigator×2/fixer/verifier），按阶段协作完成故障分级、快速止血、双路调查、根因定位、修复验证和收尾总结。使用方式：/team-incident [--auto (全自动，不询问)] [--once (仅确认一次后自动执行)] [--severity=P0|P1|P2|P3] [--lang=zh|en] 故障描述或告警信息
argument-hint: [--auto (全自动，不询问)] [--once (仅确认一次后自动执行)] [--severity=P0|P1|P2|P3] [--lang=zh|en] 故障描述或告警信息
---

**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--severity=P0|P1|P2|P3`：预设故障等级（跳过 triager 分级，直接采用）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从故障描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 固定节点 + 条件节点，全部 AskUserQuestion | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅固定节点（止血方案确认、修复方案确认） | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下条件节点自动决策规则：
- **故障等级不确定** → triager 自行判断，在止血/调查前告知用户判定结果
- **investigator-1/investigator-2 结论分歧** → team lead 综合双路调查结果裁决，在根因确认时附带分歧摘要和裁决理由
- **调查期间需要扩大排查范围** → investigator 自行扩展
- **止血方案选择** → **不可跳过，必须暂停问用户**（P0/P1 止血风险过高）
- **修复后验证失败超 2 轮** → **不可跳过，必须暂停问用户**（熔断机制）
- **P2/P3 修复方案确认** → fixer 自行决策，收尾时汇总

完全自主模式下：所有节点均自动决策，不询问用户。仅熔断机制（止血方案确认、修复后验证失败超 2 轮）仍然生效，触发时必须暂停。

使用 TeamCreate 创建 team（名称格式 `team-incident-{YYYYMMDD-HHmmss}`，如 `team-incident-20260311-143022`，避免多次调用冲突），你作为 team lead 按以下流程协调。

## 故障严重程度定义

| 等级 | 描述 | 响应时间 | 处置策略 |
|------|------|---------|---------|
| **P0** | 服务完全不可用 / 数据丢失 / 安全事件 / GPU 集群大规模掉线 | 立即 | 止血优先，全员响应 |
| **P1** | 核心功能受损 / 严重性能退化 / API 网关大面积超时 / 训练任务批量失败 | <15min | 止血优先 |
| **P2** | 非核心功能受损 / 部分用户影响 / 单节点异常 / 队列积压可自愈 | <1h | 正常调查流程，跳过止血 |
| **P3** | 轻微异常 / 不影响用户 / 监控告警无业务影响 | <4h | 仅调查，不需要止血 |

## 流程概览

```
阶段零  故障分级 → triager 评估严重程度 (P0-P3) + 影响范围 + 紧急度
         ↓
阶段一  快速止血（仅 P0/P1）→ 如有明显回滚/降级方案，先执行止血
         ↓
阶段二  双路调查 → investigator-1 (变更角度) + investigator-2 (系统角度) 独立调查
         ↓
阶段三  根因确认 → merge investigations → consensus / divergence / arbitration
         ↓
阶段四  修复实施 → fixer 实施修复 (hotfix branch) → verifier 验证
         ↓
阶段五  收尾 → incident summary + timeline + action items + cleanup

P0/P1 流程：阶段零 → 阶段一（止血）→ 阶段二 → 阶段三 → 阶段四 → 阶段五
P2/P3 流程：阶段零 → 跳过止血 → 阶段二 → 阶段三 → 阶段四 → 阶段五
```

## 角色定义

| 角色 | 职责 |
|------|------|
| triager | 快速评估故障严重程度、影响范围、紧急度。识别相关代码模块和日志来源。完成评估后输出分级报告。**仅在分级阶段工作，分级完成后释放。** |
| investigator-1 | 独立调查——从**代码变更角度**（最近的 commit/deploy 变更分析、代码 diff 审查、配置变更追踪）。**不与 investigator-2 交流，独立输出调查报告。** |
| investigator-2 | 独立调查——从**系统状态角度**（日志分析、配置检查、资源使用、Pod 状态、队列状况）。**不与 investigator-1 交流，独立输出调查报告。** |
| fixer | 根据根因分析实施止血方案或永久修复（hotfix/rollback/config change），编写修复代码和回归测试。 |
| verifier | 验证止血效果或修复效果，运行回归测试，确认服务恢复，输出验证报告。 |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| triager | 阶段零（步骤 1） | 阶段零完成后（步骤 2 分级确认） | 分级报告交付后即释放 |
| investigator-1 | 阶段二（步骤 5） | 阶段三完成后（步骤 8 根因确认） | 如需交叉验证则保留到仲裁结束 |
| investigator-2 | 阶段二（步骤 5） | 阶段三完成后（步骤 8 根因确认） | 如需交叉验证则保留到仲裁结束 |
| fixer | 阶段一（步骤 4，P0/P1）或阶段四（步骤 10） | 阶段四合并完成后（步骤 13） | 全程保持存活（止血 + 修复） |
| verifier | 阶段一（步骤 4，P0/P1）或阶段四（步骤 11） | 阶段四合并完成后（步骤 13） | 全程保持存活（验证止血 + 验证修复） |

## 阶段零：故障分级

### 步骤 1：启动 triager 进行故障评估

如果用户通过 `--severity` 指定了故障等级，跳过此步骤，直接采用指定等级进入对应流程。

否则，启动 triager，传入故障描述/告警信息，triager 执行快速评估：

1. **故障现象确认**：从告警信息/用户描述中提取关键信号

2. **影响范围评估**：
   - 受影响的服务/模块（API Gateway、调度器、存储、GPU 节点等）
   - 受影响的用户范围（全部用户 / 特定租户 / 内部服务）
   - 数据完整性风险

3. **严重程度自动判定**：Triager 按以下信号-等级映射表进行自动分级，匹配到的最高等级为最终判定：

| 信号特征 | 自动判定等级 | 判定依据 |
|---------|------------|---------|
| 告警含 "down"/"unreachable"/"5xx 100%" | **P0** | 服务完全不可用 |
| 告警含 "data loss"/"data corruption"/"数据丢失" | **P0** | 数据完整性受损 |
| 告警含 "security breach"/"unauthorized access"/"安全事件" | **P0** | 安全事件 |
| 多个服务同时告警（≥3 个独立服务） | **P0** | 大规模故障 |
| 告警含 "OOMKilled"/"OOM" 且影响核心服务 | **P0** | 核心服务 OOM |
| 错误率 > 50% 且持续 > 5 分钟 | **P1** | 核心功能严重受损 |
| P95 延迟 > 正常值 10x | **P1** | 严重性能退化 |
| 告警含 "CrashLoopBackOff" 且影响核心 Pod | **P1** | 核心服务反复崩溃 |
| 训练任务批量失败（>50%） | **P1** | GPU 集群大面积故障 |
| API 网关超时率 > 30% | **P1** | 大面积超时 |
| 错误率 10%-50% 且仅影响部分功能 | **P2** | 非核心功能受损 |
| 单节点异常（CPU/内存/GPU 告警） | **P2** | 单点故障 |
| 队列积压但有自愈趋势 | **P2** | 可自愈的积压 |
| 告警无业务影响（纯监控指标波动） | **P3** | 轻微异常 |
| 告警已自动恢复 | **P3** | 瞬时抖动 |

**自动判定置信度**：
- 匹配到明确信号词 → 高置信度，直接判定
- 仅根据描述推断 → 中置信度，标注"建议用户确认"
- 信息不足以判定 → 低置信度，默认 P2 并请求更多信息

4. **相关代码模块识别**：定位可能涉及的代码目录和配置文件
5. **日志/监控来源建议**：指出应该检查哪些日志文件和监控指标

### 步骤 2：分级确认

Triager 输出结构化分级报告：
- 故障等级 (P0/P1/P2/P3)
- 故障现象摘要
- 影响范围
- 相关模块和文件
- 建议的调查方向

确认方式：
- **标准模式**：向用户展示分级报告，AskUserQuestion 确认等级是否正确
- **单轮确认模式**：triager 自行判定，team lead 在后续阶段告知用户判定结果
- **完全自主模式**：自动决策，不询问用户

分级确认后释放 triager（关闭 teammate）。

## 阶段一：快速止血（仅 P0/P1）

P2/P3 跳过此阶段，直接进入阶段二。

### 步骤 3：评估止血方案

Team lead 根据分级报告中的信息，快速评估可行的止血方案：

| 止血手段 | 适用场景 | 风险等级 |
|---------|---------|---------|
| **版本回滚** | 最近有明确的代码变更导致故障 | 中（可能丢失新功能） |
| **配置回滚** | 最近有配置变更导致故障 | 低 |
| **服务降级** | 非核心功能异常拖累核心功能 | 低 |
| **节点隔离** | 特定 GPU 节点/Pod 异常，需要隔离故障节点 | 低 |
| **流量切换** | 多活/容灾场景下切换流量 | 中 |
| **扩容/重启** | 资源不足或服务状态异常 | 低 |

如果无法判断止血方案，直接进入阶段二调查。

### 步骤 4：执行止血

启动 fixer 执行止血方案，启动 verifier 验证止血效果。

**止血方案无论标准模式还是单轮确认模式/完全自主模式，都必须经用户确认后执行**（P0/P1 操作风险过高，不可自动决策）。

Team lead 用 AskUserQuestion 向用户展示：
- 建议的止血方案
- 预期效果和风险
- 是否执行

用户确认后：
1. Fixer 执行止血操作（回滚/降级/隔离等）
2. Verifier 验证止血效果（服务可用性、错误率下降、关键指标恢复）
3. Verifier 输出止血验证报告

止血成功后继续进入阶段二做根因调查。止血失败则：
- **标准模式**：AskUserQuestion 向用户汇报并请示下一步
- **单轮确认模式/完全自主模式**：**不可跳过，必须暂停问用户**（熔断机制）

---

## 阶段二：双路调查

### 步骤 5：启动双路独立调查

同时启动 investigator-1 和 investigator-2，各自独立调查，team lead 不让两位 investigator 看到对方的调查过程和结论。

**Investigator-1（变更角度）**：
- 检查最近的 git 提交历史（`git log --oneline -20`、`git diff`）
- 查找最近的部署/发布记录
- 分析可疑的代码变更（特别是与故障相关模块的 diff）
- 检查配置文件的最近修改（Kubernetes YAML、环境变量、feature flag）
- 检查依赖版本变更
- 输出：可疑变更列表 + 每个变更与故障的关联度评估

**Investigator-2（系统状态角度）**：
- 分析应用日志中的错误信息和异常栈
- 检查系统资源使用情况（CPU/内存/磁盘/GPU 利用率）
- 检查 Kubernetes Pod 状态（CrashLoopBackOff、OOMKilled、Pending 等）
- 检查网络连接和 API 响应状态
- 检查队列积压和消费延迟
- 检查数据库连接池和慢查询
- 输出：系统异常列表 + 时间线 + 关联分析

### 步骤 6：调查期间的决策升级

- Investigator 需要扩大排查范围（如检查其他服务的日志）：
  - **标准模式**：通知 team lead，team lead 决策
  - **单轮确认模式/完全自主模式**：investigator 自行扩展
- Investigator 发现需要执行诊断命令或临时操作：
  - 只读诊断命令可直接执行
  - 写操作（重启、清理等）必须升级 team lead

## 阶段三：根因确认

### 步骤 7：合并调查结论

两份调查报告完成后，team lead 合并分析。

**共识度计算公式**：
```
对比维度（共 4 项，每项 25%）：
  ① 根因指向（是否指向相同的变更/组件/配置）
  ② 故障触发链（是否描述相同的因果链路）
  ③ 影响范围（是否一致评估影响的服务和用户范围）
  ④ 时间线吻合度（两方还原的事件时间线是否一致）

共识度 = 一致维度数 × 25%
分歧度 = 100% - 共识度

分歧度 > 50%（≥3 项分歧）→ 需要仲裁（交叉验证或升级用户）
```

**情况一：结论一致**（共识度 = 100%）
- 两路调查指向相同根因 → 直接采纳，高置信度

**情况二：结论互补**（共识度 50%-75%）
- 两路调查从不同角度揭示同一问题的不同侧面 → 合并为完整根因分析

**情况三：结论分歧**（共识度 < 50%）
- 两路调查指向不同根因 → 需要仲裁：
  - Team lead 将两份报告同时发给 investigator-1 和 investigator-2
  - 两位 investigator 针对对方的证据进行交叉验证
  - 如仍无法收敛：
    - **标准模式**：AskUserQuestion 向用户展示两个方向，请用户根据业务知识判断
    - **单轮确认模式**：team lead 综合证据强度裁决，收尾时汇总
    - **完全自主模式**：自动决策，不询问用户

### 步骤 8：根因报告确认

Team lead 向用户输出根因分析报告：
- 根因描述（一句话总结）
- 证据链（从两路调查中提取的关键证据）
- 故障时间线（从故障触发到发现的完整时间线）
- 修复建议（永久修复方向）

确认方式：
- **标准模式**：AskUserQuestion 向用户确认根因分析是否合理
- **单轮确认模式**：team lead 自行确认，进入修复阶段
- **完全自主模式**：自动决策，不询问用户

## 阶段四：修复实施

### 步骤 9：制定修复方案

Team lead 根据根因分析，让 fixer 制定修复方案（如果阶段一已止血，此处为永久修复；否则为首次修复）：

- **Hotfix 分支工作流**：fixer 创建 `hotfix/<team-name>/<issue-summary>` 分支
- 修复方案包含：
  - 需要修改的文件列表
  - 修改内容说明
  - 回归测试计划
  - 是否需要配置变更

修复方案确认：
- **P0/P1 标准模式**：AskUserQuestion 向用户确认
- **P0/P1 单轮确认模式/完全自主模式**：AskUserQuestion 向用户确认（不可跳过）
- **P2/P3 标准模式**：AskUserQuestion 向用户确认
- **P2/P3 单轮确认模式**：fixer 自行决策，收尾时汇总
- **P2/P3 完全自主模式**：自动决策，不询问用户

### 步骤 10：实施修复

Fixer 在 hotfix 分支上实施修复：

1. 编写修复代码
2. 编写针对故障场景的回归测试（确保同类问题不再复发）
3. 运行现有测试确保不引入新问题
4. 执行自检：
   - 修改范围是否精准（不引入无关变更）
   - 测试是否覆盖故障场景
   - 是否有潜在副作用
5. 通知 verifier 和 team lead

### 步骤 11：验证修复

启动 verifier（如未启动）验证修复效果：

1. **代码审查**：审查 hotfix 分支的 diff，确认修复逻辑正确
2. **测试验证**：运行 fixer 编写的回归测试 + 全量测试
3. **故障复现验证**：确认故障场景在修复后不再复现
4. **副作用检查**：确认修复未引入新问题

Verifier 输出验证报告（通过/不通过 + 详细结果）。

### 步骤 12：验证不通过时

Verifier 列出具体问题，fixer 修复后重新提交，循环直到验证通过。

**熔断机制**：验证失败超过 2 轮，**无论标准模式还是单轮确认模式/完全自主模式**，必须暂停向用户汇报：
- 当前修复状态
- 2 轮失败的原因分析
- 建议：继续修复 / 更换方案 / 人工介入

### 步骤 13：合并修复

验证通过后，team lead 执行合并：
1. 在主分支执行 `git merge <hotfix-branch>` 合并修复
2. 如有合并冲突，fixer 解决后重新提交
3. Verifier 在主分支再次运行全量测试确认
4. 合并成功后清理 hotfix 分支

---

## 阶段五：收尾

### 步骤 14：故障总结报告

Team lead 按 `--lang` 指定的语言向用户输出故障总结报告：

```
## 故障总结报告

### 元信息
- 生成时间：YYYY-MM-DD HH:mm:ss
- 团队名称：team-incident-{YYYYMMDD-HHmmss}
- 执行模式：标准模式 / 单轮确认模式 / 完全自主模式
- 输出语言：zh / en
- 故障等级：P0/P1/P2/P3（来源：--severity 参数 / triager 自动判定）
```

**一、故障概要**
- 故障等级 (P0/P1/P2/P3)

## 智算云场景特别关注

故障调查和修复过程中，需特别关注以下智算云特有场景：

| 场景 | 排查要点 | 常见根因 |
|------|---------|---------|
| **GPU 节点异常** | nvidia-smi 输出、GPU ECC 错误、驱动版本、CUDA 版本兼容性 | 硬件故障、驱动不兼容、显存泄漏 |
| **训练任务失败** | 任务日志、NCCL 通信错误、分布式训练拓扑 | 节点间网络中断、GPU 故障、OOM |
| **K8s Pod 问题** | Pod 事件、Node 资源、调度约束、存储挂载 | 资源不足、镜像拉取失败、PVC 异常 |
| **API 网关超时** | 上下游响应时间、连接池状态、限流配置 | 后端服务慢、连接池耗尽、配置错误 |
| **队列积压** | 消费者状态、处理速率、死信队列 | 消费者异常、处理逻辑阻塞、资源不足 |
| **存储异常** | IOPS、延迟、容量、挂载状态 | 磁盘满、网络存储抖动、权限问题 |

## 核心原则

- **止血优先**：P0/P1 先止血再查根因，减少影响时长
- **分级响应**：不同等级不同流程，避免过度响应或响应不足
- **独立调查**：双路独立调查避免思维定势，提高根因定位准确率
- **变更即嫌疑**：优先排查最近的代码/配置/部署变更
- **证据驱动**：根因确认必须有证据链支撑，不接受猜测
- **最小修复**：hotfix 只修故障本身，不搭车做其他改动
- **验证闭环**：每次修复必须经过验证确认，不能"改了就算"
- **熔断保护**：止血失败、验证失败超 2 轮必须暂停问用户
- **时间线记录**：全程记录关键事件时间点，用于后续复盘

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 故障信息不足以分级 | Triager 列出缺失信息，team lead 用 AskUserQuestion 向用户补充 |
| 止血操作失败 | **必须暂停问用户**，fixer 汇报失败原因，team lead 展示替代方案 |
| 止血后故障未缓解 | 记录止血无效，继续进入调查阶段，扩大排查范围 |
| 双路调查均无结论 | Team lead 综合现有线索提出新调查方向，重新分配给 investigator；如仍无法定位，暂停问用户 |
| 根因涉及外部依赖（第三方服务、硬件） | 标注为外部因素，修复方案聚焦于防护措施（降级、重试、隔离）而非修复根因本身 |
| 修复引入新问题（verifier 发现） | 立即回滚修复，fixer 重新分析并修复，验证周期重置 |
| 验证失败超 2 轮 | **必须暂停问用户**，汇报当前状态和失败原因，请用户决策 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从最近的检查点恢复 |
| 合并冲突 | Fixer 解决冲突后重新提交，verifier 重新验证 |
| 故障在调查过程中自愈 | 仍需完成根因定位（防止复发），但可降低修复优先级，重点输出 action items |
| Git 仓库状态异常（脏工作区/未提交变更） | Team lead 在创建 hotfix 分支前检测工作区状态，提示用户先 stash 或提交变更，避免修复与未提交变更冲突 |
