**参数解析**：从 `$ARGUMENTS` 中检测以下标志：
- `--auto`：完全自主模式（不询问用户任何问题，全程自动决策）
- `--once`：单轮确认模式（将所有需要确认的问题合并为一轮提问，确认后全程自动执行）
- `--period=1w|2w|1m|3m`：分析周期（可选，默认 `2w`，即两周）
- `--compare`：与上一周期对比（可选，启用后输出趋势变化）
- `--lang=zh|en`：输出语言（默认 `zh` 中文）

解析后将标志从分析需求描述中移除。

| 模式 | 用户确认范围 | 条件节点处理 |
|------|-------------|-------------|
| **标准模式**（默认） | 数据源确认 + 最终报告确认 | 正常询问用户 |
| **单轮确认模式**（`--once`） | 仅最终报告确认 | 自动决策 + 收尾汇总 |
| **完全自主模式**（`--auto`） | 不询问用户 | 全部自动决策，收尾汇总所有决策 |

单轮确认模式下自动决策规则：
- **数据源不完整（缺少 CI/CD 数据）** → collector 标注"该数据源不可用，基于 git 推断"，不阻塞流程
- **两位 analyzer 对同一指标的评级差异过大（相差 2 级及以上）** → **不可跳过，必须暂停问用户**（熔断机制）
- **发现 DORA 指标全部处于 Low 水平** → **不可跳过，必须暂停问用户**（熔断机制）
- **git 历史不足以覆盖指定周期** → collector 标注"数据不足，实际分析周期为 X"，不阻塞流程
- **发现异常数据点（如单次提交含 1000+ 文件变更）** → analyzer 自行判断是否过滤

完全自主模式下：所有节点均自动决策，不询问用户。熔断机制仍然生效（评级差异过大、全部 Low 水平时仍必须暂停问用户）。

## 文件交接规范（File-Based Handoff）

所有 agent 间传递详细报告时，必须采用**文件交接模式**（防止上下文溢出触发 20MB 限制）：

1. **写入文件**：将完整报告写入团队工作目录：
   - 目录路径：`/tmp/{team-name}/`（team lead 在 TeamCreate 后执行 `mkdir -p /tmp/{team-name} && chmod 700 /tmp/{team-name}`）
   - 单个文件 ≤ 2000 行；超大报告拆分为 summary + details 文件
2. **发送引用**：通过 SendMessage 仅发送（≤500 字符）：
   - 文件路径（1 行）
   - 关键摘要（含核心指标/发现/评分）
3. **按需读取**：接收方使用 Read 按需读取文件，发送方不内联完整内容
4. **路径转发**：team lead 转发报告时只转发文件路径 + 摘要，不 Read 后再 SendMessage
5. **遵从校验**：team lead 收到超 1000 字符且不含 `/tmp/team-` 路径前缀的消息时，要求 agent 以文件交接模式重发

## 流程概览

```
阶段零  数据源识别 → 解析 period，识别 git/CI/CD 数据源，确认分析范围
         ↓
阶段一  数据采集 + 双路独立分析（并行）
         ├─ collector：git log/CI pipeline/deploy 数据采集
         ├─ analyzer-1：吞吐量分析（部署频率 + 变更前置时间）
         └─ analyzer-2：稳定性分析（变更失败率 + 恢复时间）
         ↓
阶段二  合并分析 → advisor 合并评级、交叉校验、输出统一 DORA 评级
         → 熔断检查（评级差异过大 / 全部 Low）
         → 生成改进建议
         ↓
阶段三  报告生成 → reporter 生成结构化 DORA 报告（含趋势、对比、路线图）
         ↓
阶段四  收尾 → 保存报告 + 清理团队
```

## 角色定义

| 角色 | 职责 |
|------|------|
| collector | 从 git 历史、CI/CD 管道、部署记录中采集原始数据，清洗异常值，输出结构化数据集。**仅在采集阶段工作，完成后关闭。** |
| analyzer-1 | **吞吐量维度分析**：部署频率（Deployment Frequency）和变更前置时间（Lead Time for Changes）。输出结构化分析报告。**独立分析，不与 analyzer-2 交流。** |
| analyzer-2 | **稳定性维度分析**：变更失败率（Change Failure Rate）和恢复时间（Time to Restore Service）。输出结构化分析报告。**独立分析，不与 analyzer-1 交流。** |
| advisor | 合并两份分析报告和 collector 原始数据，交叉校验、统一评级、生成改进建议和实施路线图。**不直接阅读代码/git，只对比和合并报告。完成后关闭。** |
| reporter | 基于统一 DORA 评级和改进建议生成结构化报告（含趋势图、对比表、路线图），按 `--lang` 语言输出。**完成后关闭。** |

### 角色生命周期

| 角色 | 启动阶段 | 关闭时机 | 说明 |
|------|---------|---------|------|
| collector | 阶段一（步骤 3） | 阶段一采集完成后（步骤 3） | 数据集交付后即释放 |
| analyzer-1 | 阶段一（步骤 4，收到数据后启动） | 阶段一分析完成后（步骤 5） | 分析报告交付后释放 |
| analyzer-2 | 阶段一（步骤 4，与 analyzer-1 并行） | 阶段一分析完成后（步骤 5） | 分析报告交付后释放 |
| advisor | 阶段二（步骤 6） | 阶段二合并完成后（步骤 9） | 统一评级和改进建议输出后释放 |
| reporter | 阶段三（步骤 10） | 阶段三报告生成后（步骤 12） | 报告确认后释放 |

## DORA 四大指标定义

| 指标 | 英文名 | 含义 | 数据来源 |
|------|--------|------|---------|
| **部署频率** | Deployment Frequency (DF) | 代码变更部署到生产环境的频率 | git tags/releases + CI/CD deploy logs |
| **变更前置时间** | Lead Time for Changes (LT) | 从代码提交到成功部署到生产的时间 | git commit → merge → deploy 时间链 |
| **变更失败率** | Change Failure Rate (CFR) | 导致生产故障的变更占总变更的比例 | revert commits + hotfix branches + CI failure logs |
| **恢复时间** | Time to Restore Service (MTTR) | 从生产故障发生到服务恢复的时间 | incident → fix commit → deploy 时间链 |

---

## DORA 评级标准（2023 State of DevOps Report）

| 指标 | Elite | High | Medium | Low |
|------|-------|------|--------|-----|
| **部署频率** | 按需（每天多次） | 每天至每周 | 每周至每月 | 每月至每半年 |
| **变更前置时间** | < 1 小时 | 1 天至 1 周 | 1 周至 1 个月 | 1 个月至 6 个月 |
| **变更失败率** | 0-5% | 5-10% | 10-15% | 15-64% |
| **恢复时间** | < 1 小时 | < 1 天 | 1 天至 1 周 | > 1 周 |

**综合评级**：取四项指标中最低的评级作为综合 DORA 评级（木桶原则）。

## 数据采集方法

| 数据类别 | 采集方式 | 工具/命令 |
|---------|---------|----------|
| 提交历史 | `git log --format` | 提交时间、作者、合并记录 |
| 分支生命周期 | `git branch --merged` + PR/MR 合并时间 | 分支创建→合并时间 |
| 部署记录 | git tags + GitHub Releases + CI/CD deploy logs | 部署时间、版本号 |
| CI/CD 管道 | GitHub Actions / GitLab CI / Jenkins API | 构建时间、成功/失败率 |
| 回滚记录 | revert commits + hotfix branches | 失败变更识别 |
| 故障记录 | incident labels + 紧急修复 PR | 故障持续时间 |

**如果某些数据源不可用**：collector 标注"该数据源不可用"，基于可用数据推断，报告中说明置信度。

## 分析维度分配

双 analyzer 按各自负责的维度独立分析，advisor 合并为统一 DORA 评级。

| 维度 | 权重 | 负责人 | 分析内容 |
|------|------|--------|---------|
| 部署频率 | 25% | analyzer-1 | 部署次数/周期、部署节奏稳定性、部署时间分布（工作日/周末）、自动化部署比例 |
| 变更前置时间 | 25% | analyzer-1 | 从 commit 到 deploy 的中位数/P90、各阶段耗时拆解（编码→review→CI→deploy）、瓶颈识别 |
| 变更失败率 | 25% | analyzer-2 | 失败变更数/总变更数、失败类型分布（构建失败/测试失败/生产故障）、失败趋势 |
| 恢复时间 | 25% | analyzer-2 | 故障→恢复中位数/P90、恢复方式分布（回滚/热修复/完整修复）、故障检测时间 |

**analyzer-1 总权重：50%**（部署频率 25% + 变更前置时间 25%）
**analyzer-2 总权重：50%**（变更失败率 25% + 恢复时间 25%）

## 阶段零：数据源识别

### 步骤 1：解析分析范围

Team lead 分析项目和参数：

1. 解析 `--period` 参数，确定分析时间窗口
2. 阅读项目结构，识别开发工作流：
   - 版本控制（git branching model：GitFlow/GitHub Flow/Trunk-based）
   - CI/CD 管道（GitHub Actions/GitLab CI/Jenkins/CircleCI）
   - 部署方式（手动/自动部署、蓝绿/金丝雀/滚动）
   - 版本发布方式（git tags/GitHub Releases/语义化版本）
   - 团队规模和协作模式（PR review 流程）
3. 识别可用数据源：
   - git 历史（commits/branches/tags/merges）
   - CI/CD 配置和日志
   - GitHub/GitLab API（PR/MR、Issues、Releases）
   - 部署日志和监控数据
4. 输出数据源清单和分析范围

### 步骤 2：用户确认分析范围

**标准模式**：向用户展示数据源清单和分析范围，AskUserQuestion 确认：
- 确认分析范围和时间窗口
- 补充数据源（如外部部署系统、故障记录）
- 调整分析关注点

**单轮确认模式**：跳过确认，直接进入阶段一。
**完全自主模式**：自动决策，不询问用户，直接进入阶段一。

## 阶段一：数据采集 + 双路独立分析

### 步骤 3：启动 collector 采集数据

**Collector 数据采集**：
1. 根据 `--period` 确定时间窗口，如 `--compare` 则采集两个连续周期的数据
2. 采集 git 原始数据：

| 数据项 | 采集命令/方式 | 输出格式 |
|--------|-------------|---------|
| 提交记录 | `git log --format='%H|%aI|%an|%s' --after=DATE` | commit_hash\|timestamp\|author\|message |
| 合并记录 | `git log --merges --format='%H|%aI|%P|%s'` | merge 节点及父提交 |
| 分支信息 | `git branch -a --format='%(refname:short)|%(creatordate:iso)'` | 分支名\|创建时间 |
| 标签/发布 | `git tag --sort=-creatordate --format='%(refname:short)|%(creatordate:iso)'` | 版本\|发布时间 |
| 回滚记录 | `git log --grep='revert\|rollback\|hotfix' -i` | 回滚/修复提交 |

3. 采集 CI/CD 数据（如可用）：
   - 构建历史（成功/失败/持续时间）
   - 部署历史（时间戳/环境/版本）
   - 测试覆盖率趋势

4. 数据清洗：
   - 过滤 merge commits（避免重复计数）
   - 过滤 bot commits（dependabot/renovate）
   - 识别异常数据点（如单次提交 1000+ 文件）
   - 标注数据缺失区间

5. 汇总输出**结构化数据集**发送给 team lead
6. Collector 完成后关闭

### 步骤 4：分发数据集并启动双 analyzer

Team lead 收到 collector 数据集后，将原始数据分发给两位 analyzer，**并行启动**。

### 步骤 5：独立并行分析

两位 analyzer 各自分析负责的指标，**互不交流**。

**Analyzer-1（吞吐量维度）** — 分析以下指标：

| 分析项 | 计算方法 | 输出 |
|--------|---------|------|
| **部署频率** | 统计周期内部署到生产的次数 / 周期天数 | 日均部署次数 + DORA 评级 |
| **部署节奏** | 部署时间间隔的标准差 | 稳定性评估（规律/不规律） |
| **部署时间分布** | 按星期几/小时统计部署分布 | 热力图 + 风险时段识别 |
| **变更前置时间** | commit timestamp → deploy timestamp 的中位数和 P90 | 中位数 + P90 + DORA 评级 |
| **阶段耗时拆解** | 编码→PR创建→Review通过→CI通过→部署 各阶段耗时 | 瓶颈环节识别 |
| **PR Review 效率** | PR 创建到首次 review、到合并的时间 | Review 瓶颈分析 |
| **批量大小** | 每次部署包含的 commit 数、变更文件数 | 批量大小趋势 |

- 对每个指标输出：数值 + DORA 评级 + 趋势方向（如启用 --compare）+ 分析说明
- 识别周期内的异常时段（如连续 2 周无部署）
- 按团队成员拆分（如可识别）输出个人前置时间

**Analyzer-2（稳定性维度）** — 分析以下指标：

| 分析项 | 计算方法 | 输出 |
|--------|---------|------|
| **变更失败率** | 导致回滚/热修复的变更数 / 总变更数 × 100% | 失败率 + DORA 评级 |
| **失败类型分布** | 按失败原因分类（构建失败/测试失败/生产故障/配置错误） | 类型占比饼图 |
| **失败根因** | 分析失败变更的共同模式（文件类型/模块/作者） | 高风险变更模式 |
| **恢复时间** | 故障发生→服务恢复的中位数和 P90 | 中位数 + P90 + DORA 评级 |
| **恢复方式分布** | 回滚/热修复/完整修复 的比例 | 恢复策略评估 |
| **故障检测时间** | 从部署到发现故障的时间 | 监控有效性评估 |
| **CI 健康度** | CI 管道成功率、平均构建时间、Flaky test 比例 | CI 稳定性评估 |

- 对每个指标输出：数值 + DORA 评级 + 趋势方向（如启用 --compare）+ 分析说明
- 识别失败模式（如特定模块变更失败率高）
- 评估测试覆盖率与变更失败率的相关性

## 阶段二：合并分析

### 步骤 6：启动 advisor

两位 analyzer 分析完成后，启动 advisor。Team lead 将以下材料发送给 advisor：
- Collector 结构化数据集
- Analyzer-1 吞吐量分析报告
- Analyzer-2 稳定性分析报告
- 项目开发工作流信息

### 步骤 7：合并评级与交叉校验

Advisor 执行合并分析：

1. **评级汇总**：汇总四项 DORA 指标的评级
2. **交叉校验**：检查两位 analyzer 的数据一致性（如 analyzer-1 统计的总变更数 与 analyzer-2 使用的总变更数是否一致）
3. **综合评级**：取四项指标中最低评级作为综合 DORA 评级
4. **相关性分析**：分析指标间的关联（如部署频率高是否伴随高失败率）
5. **趋势分析**（如启用 --compare）：与上一周期对比，输出改善/恶化方向

**共识度计算公式**：
```
两位 analyzer 分析有部分数据交叉（如 analyzer-1 的变更前置时间 与 analyzer-2 的变更失败率都涉及"变更"的定义和计数）。
对于交叉数据：

数据一致性 = 两位 analyzer 使用的基础数据（总变更数、部署次数）一致的比例
评级一致性 = 两位 analyzer 对重叠评估维度评级一致的比例

数据一致性 < 80% → 触发数据校验，advisor 重新核算
评级差异 ≥ 2 级 → 标注为争议项

争议率 = 争议项数 / 四项指标总数 × 100%
争议率 > 50% → 触发熔断，必须暂停问用户
```

### 步骤 8：差异校验与熔断检查

**评级差异校验**：
- 如果两位 analyzer 对某指标的评级差异 ≥ 2 级（如一方定 Elite、另一方定 Medium），advisor 标注为"争议项"
- 争议项占比 > 50%（即 ≥ 3 项指标有争议）→ **不可跳过，必须暂停问用户**（熔断机制）

**全低水平熔断**：
- 四项指标全部处于 Low 水平 → **不可跳过，必须暂停问用户**（熔断机制）
- 向用户说明工程效能状况，确认是否继续生成完整报告和改进路线图

**无熔断触发**时，advisor 输出**统一 DORA 评级和改进建议**：


## 核心原则

- **数据驱动**：所有评级基于客观数据，不做主观推测，数据不足时标注置信度
- **双路独立**：两位 analyzer 从不同维度独立分析，互不交流，避免偏见
- **DORA 对齐**：评级标准严格对齐 DORA 研究报告（2023 State of DevOps Report）
- **木桶原则**：综合评级取最低值，改进优先从短板开始
- **可操作性**：改进建议必须具体可执行，附带预期效果和时间窗口
- **持续跟踪**：建议定期运行并与历史数据对比，形成改进闭环
- **隐私保护**：团队成员维度数据仅用于识别改进方向，不用于绩效考核
- **上下文感知**：考虑团队规模、项目阶段、行业特征等上下文因素

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| git 历史不足以覆盖指定周期 | Collector 标注实际可用时间范围，基于可用数据分析 |
| CI/CD 数据不可用 | 仅基于 git 数据推断部署频率和失败率，标注"基于 git 推断，置信度中等" |
| 无部署标签/发布 | 以合并到主分支的 PR 作为"部署"代理指标 |
| 单人项目（无 PR 流程） | 调整前置时间计算方式（commit → push 到主分支），标注"单人项目，指标含义不同" |
| 异常数据点（如大规模合并/迁移） | Analyzer 标注并排除异常值，报告中说明 |
| 两位 analyzer 评级差异过大（争议项 > 50%） | 触发熔断，暂停问用户裁决 |
| 四项指标全部 Low | 触发熔断，暂停向用户说明现状并确认 |
| Monorepo（多服务共享仓库） | Collector 按目录/标签分服务采集，报告按服务拆分 |
| Teammate 无响应/崩溃 | Team lead 重新启动同名 teammate（传入完整上下文），从当前步骤恢复 |

### 共识度计算

team lead 按五维度评估双路分析的共识度：

| 维度 | 权重 |
|------|------|
| 发现一致性（相同问题/结论） | 20% |
| 互补性（独有但不矛盾的发现） | 20% |
| 分歧程度（直接矛盾的结论） | 20% |
| 严重度一致性（同一问题的严重等级差异） | 20% |
| 覆盖完整性（两路合并后的覆盖面） | 20% |

共识度 = 各维度加权得分之和

- **≥ 60%**：自动合并，分歧项由 team lead 裁决
- **50-59%**：合并但标注分歧，收尾时汇总争议点
- **< 50%**：触发熔断，暂停并向用户确认方向
